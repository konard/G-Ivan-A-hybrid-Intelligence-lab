# RFC: Documentation Architecture Balance (Anti-Inflation vs Atomicity)

## Proposal

Принять рамку выбора гранулярности документа из **трёх уровней детализации** —
**Index → Summary → Full** — как способ примирить два уже принятых принципа Хаба,
которые в пределе тянут в разные стороны:

- **Anti-Inflation** ([governance/repo-model.md](../repo-model.md)) — «артефакт
  только при операционной боли», не плодить документы → склонность *сливать*
  связанное в один файл;
- **Atomicity** ([standards/executable-documentation-standard.md](../../standards/executable-documentation-standard.md),
  раздел «Atomization») — один артефакт = один файл → склонность *дробить*.

RFC фиксирует **решение и rationale**. Обязательная норма (если она потребуется)
делегируется в active artifact — `governance/repo-model.md` (Decision Rules) или
отдельный standard. До такой делегации downstream-проект ссылается на RFC как на
обоснование, но не обязан выполнять его механически
([governance/rfc/README.md](README.md)).

> 🧭 **Index / Summary / Full — это уровни *детализации одного знания*, а НЕ
> уровни документации L1–L4.** Framework L1–L2 (`docs/`) и Methodology L3–L4
> (`governance/`, `standards/`, …) описаны в
> [governance/artifact-map.md](../artifact-map.md) и
> [governance/rfc/knowledge-lifecycle-proposal.md](knowledge-lifecycle-proposal.md).
> Чтобы избежать путаницы, здесь сознательно не используются «L0/L1/L2».

## Problem

ИИ-агент, которому нужно одно правило, вынужден читать файл целиком: правило
живёт внутри большого документа без точки входа. Стоимость — токены и контекст —
растёт линейно с размером файла и **умножается на масштаб экосистемы** (сегодня
4 репозитория, в пределе 5–10–20).

Конфликт принципов:

- Если всегда *сливать* (Anti-Inflation в крайности) — рождаются монолиты по
  десятки тысяч строк, дорогие для чтения и для diff-review.
- Если всегда *дробить* (Atomicity в крайности) — рождаются сотни мелких файлов,
  дорогие для навигации; каталог сам становится контекстным шумом (прецедент
  «91 000+ skills», [practices/agent-work/skill-catalog-token-budget.md](../../practices/agent-work/skill-catalog-token-budget.md)).

Нужна не победа одного принципа, а **критерий выбора** между ними для каждого
документа — дешёвый в применении и без новой бюрократии.

## Decision Scope

RFC предлагает **рамку классификации и навигации**. Он:

- НЕ вводит CI-скраперы, авто-парсинг документов или `artifact-graph.json`
  (прямой запрет НФТ; Anti-Inflation). Навигация — markdown-ссылки + якоря +
  [governance/artifact-map.md](../artifact-map.md);
- НЕ добавляет обязательных полей frontmatter сверх четырёх
  ([standards/frontmatter-standard.md](../../standards/frontmatter-standard.md));
- НЕ переименовывает и не дробит файлы сам по себе — повышение/понижение класса
  документа остаётся **решением человека**;
- развивает выводы PR #229
  ([governance/rfc/external-knowledge-integration.md](external-knowledge-integration.md)),
  а не дублирует их: тот RFC отвечает на вопрос *«какие внешние источники ценны и
  как они текут в практику»*, этот — *«как нарезать сам документ, чтобы агент
  читал только нужный срез»*.

**Выводы из текущего состояния:**

1. Боль не равномерна: Хаб — образец, спок `clarify-engine-ai` — антипаттерн.
   Единое жёсткое правило для всех навредит (Хаб не нужно дробить дальше).
2. Поверсионные копии целого файла (`…_v1.4.md`, `…_v1.5.md`) — это инфляция:
   diff истории должен жить в git и `CHANGELOG`, а не в копиях-артефактах.
3. Крупные нормативные файлы (ADR/RFC/CONCEPT) страдают не от размера как такого,
   а от **отсутствия точки входа**: агенту негде взять «оглавление правил».

## 2. Международные практики (развитие PR #229, без дублирования)

Источники уже зафиксированы в Base Registry
([research/external-knowledge/external-sources-registry.md](../../research/external-knowledge/external-sources-registry.md)).
Этот RFC не заводит новые строки реестра, а **достраивает срез про гранулярность
документации**:

- **`ext-006` Diátaxis** (`stage: observation`) — разделяет документацию по
  *назначению* (tutorial / how-to / reference / explanation). Наш вклад: назначение
  коррелирует с уровнем детализации — *reference* тяготеет к Index/атому, а
  *explanation* (rationale, research) — к Full. То есть Diátaxis отвечает «что это
  за текст», а Index/Summary/Full — «насколько мелко его читать».
- **`ext-008` Effective Context Engineering / `ext-007` контекст-инжиниринг (ru)**
  (`stage: observation`) — «минимально достаточный контекст в окне». Наш вклад:
  это аргумент за **lazy loading** — агент подтягивает Full только когда Index не
  закрыл вопрос.
- **`ext-001` Building Effective Agents** (`stage: research`) — retrieval по
  индексам, а не дамп контента. Прямой прецедент в Хабе уже зафиксирован как
  practice-node [skill-catalog-token-budget.md](../../practices/agent-work/skill-catalog-token-budget.md)
  (кейс «91 000+ skills»): каталог без дисциплины извлечения становится шумом.

Российское комьюнити (Habr, кейс slam) и международные практики сходятся в одном:
**discoverability через индекс/метаданные, загрузка — только нужного узла**.
Регуляторика (NIST AI RMF, ISO/IEC, EU AI Act) задаёт требование
*трассируемости* решений, но не предписывает гранулярность файлов — поэтому она
здесь учитывается как ограничение (ссылки не должны рваться при дроблении), а не
как источник классификации. Углублённый разбор регуляторики — зона
[research/hub/2026-06-12-international-ai-governance-practices.md](../../research/hub/2026-06-12-international-ai-governance-practices.md),
дублировать его не нужно.

## 3. Классификация документов (максимум 3 класса)

| Класс | Когда | Как организован | Чтение агентом |
| --- | --- | --- | --- |
| **A — Index** (критичные / частые) | Правила, контракты, часто запрашиваемые нормы; короткий атомарный смысл; используется ≥2 репозиториями. | **Фрагментация**: один файл = одно правило/контракт. Каталог собирается индексом (`README.md`/`artifact-map.md`). | Один файл = один ответ. Дамп каталога не нужен. |
| **B — Summary** (средние / контекстные) | ADR, стандарты с разделами, гайды; читаются по частям в зависимости от вопроса. | **Один файл + структура для частичного чтения**: устойчивые заголовки-якоря и короткое оглавление вверху (Index-голова внутри файла). | Агент идёт по якорю к нужному разделу, а не читает целиком. |
| **C — Full** (редкие / справочные) | Research-отчёты, исторический rationale, концепты, аудиты. | **Один файл, читается целиком** при редком обращении; версии — в git/CHANGELOG, не в копиях-файлах. | Открывается целиком только когда реально нужен; в норме закрыт инсайтом/Summary. |

**Критерии отнесения (3 оси):**

- **Частота запроса** — как часто агент/человек обращается за фрагментом
  (высокая → A, средняя → B, низкая → C).
- **Размер / сложность** — короткий атом (A) · структурируемый средний (B) ·
  крупный нарратив (C).
- **Охват** — нужен многим репозиториям (тянет в A, атомарность ради переиспользования)
  · одному проекту контекстно (B) · разовая справка (C).

При конфликте осей **частота запроса важнее размера**: часто дёргаемое правило
дробим в A, даже если оно короткое и могло бы «ужаться» в Summary.

**Переходы между классами (решение человека, фиксируется в PR/CHANGELOG):**

```text
C (Full) --растёт частота запросов--> B (Summary): добавить якоря/оглавление
B (Summary) --раздел стал самостоятельным правилом--> A (Index): выделить атом + ссылка
A (Index) --атом перестали запрашивать--> C (Full): слить обратно (Anti-Inflation)
```

Понижение (A→C) так же легитимно, как повышение: если атом перестали читать, он
становится инфляцией и сливается обратно. Это и есть баланс — класс не статичен.

**Примеры на текущих файлах:**

| Файл | Класс сейчас | Рекомендация |
| --- | --- | --- |
| `standards/frontmatter-standard.md` (4 поля) — норма, дёргается постоянно | A | Оставить атомом; уже корректно. |
| mango `docs/adr/002-pattern-standard.md` (38 КБ) | B | Добавить оглавление-якоря; не дробить файл ADR. |
| clarify `docs/research/…_v2.md` (220 КБ) | C | Оставить Full; завести короткий Summary-вход; **версии-копии убрать в git/CHANGELOG**. |
| Хаб `research/mango/2026-05-27-capability-decomposition.md` (1774 стр.) | C | Не дробить ради дробления (Anti-Inflation); по желанию — Summary-голова. |

## 4. Гибкий подход без бюрократии

1. **Уровни детализации одного знания — Index / Summary / Full.** Это не новые
   файлы по умолчанию, а *способ читать*. Index часто = уже существующий
   `README.md`/`artifact-map.md`; Summary-голова = первые строки самого файла;
   Full = тело документа. Новый артефакт заводится только при операционной боли
   (Anti-Inflation).
2. **Навигация — без новой инфраструктуры.** Используем то, что есть: markdown-
   ссылки в теле, устойчивые заголовки-якоря и
   [governance/artifact-map.md](../artifact-map.md). Запрет на `artifact-graph.json`
   и авто-парсинг сохраняется.
3. **Frontmatter остаётся минимальным.** RFC сознательно **не** добавляет
   навигационные поля во frontmatter: это противоречило бы
   [standards/frontmatter-standard.md](../../standards/frontmatter-standard.md)
   («навигация живёт в artifact-map, а не во frontmatter») и добавило бы
   бюрократию. «Frontmatter как навигация» из issue решается заголовками-якорями и
   картой артефактов, а не разрастанием метаданных. (См. Open Decision №1.)
4. **Lazy loading как протокол чтения.** Агент: сначала Index → если мало,
   Summary-якорь → Full только при необходимости. Это поведенческое правило, а не
   код.

## 6. Поэтапный план внедрения

- **Фаза 0 (этот PR).** Зафиксировать рамку Index/Summary/Full и классификацию
  как `draft` RFC. Никаких массовых правок файлов. *(Атомарность Phase 0 = файлы +
  markdown-ссылки; графовые механизмы — предмет будущего обсуждения, не
  документации.)*
- **Фаза 1 (по решению человека).** Пилот на одном болевом файле каждого класса
  (например, mango ADR → якоря; clarify research → Summary-вход + чистка
  версий-копий). Замерить эффект на одном агенте-запросе.
- **Фаза 2 (опционально, после пилота).** Если рамка подтверждается — делегировать
  норму в `governance/repo-model.md` (Decision Rules) или отдельный standard и
  отразить в `artifact-map.md`. Только тогда RFC может быть повышен.

## 7. Оценка эффекта

- **Чтение по якорю вместо файла целиком** для класса B даёт экономию контекста
  пропорционально доле нерелевантных разделов (для ADR ≈ 38 КБ обращение к одному
  разделу — это единицы КБ вместо десятков).
- **Чистка версий-копий** в `clarify-engine-ai` (`v1…v1.5`) снимает кратное
  дублирование артефакта без потери истории (она в git).
- **Экономия растёт с масштабом**: правило, дёрнутое в N репозиториях, окупает
  атомизацию (класс A) тем сильнее, чем больше N.
- Риск/стоимость низкие: Фаза 0 — только текст RFC; необратимых правок нет.
