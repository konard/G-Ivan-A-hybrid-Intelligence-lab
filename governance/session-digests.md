# Session Digests — суммарии сессий

Единая точка хранения **суммарий длинных сессий** с внешними агентами
(ChatGPT, Claude, другие LLM-чаты) для передачи контекста между чатами. Файл —
часть механизма [Передачи контекста между чатами](../standards/session-handover-standard.md):
когда сессия становится длинной, агент по согласию пользователя фиксирует здесь
сжатый контекст, чтобы следующий чат продолжил работу из репозитория, а не из
пересказа по памяти.

> 🧭 **Назначение в одну строку.** Репозиторий — источник истины; суммарии сессий
> тоже живут в репозитории, а не только в памяти LLM, где проходило обсуждение.

## Кому это нужно (и кому НЕ нужно)

| Роль | Использует механизм? | Почему |
| --- | --- | --- |
| **Внешний агент** (ChatGPT/Claude в длинном диалоге с пользователем) | ✅ Да | Накопляет контекст в сессии; суммаризация нужна для передачи контекста в новый чат. |
| **Агент-исполнитель задачи** (task executor, см. [роли](../AI_GOVERNANCE.md)) | ❌ Нет | Получает задачу → исполняет → создаёт PR. Не ведёт длинных диалогов и не накапливает контекст между задачами. **Не должен жечь токены на суммаризацию** при исполнении задач. |

> ⛔ **Явное ограничение.** Механизм суммаризации **НЕ влияет на работу
> агента-исполнителя** при исполнении задач. Если ты читаешь этот файл как
> агент-исполнитель в рамках задачи в репозитории — он для тебя справочный
> артефакт, а не триггер действий. Суммаризацию инициирует только внешний агент в
> длинном чате и только **при передаче контекста в чат**, по согласию пользователя.

## Как пользоваться (экономия токенов)

1. **При передаче контекста в новый чат** читай сначала только **индекс** (таблицу
   ниже): он даёт карту тем и открытых вопросов без полного текста суммарий.
2. **Полную суммарию** читай только когда тема релевантна текущей задаче.
3. **Не пересказывай контекст из памяти** — ссылайся на якорь суммарии в этом файле
   и на артефакты репозитория.

## Как добавить суммарию

1. Спроси пользователя, нужно ли суммаризировать сессию (порог — 30 обращений или
   ~50К токенов; см. [handover prompt](../templates/htom/AI_SESSION_HANDOVER_PROMPT.md)).
2. При согласии — сформируй задачу (issue) на сохранение суммарии: правки в репо
   проходят через `issue → PR → review`, агент не коммитит напрямую.
3. Добавь строку в **Индекс** и блок суммарии с якорем `## YYYY-MM-DD — тема`.
4. Структура блока: **Контекст**, **Решения**, **Открытые вопросы**, **Следующие
   шаги**. Все ссылки — на артефакты в репозитории, не на обсуждения в чатах.
5. Если в суммарии есть **Открытые вопросы**, добавь их в
   [`governance/backlog.md`](backlog.md), секция «Открытые вопросы». Если вопрос
   уже есть в BACKLOG — не дублируй, а добавь ссылку на дайджест в колонку
   «Связанные дайджесты».

## Как удалять решённые вопросы (Traceability)

- Когда открытый вопрос **зафиксирован в артефакте** (стандарт, RFC, governance-файл)
  — удали его из блока «Открытые вопросы» и оставь ссылку на артефакт в «Решениях».
- Когда **вся** суммария отражена в артефактах — её можно удалить из файла или
  пометить статусом `решено` со ссылкой на артефакты. Файл хранит **актуальный**
  контекст, а не архив: решённое уходит в артефакты, открытое остаётся здесь.

## Разложение на проектные репо

Механизм переносится в проекты (HTOM-команды, spoke-репозитории) через геном
шаблона: [`templates/htom/AI_SESSION_HANDOVER_PROMPT.md`](../templates/htom/AI_SESSION_HANDOVER_PROMPT.md)
синхронизируется Smart Sync ([guides/sync-with-projects.md](../guides/sync-with-projects.md)).
Handover prompt в каждом проекте инструктирует агента вести `governance/session-digests.md`
**в текущем репозитории** проекта (создать при первом использовании). Так
суммарии каждого проекта остаются его источником истины и не смешиваются с хабом.

## 2026-06-13 — Архитектура документации и баланс Anti-Inflation vs атомарность

**Сессия:** длинное обсуждение (100+ сообщений) архитектуры документации хаба с
внешним агентом. Первая суммария, созданная по новому механизму.

### Контекст

- Проблема **токенов при чтении файлов агентом**: крупные файлы дорого читать
  целиком, а дробление на атомы увеличивает число файлов и риск инфляции
  (см. [governance/repo-model.md](repo-model.md), Anti-Inflation principle).
- Нужен баланс между **атомарностью** (мелкие переиспользуемые единицы знаний) и
  **Anti-Inflation** (не плодить файлы без операционной боли).
- Связано с разделением типов передачи контекста, зафиксированным ранее
  (PR #216): Runtime-онбординг vs передача контекста между чатами
  (см. [standards/session-handover-standard.md](../standards/session-handover-standard.md)).

### Решения

- Принят **гибрид B+C**: индекс-навигация поверх документов + сжатые сводки, чтобы
  агент читал индекс, а полный текст — по необходимости.
- Договорённость о наименовании уровней: `L0/L1/L2 → Index/Summary/Full`
  (читаемые имена вместо номеров уровней).
- Зафиксирована идея **CI-проверки `INDEX.md`** (индекс не расходится со
  структурой) — как развитие существующих проверок
  [tools/validate-repository-structure.sh](../tools/validate-repository-structure.sh).
- Создан сам механизм суммаризации сессий (этот файл + раздел в handover prompt) —
  issue #225.

### Открытые вопросы

1. **Когда переходить к графовой структуре** знаний (атомы + явные связи) вместо
   плоского индекса — по какому порогу боли/объёма.
2. **Как фиксировать связи** между атомами, чтобы не дублировать метаданные и не
   нарушать Anti-Inflation (см. обсуждение в
   [governance/rfc/draft-triage-and-exit-plan.md](rfc/draft-triage-and-exit-plan.md):
   Unidirectional Links, Index over Frontmatter).
