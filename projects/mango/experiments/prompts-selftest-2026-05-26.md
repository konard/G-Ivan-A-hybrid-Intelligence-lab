# Само-тест готовых Mango prompts

Версия: 0.1

Дата: 2026-05-26

## Назначение

Этот файл фиксирует ручной dry-run шести готовых prompt assets из
`projects/mango/prompts/`. Проверка не заменяет реальный запуск в ChatGPT или
Claude, но показывает, что каждый prompt:

1. запрашивает входные данные;
2. ведет диалог, если данных не хватает;
3. не требует доступа к исходному экспериментальному отчету;
4. задает стабильный формат результата.

## Тестовые входы

| Код | Вход |
| --- | --- |
| `TZ-1` | "Нужна облачная АТС: запись входящих и исходящих звонков, IVR, SMS клиентам, API для CRM, отчет по источникам звонков. Срок договора 24 месяца." |
| `US-1` | "Хочу статистику по обзвону, чтобы руководитель видел результат работы менеджеров." |
| `US-2` | "Робот молчит с клиентом, пусть потом сам перезвонит и запишет результат в CRM." |
| `UC-1` | User Story: "Как менеджер продаж, я хочу видеть карточку клиента и подсказку во время исходящего звонка, чтобы быстрее вести разговор." |

## Результаты self-test

| Промпт | Вход | Результат dry-run | Замечания |
| --- | --- | --- | --- |
| `tz-stats-generator_exp-2026-05.md` | `TZ-1` + выдержка `classification.md` | Запрашивает `current_tz_id`, предыдущий JSON и выдержку классификатора; ожидаемо строит таблицу классов `voice-ucaas`, `digital-channels`, `platform`, `analytics`, `commercial`. | Работает как RAG-навигатор; без выдержки классификатора должен попросить ее. |
| `tz-stats-generator_simple-2026-05.md` | `TZ-1`, классов нет | Сначала предлагает согласовать черновые классы, затем считает один класс один раз на ТЗ и помечает спорные строки `needs-review`. | Подходит для быстрого чата без репозитория; финальные `class-code` требуют ручного согласования. |
| `user-story-generator_exp-2026-05.md` | `US-1` + выдержки glossary/classification | Детектирует роль `руководитель`, цель `видеть результат`, неполные метрики и период; перед финальной story задает вопросы про показатели и фильтры. | Не должен выдумывать метрики конверсии или период. |
| `user-story-generator_simple-2026-05.md` | `US-1` | Возвращает статус `needs-clarification`, спрашивает объект статистики, метрики, период и пользователей отчета. | Выполняет главное требование: не делает финальную User Story из слишком общего входа. |
| `usecase-stepwise-generator_exp-2026-05.md` | `UC-1` + выдержка `classification.md` | Проверяет готовность, предлагает акторов, останавливается; после подтверждения предлагает capability и компоненты, снова останавливается. | Сохраняет multi-turn контроль и не пишет Use Case до подтверждений. |
| `usecase-stepwise-generator_simple-2026-05.md` | `UC-1` | Предлагает акторов простыми словами: менеджер, клиент, система телефонии, сервис подсказок; ожидает подтверждение. | Работает без repo terms; service aliases не нужны до согласования. |

## Проверка лимитов

| Группа | Ожидание | Итог |
| --- | --- | --- |
| `_exp` | Prompt body не длиннее 1000 символов. | Выполнено для трех `_exp` файлов. |
| `_simple` | Prompt body до примерно 3000 символов. | Выполнено для трех `_simple` файлов. |
| Все файлы | Есть frontmatter `type`, `variant`, `updated`, `scope`, `based_on`. | Выполнено; также добавлены обязательные поля локального validator: `status`, `version`, `ai-generated`. |
| Все prompt body | Начинаются с роли и заканчиваются запросом входа. | Выполнено. |

## Идеи на будущее

1. Автоматически генерировать `_simple` из `_exp`: брать RAG-ссылки как
   источник правил, а затем заменять их вопросами к пользователю и встроенными
   эвристиками.
2. Сделать единый prompt с переключателем `mode: exp | simple`: в `exp` он
   просит выдержки из репозитория, в `simple` - согласует термины в диалоге.
3. Собирать A/B-метрики: число уточняющих вопросов, доля домыслов,
   стабильность структуры, время до первого полезного результата и число
   ручных правок после ответа ИИ.
4. Добавить machine-readable contract для prompt output: минимальные ключи,
   допустимые статусы и перечень блокирующих ошибок.
5. После появления `projects/mango/kb/glossary.md` обновить `_exp` prompts так,
   чтобы они ссылались на подтвержденный словарь вместо fallback glossary.

## Вопросы для согласования

1. Достаточно ли шести prompt assets для пилота, или нужен отдельный prompt для
   генерации ФТ / требований к доработке?
2. Какой вариант приоритетен для первых пользователей: `_simple`, потому что он
   работает без репозитория, или `_exp`, потому что он лучше контролирует
   mapping?
3. Нужно ли в каждом `_simple` prompt запрещать финальный результат при
   `confidence: low`, или допустим явно помеченный `draft`?
4. Должны ли prompt body копироваться вместе с frontmatter, или интерфейс для
   пользователя должен показывать только текст после frontmatter?
5. Какие self-test кейсы считать обязательными для будущих изменений:
   неполный вход, сленг, составной запрос, повтор на стабильность или все
   перечисленное?

## Итог

Шесть prompt assets проходят ручной self-test как готовые тексты для внешнего
ИИ-чата. Основной остаточный риск - отсутствие живого `kb/glossary.md`; до его
появления `_exp` prompts используют `classification.md` и Mango-only glossary
как RAG-навигацию, а `_simple` prompts переводят неопределенность в вопросы.
