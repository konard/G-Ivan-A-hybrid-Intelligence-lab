# Глобальный анализ репозитория Хаба как прообраза агентной системы

## Summary / BLUF

**Тезис.** Репозиторий Хаба (`hybrid-Intelligence-lab`) уже сегодня — не «набор
документов», а **работающий прообраз агентной системы**: в нём эмпирически
сложились роутер задач, регулятор автономии, память с разделением на рабочую и
долговременную, boot-протокол, слой guardrails и петля обучения экосистемы. Это
исследование фиксирует **27 гипотез** о том, как использовать Хаб как прообраз
агентной системы, проводит по каждой **≥5 стресс-тестов** (всего 157 проверок),
и — главное — выводит **методологический вектор**: не список правил, а
*направление развития*, вдоль которого Хаб превращается из репозитория знаний в
исполнимую агентную среду, не теряя жёсткого правила «Человек > Команда Q >
Исполнитель».

**Режим.** `Creative` + `Deep Think`. Максимум свободы интерпретации, максимум
токенов, симуляция команды из **8 экспертов**, adversarial стресс-тесты по
каждой гипотезе. Это **исследование, а не решение**: ни одна гипотеза не
переводится в норму без явного RFC/ADR и слова человека
([AI_GOVERNANCE.md](../../AI_GOVERNANCE.md), правило 4).

**Ключевые результаты (сжато).**

1. **Методологический вектор** (§7): «От описательного репозитория знаний — к
   исполнимой агентной среде с человеком в петле», разложенный на 6 осей
   движения (executability, routing, memory, guardrails, sync, meta).
2. **27 гипотез** по направлениям A–H (§4), каждая заземлена на *реальный*
   артефакт Хаба, а не на общую теорию.
3. **Матрица гипотезы × стресс-тесты** (§5): 157 проверок, из них 86 ✅, 54 ⚠️
   (частично — есть пробел), 17 ❌ (гипотеза в этой точке не держит нагрузку —
   вскрыт конкретный пробел). Ни одна гипотеза не опровергнута целиком: ❌ —
   это границы и дефекты, питающие §11, а не крах гипотезы.
4. **Таксономия гипотез** (§6) по направлению, приоритету (P0–P3) и критичности.
5. **Индустриальный анализ** (§10): какие нормы применимы к Хабу, какие — нет и
   почему, с явной фиксацией границ (не legal compliance).
6. **Входные данные для 4 будущих стандартов** (§9): стандарт работы агента,
   стандарт синхронизации Hub↔project, стандарт продуктового управления,
   стандарт процесса стресс-тестирования.
7. **Найденные пробелы** (§11): 11 багов/gaps, включая 3 P1 (расхождение
   документированной модели и машинно-проверяемой формы; отсутствие OWASP-LLM
   покрытия tool-using слоя; отсутствие стандарта процесса стресс-тестирования).

**Границы.** Отчёт ничего не мигрирует, не создаёт норм и не правит поведение
агентов. Все рекомендации — кандидаты для человеческого решения.

**Прикладное дополнение (issue #398).** К базовому отчёту (issue #394) добавлен
раздел **§14** — тем же методом (Deep Think, панель из 8 экспертов, §1.2–§1.3),
но с иным *заземлением*: если §2–§13 читают **Хаб** как прообраз агентной системы,
то §14 читает **реальный работающий спок** — [`mango_ba_prompts`](https://github.com/G-Ivan-A/mango_ba_prompts)
(автоматизация бизнес-анализа Mango) — как уже действующую агентную систему и
накладывает на неё модель Хаба. §14 отвечает на 5 запросов issue #398 (логика
создания мультиагентной системы глазами команды консультантов, этапы
проектирования инфраструктуры и стека, образовательный разбор «языка индустрии»
на базе §3.4, и две **явно проверяемые гипотезы**: research base → боевые
артефакты, и различение контракт/промпт/правило + синхронизация на уровнях
проекта и продукта), даёт скелет **учебного курса для фаундера** и **инициирует
follow-up вопросы**. Три прикладные гипотезы §14 (**ПГ-1…ПГ-3**) имеют собственные
стресс-тесты и намеренно **не** переписывают счётчики §5/§6 базового отчёта.

**Корректирующее до-исследование (issue #400).** §14 заземлён на *один спок* и тем самым
рассуждал **внутри** границ проекта — что нарушает видение Хаба «действовать **вне** границ,
собирать международные практики и возвращать в проекты релевантное» (Vision). issue #400
квалифицирует применённые границы как **критическую ошибку рамки** и возвращает задачу в
до-исследование. Добавлен раздел **§15** — тем же методом (Deep Think, панель из 8 экспертов),
но с восстановленным порядком оптики: сначала **индустриальные практики создания ИИ-агентов
вне границ** (§15.1) и **структура обучающих материалов** (§15.2), затем **корреляция с
экосистемой** (§15.3), **проверка готовности** репозитория Mango к миграции (§15.4, ПГ-4),
**применимость в Mango** (§15.5) и **эволюционные варианты миграции репо→агент** (§15.6,
ПГ-5/ПГ-6, варианты V1–V4) с двумя параллельными треками (библиотека промптов по статистике ∥
план первого Агента), маршрутом фаундера «сделать Агента самому» (§15.7) и входами для RFC
(§15.8). §15 добавляет 3 прикладные гипотезы (ПГ-4…ПГ-6) и 4 варианта (V1–V4) с собственными
стресс-тестами и рейтингом, **не** переписывая счётчики §5/§6.

## 1. Scope & Context

### 1.1 Задача и режим

Источник: [issue #394](https://github.com/G-Ivan-A/hybrid-Intelligence-lab/issues/394)
— «Глобальный анализ репозитория Хаба как прообраза агентной системы», режим
`Creative` + `Deep Think`. Issue даёт **8 рекомендованных направлений** (A–H) —
не жёсткий алгоритм, а веер оптик — и требует **≥25 гипотез**, по **≥5
стресс-тестов** на каждую, симуляцию **команды экспертов**, исследование
**индустриальных норм**, **методологический вектор** как ключевой результат,
**матрицу**, **таксономию**, **рекомендации** и **входные данные для будущих
стандартов**. Требование issue «не микроменеджь как» соблюдено: метод выбран
исполнителем.

Восемь направлений issue:

| Код | Направление |
| --- | --- |
| A | Классификация и маршрутизация задач (типы, режимы, роутинг) |
| B | Процессы против операций (переиспользуемое vs разовое) |
| C | Проектное управление (Хаб) vs продуктовое управление (спок) |
| D | Минимизация галлюцинаций при максимуме свободы |
| E | Репозиторий как прообраз агентной системы |
| F | Синхронизация практик Hub↔project |
| G | Индустриальные нормы для AI-агентов |
| H | Мета-процессы (governance над governance) |

### 1.2 Метод (Deep Think)

Метод — `manual-review` + симуляция экспертной панели, по образцу
[docs/audit/2026-07-01-documentation-boundary-audit.md](../../docs/audit/2026-07-01-documentation-boundary-audit.md)
(§1.4) и
[docs/audit/2026-07-04-cross-standard-stress-tests.md](../../docs/audit/2026-07-04-cross-standard-stress-tests.md).
Пять шагов:

1. **Картирование.** Инвентаризация реальных артефактов Хаба, релевантных
   агентной системе (роли, режимы, роутинг, онбординг, валидаторы, практики,
   sync). Каждая гипотеза привязана к строке-источнику, а не к общей теории.
2. **Индустриальный бенчмарк.** Сверка с международными нормами (§3) через
   воспроизводимый source corpus. Норма, которая не применима, фиксируется с
   причиной (требование issue).
3. **Формулировка гипотез.** 27 гипотез по направлениям A–H (§4).
4. **Adversarial стресс-тестирование.** По каждой гипотезе ≥5 проверок, каждая —
   попытка *опровергнуть* гипотезу (не подтвердить). Вердикт: ✅ держит / ⚠️
   держит частично (есть пробел) / ❌ не держит в текущем виде.
5. **Синтез.** Матрица (§5), таксономия (§6), методологический вектор (§7),
   рекомендации (§8), входы для стандартов (§9), индустриальный анализ (§10),
   реестр пробелов (§11).

### 1.3 Симуляция команды из 8 экспертов

Deep Think выполнен как параллельная работа восьми экспертных оптик. Каждая
находка помечена оптикой-источником в стресс-тестах §4.

| # | Эксперт | Фокус поиска | Основные находки (кратко) |
| --- | --- | --- | --- |
| Э1 | **Архитектор агентных систем** | Отображение «артефакт Хаба → компонент агентной системы»; полнота роутера, памяти, boot | Хаб покрывает 6 из 7 канонических компонентов агентной системы; пробел — слой наблюдаемости/трейсинга прогонов (`runs/` не введён). |
| Э2 | **Специалист по управлению AI-рисками** | Соответствие NIST AI RMF, ISO/IEC 23894, EU AI Act; risk-tiering | Хаб имеет map/measure/manage/govern-петлю как practice, но risk-tier не встроен в форму issue; escalation-триггеры сильны. |
| Э3 | **Инженер минимизации галлюцинаций** | Readback, evidence-linking, fail-closed, temperature | «Спрашивай, не выдумывай» + evidence Rule 5 — сильный анти-галлюцинационный контур; слабое звено — нет автоматической проверки, что claim реально имеет source-ссылку. |
| Э4 | **Специалист по управлению процессами** | Процесс vs операция; lifecycle-граф; anti-inflation | Граф `observation→research→RFC→ADR→standard→run` — процессный хребет; разрыв: `runs/` (операционный слой) в Хабе отсутствует, операции не логируются. |
| Э5 | **Архитектор стандартов** | R/A/A/Report, contract vs standard, IL-0..IL-3, L1–L4 | Content-over-path — сильный принцип; риск — две «лестницы» (IL vs L) легко спутать; нет стандарта процесса стресс-тестирования. |
| Э6 | **Специалист по синхронизации экосистемы** | Hub↔spoke, Smart Sync, need-to-know, practice exchange | Human-in-control sync — корректная модель agent-to-agent распространения; частичный граф проекта не обновляется молча. |
| Э7 | **UX-исследователь агентных интерфейсов** | Онбординг как boot, handover-промпт, режимы как «ручки» | Онбординг-протокол = сильный boot-UX (preflight→readback→clearance); Hybrid и Deep Think не выбираются в форме issue — разрыв модель↔интерфейс. |
| Э8 | **Индустриальный аналитик** | Применимость норм; чего у нас нет; LangGraph/CrewAI/AutoGen/SK | Хаб реализует «governance-first», а не «framework-first»; оркестрационные фреймворки применимы как язык описания, но не как обязательная зависимость. |

### 1.4 Границы (что это НЕ)

- **Не решение.** Ни одна гипотеза не становится нормой без RFC/ADR и слова
  человека ([AI_GOVERNANCE.md](../../AI_GOVERNANCE.md), правило 4;
  [docs/adr/2026-06-adr-002-artifact-document-methodology.md](../../docs/adr/2026-06-adr-002-artifact-document-methodology.md)
  строки 153–164).
- **Не миграция.** Отчёт не создаёт каталогов, не переносит файлов, не вводит
  `runs/`/`kb/` (расхождение с ADR-001 сознательно сохранено —
  [ADR-007](../../docs/adr/2026-07-adr-007-hub-root-structure.md) строки 67–70).
- **Не legal compliance.** Индустриальные нормы переводятся в *proportional
  practices*, а не в сертификацию
  ([research/hub/2026-06-12-international-ai-governance-practices.md](2026-06-12-international-ai-governance-practices.md),
  раздел «Separation From Legal Compliance»).
- **Не Portal.** Хаб остаётся archetype A (Governance & Knowledge Hub); «Hub !=
  Portal» ([ADR-007](../../docs/adr/2026-07-adr-007-hub-root-structure.md)
  строка 62).

### 1.5 Замечание о размещении отчёта (routing decision)

Issue называет целевой путь `docs/research/YYYY-MM-DD-...`. Каталога
`docs/research/` в репозитории **нет**, и его создание нарушило бы строгое
правило: research-файлы размещаются только в `research/<domain>/`
([research/README.md](../README.md) строки 14–18;
[standards/research-standard.md](../../standards/research-standard.md);
[ADR-002](../../docs/adr/2026-06-adr-002-artifact-document-methodology.md)
строка 76 — «Research Хаба → `research/{domain}/YYYY-MM-DD-name.md`»). По правилу
CONTRIBUTING.md явная Meta issue имеет приоритет, **только если не нарушает hard
rule**; при нарушении фиксируется конфликт. Поэтому отчёт размещён по
методологически корректному пути
`research/hub/2026-07-04-hub-as-agent-system-global-analysis.md`, а расхождение
зафиксировано здесь как осознанное routing-решение. Это сам по себе пример
работы направления A (маршрутизация по содержанию, а не по буквальному пути) —
см. гипотезу H1.

## 2. Опорная модель: Хаб как агентная система

Прежде чем формулировать гипотезы, зафиксируем каркас, на который они
опираются. Это не новая доктрина, а свод уже принятых в репозитории решений.

### 2.1 Среда работы агента ⟂ Источник контекста

Онбординг-протокол разделяет два понятия
([ai-rules/agent-onboarding-protocol.md](../../ai-rules/agent-onboarding-protocol.md)):
**«Среда работы агента»** (чат / LLM-интерфейс, рабочая память сессии) и
**«Источник контекста»** (репозиторий, долговременная память). Агент **не
«живёт» в репозитории** — он читает его в рабочую память при старте. Это прямой
аналог архитектуры памяти агентных систем: *working memory* (контекстное окно) +
*long-term memory* (внешнее хранилище).

### 2.2 Жёсткое правило ролей: Человек > Команда Q > Исполнитель

- **Человек** — финальные решения по смыслу, целям, публикации, лицензии
  ([AI_GOVERNANCE.md](../../AI_GOVERNANCE.md) правило 4, строки 97–98).
- **Команда Q** — совещательный арбитр/комментатор (persona на базе Qwen.ai);
  её выводы — вход, а не решение; её галлюцинации явно допускаются как риск
  ([docs/analysis/2026-07-04-hub-migration-and-root-structure-plan.md](../../docs/analysis/2026-07-04-hub-migration-and-root-structure-plan.md)).
- **Исполнитель** — агент, готовящий черновики/миграции/проверки внутри scope
  задачи и governance-правил
  ([AI_GOVERNANCE.md](../../AI_GOVERNANCE.md) строки 81–86).

Это резонирует с внешней нормой «держи человека в петле» (12-factor agents,
`ext-002`; [Anthropic — человек как контрольная точка]) — см. §3, §4/H12–H16.

### 2.3 Карта соответствия «артефакт Хаба → компонент агентной системы»

Ключевая таблица исследования. Она показывает, что Хаб уже реализует 6 из 7
канонических компонентов агентной системы — и где пробел.

| Компонент агентной системы | Артефакт(ы) Хаба | Статус | Источник |
| --- | --- | --- | --- |
| **Boot / инициализация** | `ai-rules/agent-onboarding-protocol.md` (executable, entrypoint): governance→context→readback→clearance | ✅ реализован | ai-rules/agent-onboarding-protocol.md |
| **Роутер задач** | 5-шаговый конвейер: интерпретировать→тип→контракты→режим→выполнить | ✅ реализован (как research-модель) | research/hub/2026-07-02-task-execution-modes-research.md:86–90 |
| **Регулятор автономии** | Operating Mode: Structured / Creative / Hybrid (+ Deep Think модификатор) | ⚠️ частично (Hybrid/Deep Think не в форме issue) | glossary.md:48; task.yml:58–68 |
| **Долговременная память** | Репозиторий: `standards/`, `practices/`, `research/`, `docs/` | ✅ реализован | ADR-002; repo-model.md |
| **Инструменты / скиллы** | Исполнимые контракты (IL-0..IL-2), `practices/agent-work/skills-as-reusable-workflows` | ✅ реализован | ADR-002:62–67; practices/agent-work/ |
| **Guardrails / тесты** | `tools/validate-*.sh`, fail-closed Structured mode, hard bans | ✅ реализован | AI_GOVERNANCE.md:124–136; tools/ |
| **Наблюдаемость / трейс прогонов** | `runs/` — **не введён** в Хабе | ❌ пробел | ADR-007:67–70 |

Вывод Э1: Хаб — почти полный прообраз агентной системы; единственный
структурный пробел — слой наблюдаемости (логирование операционных прогонов),
сознательно отложенный по анти-инфляции. Этот пробел — не дефект, а точка на
методологическом векторе (§7): он откроется, когда появится операционная боль.

## 3. Индустриальные нормы (направление G)

Требование issue: исследовать индустриальные нормы для AI-агентов, а норму,
которая **не применима**, — фиксировать с причиной. Хаб уже имеет собственный
source-backed корпус
([research/hub/2026-06-12-international-ai-governance-practices.md](2026-06-12-international-ai-governance-practices.md)
и `practices/ai-governance/`); ниже он сведён с независимо проверенными нормами.

### 3.1 Таблица источников (проверено)

| Норма | Издатель | Ядро паттерна | Верификация |

### 3.2 EU AI Act — проверенные факты (для §10)

- Regulation (EU) 2024/1689, «Artificial Intelligence Act»; опубликован в OJ
  **12.07.2024**, вступил в силу **01.08.2024**.
- **Четыре уровня риска:** *unacceptable* (запрещённые практики, Art. 5),
  *high-risk* (Title III), *limited risk* (прозрачность, Art. 50), *minimal
  risk* (не регулируется).
- Отдельный трек **systemic risk** для GPAI: презумпция при совокупном обучении
  > **10^25 FLOP** (Art. 51); доп-обязательства — model evaluations, adversarial
  testing, incident reporting, cybersecurity (Art. 55).
- Даты применения: Chapters I–II — 02.02.2025; GPAI-обязательства — 02.08.2025;
  общее применение — 02.08.2026; Art. 6(1) — 02.08.2027.

### 3.4 Оркестрационные фреймворки и агентные паттерны (проверено)

Ключ для направлений A, E, F. Индустрия уже выработала *язык описания* агентных
систем — Хаб полезно читать через него, но **не обязан** брать эти фреймворки как
зависимость (это остаётся продуктовым решением спока, не нормой Хаба).

- **Anthropic «Building Effective Agents»** (2024-12-19, Schluntz & Zhang) —
  различает **Workflows** («systems where LLMs and tools are orchestrated through
  predefined code paths») и **Agents** («systems where LLMs dynamically direct
  their own processes and tool usage»). Каталог паттернов (verbatim заголовки):
  *Prompt chaining*, **Routing** («classifies an input and directs it to a
  specialized followup task» — прямой аналог 5-шагового роутера Хаба, H1/H2),
  *Parallelization* (**Sectioning**, **Voting** — «running the same task multiple
  times to get diverse outputs», аналог панели Deep Think, H4), **Orchestrator-
  workers** («a central LLM dynamically breaks down tasks, delegates them to
  worker LLMs, and synthesizes their results» — аналог hub-and-spoke, H9),
  **Evaluator-optimizer** (петля «генерация↔оценка» — аналог петли обучения, H23).
  Главный принцип — «start with the simplest solution… add complexity only when
  needed» — дословно совпадает с anti-inflation Хаба (H7, H21).
- **LangGraph** (LangChain) — граф состояний с явными узлами/рёбрами; **CrewAI**
  — роли и «команды» агентов; **AutoGen** и **Semantic Kernel** объединены в
  **Microsoft Agent Framework** (GA 1.0, апрель 2026). Дают словарь «state graph
  / roles / planner / skills», совпадающий с моделью Хаба «режимы + контракты +
  скиллы».
- **Интероперабельность:** **MCP** (Model Context Protocol; передан в
  Agentic AI Foundation под Linux Foundation, дек. 2025) — стандарт подключения
  инструментов/контекста; **A2A** (agent-to-agent) — обсуждаемый, но **не
  адоптированный** отраслевой стандарт; **IETF AgentID** — только индивидуальные
  drafts, **нет принятого RFC**. Для §10: A2A/AgentID фиксируем как *ещё не
  применимые* (нет стабильного стандарта), но идейно созвучные Smart Sync (H22).

### 3.5 Нормы риск-менеджмента и этики проектирования (проверено)

- **NIST AI RMF (AI 100-1, янв. 2023):** четыре функции **GOVERN / MAP / MEASURE
  / MANAGE** (GOVERN — сквозная, «infused» в остальные три); структура 4 функции →
  19 категорий → 72 подкатегории. Проецируется на Хаб как issue/PR-петля риск-
  профиля (уже `nist-ai-rmf-profile-loop.md`).
- **NIST AI 600-1 (Generative AI Profile, июль 2024):** тот же скелет + 12
  GAI-специфичных рисков и 200+ действий. Прямо релевантны Хабу: **Confabulation**
  (галлюцинации → H12–H13), **Information Integrity** (заземление Rule 5 → H13),
  **Human-AI Configuration** (роль человека → §2.2), **Value Chain and Component
  Integration** (внешние источники/практики → LLM03 Supply Chain).
- **ISO/IEC 42001:2023:** AI management system (политики, роли, процессы,
  continual improvement) — спроецирован как «management loop» (`iso-42001-
  management-loop.md`, H24).
- **ISO/IEC 23894:2023:** guidance по управлению AI-рисками (проекция ISO 31000 на
  ИИ). Релевантна как «risk treatment / safe-default» (H16). Хаб применяет её *дух*
  (пропорциональный риск-менеджмент), не сертификацию.
- **IEEE 7000-2021** «Model Process for Addressing Ethical Concerns during System
  Design» (Value-Based Engineering): элицитация этических ценностей → **Ethical
  Value Requirements** → **Value Register** → трассируемость через дизайн. Для
  Хаба — редкий пример *процесса* превращения ценностей в требования; частично
  созвучен пути «research → RFC → ADR → norm» (H5), но Хаб не ведёт формальный
  Value Register (кандидат §10, не пробел).

### 3.6 Программы безопасности ИИ (проверено)

- **Constitutional AI** (Anthropic, arXiv:2212.08073, дек. 2022): обучение
  безвредности через самокритику-и-ревизию против письменной «конституции» (SL),
  затем **RLAIF** («RL from AI Feedback»). Хаб-аналог: свод норм (`AI_GOVERNANCE`,
  standards) как «конституция» + Команда Q как совещательный критик — но
  **человек**, не AI, даёт финальный сигнал (жёстче, чем RLAIF; §2.2).
- **Anthropic RSP / ASL** (Responsible Scaling Policy, сент. 2023; текущая v3.3,
  май 2026): **AI Safety Levels ASL-1..4**, пороги capability по модели BSL →
  усиление safeguards при росте возможностей. Хаб-аналог: `anthropic-capability-
  thresholds.md` — усиление контроля при появлении tool-autonomy/external-writes
  (H25, направление D/E).
- **OpenAI Preparedness Framework v2** (апр. 2025): три Tracked Categories (Bio/
  Chem, Cybersecurity, AI Self-improvement), пороги **High/Critical**, governance
  через Safety Advisory Group → Leadership → Board SSC. Хаб-аналог: пред-релизные
  проверки (валидаторы, стресс-тесты) + пост-релизная обратная связь
  (`openai-evaluation-and-feedback-loop.md`, H20, H27).

## 4. Гипотезы (27) и стресс-тесты

Обозначения вердиктов: **✅** — гипотеза держит нагрузку; **⚠️** — держит
частично, есть пробел/оговорка; **❌** — в текущем виде не держит (найден дефект
или граница). Каждый стресс-тест — попытка *опровергнуть* гипотезу. В скобках —
оптика эксперта (Э1–Э8), поднявшего находку.

### Направление A — Классификация и маршрутизация задач

#### H1. Пятишаговый конвейер маршрутизации — рабочий роутер агентной системы

> **Гипотеза.** Эмпирически сложившийся конвейер «интерпретировать → определить
> тип → найти контракты → выбрать режим → выполнить»
> ([research/hub/2026-07-02-task-execution-modes-research.md](2026-07-02-task-execution-modes-research.md):86–90)
> — готовый прообраз роутера задач агентной системы: детерминированный, по
> минимальной формулировке, с человеком как переопределяющей инстанцией.

| # | Стресс-тест (попытка опровержения) | Вердикт |
| --- | --- | --- |
| 1.1 | Задача без глагола («Хаб как агентная система») — роутер определит тип? Тема ≠ сигнал; нужен deliverable (Test 1 источника: 44%→94% при добавлении действия). (Э1) | ⚠️ |

| 1.3 | Тип меняется по ходу (research→standard) — роутер устареет? Режим = функция текущей формулировки, не прошлого состояния (В3). (Э1) | ✅ |

**По приоритету (что двигать первым):**

| Приоритет | Критерий | Гипотезы/пробелы |
| --- | --- | --- |
| **P1 — критично** | Дефект в несущем свойстве, дешёвая правка, высокий эффект | B1 (H3), G2 (H20/H25), C1 (H4/H27) |
| **P2 — важно** | Структурный пробел, требует человеческого решения о слое | A2 (H5/H6/H17), A1 (H1/H2), D1 (H12/H13/H20), G4 (H19/H26), F1 (H22) |
| **P3 — под триггер** | Полезно, но вводить только под операционную боль | D2 (H4/H8/H16), G3 (H24), F2 (H23) |

**По критичности для агентной системы:**

- **Несущие (must-keep):** роль-иерархия (§2.2), fail-closed (H16), anti-inflation
  (H7), evidence-linking (H13), онбординг-boot (H19) — без них система теряет
  безопасность/управляемость.
- **Дифференцирующие (advantage):** режим-как-регулятор-автономии (H3), Deep Think
  (H4), temperature-ручка (H15), governance² (H26) — то, что делает Хаб именно
  *агентной* системой, а не просто репозиторием.
- **Незрелые (build-next):** Smart Sync (H22), петля-метрика (H23), операционная
  память runs/ (H5/H6/H17) — точки роста на методологическом векторе (§7).

## 7. Методологический вектор (ключевой результат)

> **Вектор:** *«От описательного репозитория знаний — к исполнимой агентной среде
> с человеком в петле, растущей строго под операционную боль».*

Это не список правил и не план миграции, а **направление**: любое будущее решение
о Хабе можно проверить вопросом «двигает ли оно нас вдоль вектора, не нарушая
роль-иерархию и anti-inflation?». Вектор раскладывается на **6 осей движения** —
каждая имеет замер «где мы сейчас» и «направление», но не обязывает к конкретной
дате или форме (это остаётся за человеком).

| Ось | Сейчас | Направление (куда, не «как») | Тянут гипотезы | Тормозит (осознанно) |
| --- | --- | --- | --- | --- |
| **1. Executability** (описание → исполнение) | executable-контракты есть; исполнение частью процедурное | больше машинно-проверяемого, меньше «прочитай и сделай» | H18, H20 | anti-inflation (H7) |
| **2. Routing** (интуиция → детерминизм) | 5-шаговый роутер как research-модель | роутер, устойчивый к verb-less/unknown входу; режимы в машинной форме | H1–H3 | риск over-engineering |
| **3. Memory** (2 слоя → 3 слоя) | working ⟂ long-term | добавить эпизодический слой (runs/) — когда появится боль | H5, H6, H17 | anti-inflation до боли |
| **4. Guardrails** (форма → смысл) | форму проверяет CI | семантические guardrails (заземление, консистентность) | H12, H13, H16 | стоимость/ложные срабатывания |
| **5. Sync** (раздача → петля) | Hub→spoke раздача норм | двунаправленная петля с трассируемостью и человеком-арбитром | H22, H23 | сложность, риск дрейфа |
| **6. Meta** (прецедент → стандарт) | governance² по прецедентам | стресс-тест-как-стандарт; проверка непротиворечивости корпуса | H26, H27 | бюрократизация |

**Инвариант вектора (не сдвигается никогда):** на *каждой* оси человек остаётся
финальной инстанцией (Rule 4), а рост слоёв подчинён anti-inflation. Движение
вдоль вектора **никогда** не должно ослаблять «Человек > Команда Q > Исполнитель»
или вводить артефакт без операционной боли — иначе это движение *поперёк* вектора,
даже если технически «улучшает» систему.

человеческого RFC/ADR (ADR-002:44). Отсортированы по приоритету (§6); все проходят
проверку вектором (§7) и инвариантом (роль-иерархия + anti-inflation). Форма
(«как») намеренно не диктуется — фиксируется только *что* и *зачем*.

**P1 — критично, дёшево, высокий эффект (кандидаты «сейчас»):**

## 9. Входные данные для будущих стандартов

Ниже — **сырьё** для четырёх стандартов, не сами стандарты (стандарт рождается
только через человеческий RFC/ADR, ADR-002:34–46). Для каждого зафиксированы:
scope, инварианты (что нельзя нарушать), кандидатные правила, открытые вопросы для
человека.

### 9.1. Стандарт агентной работы (agent-work standard)

- **Scope:** как агент интерпретирует задачу, выбирает режим/автономию, ведёт
  онбординг, оформляет результат. Опора: `ai-rules/agent-onboarding-protocol.md`,
  `practices/agent-work/`.
- **Инварианты (не нарушать):** роль-иерархия «Человек > Команда Q > Исполнитель»
  (Rule 4); «спрашивай, не выдумывай»; fail-closed по умолчанию (H16); онбординг
  до исполнения (H19).
- **Кандидатные правила:** 5-шаговый роутер (H1) как норма интерпретации;
  тип-по-глаголу (H2); Operating Mode как регулятор автономии (H3) с машинной
  формой; token-budget/скиллы (H21) как контракт ресурса; промпт/контекст —
  версионируемые артефакты (ext-002, ext-011).
- **Открытые вопросы (человеку):** дефолт роутера на unknown-type (A1)? Нужен ли
  эпизодический журнал прогонов (A2)? Где граница между «онбординг проверил вход»
  и «нужен runtime-guardrail» (D1)?

### 9.2. Стандарт синхронизации Hub↔project (sync standard)

- **Scope:** как норма/практика Хаба попадает в HTOM-проект и как обратная связь
  возвращается. Опора: H22 (Smart Sync), H23 (петля обучения),
  практики ai-governance.
- **Инварианты:** need-to-know / контекст-изоляция (H10); проект не обязан брать
  всё (proportionality, H24); человек-арбитр конфликта; никакой утечки секретов
  между Hub и project (Rule 6).
- **Кандидатные правила:** «least materialized» раздача (только нужное); трасса
  синка (что, когда, кем, версия нормы); двунаправленность (проект → практика →
  норма) через явный RFC; evaluation+feedback на каждый синк (OpenAI-паттерн,
  practices/ai-governance).
- **Открытые вопросы:** механика обратного потока (F1) — pull проекта или push
  Хаба? Как мерить, что практика «прижилась» (F2)? Автослияние vs ручной арбитраж?

### 9.3. Стандарт продуктового управления (product-management standard)

- **Scope:** различение «проектное управление Хабом» vs «продуктовое управление»
  (issue-направление C). Опора: H9 (hub-and-spoke=orchestrator-workers), H11 (HTOM
  vs Spoke), архетипы A–D (ADR-001).
- **Инварианты:** Hub = Архетип A (Governance & Knowledge), не Portal (ADR-007);
  «2FA»-обоснование подкаталога (операционная боль И повторяемость, ADR-001:84);
  anti-inflation как продуктовый фильтр фич (H7).
- **Кандидатные правила:** оркестратор-воркеры как модель координации проектов;
  различать «процесс» (переиспользуемый) и «операцию» (разовую) при планировании
  (H6); критерий продуктовой фичи = операционная боль + повторяемость + не ломает
  роль-иерархию.
- **Открытые вопросы:** где кончается «governance Хаба» и начинается «продукт для
  HTOM»? Нужен ли отдельный продуктовый бэклог или достаточно issue/PR-петли?

### 9.4. Стандарт процесса стресс-тестирования (stress-testing standard)

- **Scope:** как проверять гипотезу/норму/артефакт на устойчивость. Опора: H27,
  этот отчёт (§4), `docs/audit/2026-07-04-cross-standard-stress-tests.md` (≥2-й
  прецедент).
- **Инварианты:** каждый тест даёт вердикт из фиксированного набора (✅ держит /
  ⚠️ держит с оговоркой / ❌ вскрывает границу); ❌ = *граница*, а не крах; каждый
  ❌ картируется в реестр пробелов с приоритетом; вывод — за человеком.
- **Кандидатные правила:** ≥5 тестов на гипотезу; экспертная панель (Deep Think,
  8 перспектив) как источник разнообразия тестов; матрица-свод (§5) как
  обязательный артефакт; единый словарь вердиктов; консолидация ❌ в приоритези-
  рованные пробелы (§11).
- **Открытые вопросы:** критерий «достаточно тестов»? Как защититься от
  само-галлюцинации панели (D2)? Единое fail-состояние для всех типов проверок?

## 10. Индустриальный анализ (применимость норм)

Принцип отбора — **пропорциональность, не сертификация**. Хаб превращает нормы в
соразмерные операционные паттерны и *не* претендует на юридический комплаенс
(`research/hub/2026-06-12-international-ai-governance-practices.md:47–52`,
«Separation From Legal Compliance»). Ниже — что применимо, что уже спроецировано
и — по требованию задачи — **что НЕ применимо и почему**.

### 10.1. Уже спроецировано как практики (опора на корпус 2026-06-12)

| Норма | Проверенный факт | Применимость | Спроецировано как |
| --- | --- | --- | --- |
| NIST AI RMF (AI 100-1, янв 2023) | 4 функции GOVERN/MAP/MEASURE/MANAGE, 19 категорий, 72 подкатегории | Высокая: цикл риск-профиля ложится на issue/PR-governance | `practices/ai-governance/nist-ai-rmf-profile-loop.md` |
| ISO/IEC 42001:2023 | AI Management System (AIMS), owner + review cadence | Высокая: не даёт нормам/шаблонам устаревать | `practices/ai-governance/iso-42001-management-loop.md` |
| EU AI Act — Reg (EU) 2024/1689 | OJ 12.07.2024, в силе 01.08.2024, 4 уровня риска + GPAI systemic-risk >10²⁵ FLOP, общее применение 02.08.2026 | Средняя: как **триаж** риска, не как правовой комплаенс | `practices/ai-governance/eu-ai-act-risk-tiering.md` |

### 10.3. Явно НЕ применимо — и почему (требование задачи)

Фиксируем осознанно, чтобы отсутствие не читалось как упущение:

| Норма / элемент | Почему НЕ применимо сейчас | Условие пересмотра |
| --- | --- | --- |
| **OWASP LLM08:2025** (Vector & Embedding Weaknesses) | В Хабе нет RAG / векторного хранилища — риск физически отсутствует | Появление RAG-компонента в Хабе или spoke |
| **Anthropic ASL-уровни** (ASL-1..4) | Хаб не обучает и не хостит frontier-модели; пороги ASL измеряют возможности модели, а не репозитория | Хаб начнёт управлять обучением/весами моделей |
| **OpenAI Preparedness Tracked Categories** (Bio/Chem, Cyber, AI Self-improvement) | Категории про катастрофические возможности frontier-модели; у Хаба таких возможностей нет | Тот же триггер, что для ASL |
| **RLAIF / обучение по Constitutional AI** | Хаб не ведёт обучения модели — берём только концепт «конституции принципов», не метод | Появление собственного дообучения |
| **A2A-протокол + IETF AgentID** | **Нет принятого стандарта** (draft); в Хабе нет multi-agent wire-протокола | Стандарт примут И появится межагентный обмен → watch-item (R10) |
| **Orchestration-фреймворки как зависимость** (LangGraph, CrewAI, AutoGen→MS Agent Framework 1.0, апр 2026; Semantic Kernel) | Хаб провайдер-агностичен; «репозиторий-как-агентная-система» — концепт, не порт фреймворка | Продуктовое решение о рантайме (человек) |
| **MCP как принятая зависимость** (передан в Agentic AI Foundation/Linux Foundation, дек 2025) | Executable-контракты = «tools» концептуально; протокол MCP не внедрён | Появление MCP-сервера/клиента в экосистеме |
| **Полный EVR-процесс IEEE 7000** | Тяжеловесно и непропорционально текущему масштабу (anti-inflation, H7) | Рост до продукта с явными stakeholder-ценностями |
| **Сертификация / правовой комплаенс любой нормы** | Хаб даёт пропорциональные паттерны, не сертификацию (Separation From Legal Compliance) | Продуктовое решение проекта о формальном аудите |

### 10.4. Вывод индустриального анализа

Ни одна применимая норма не требует «изобретать велосипед»: **8 из 9** релевантных
семейств уже спроецированы или прямо ложатся на существующие практики. Единственный
структурный дефицит — консолидация security-правил под карту OWASP-LLM/SAIF (G2,
R2). Всё «не применимое» отклонено по одной из двух причин: (1) риск физически
отсутствует (нет RAG, нет обучения frontier-моделей, нет wire-протокола) либо
(2) непропорционально масштабу (anti-inflation). Это — здоровый признак: система
не тянет нормы «для галочки», а берёт ровно то, что снимает реальную боль (H24).

## 11. Реестр пробелов

Консолидация всех 17 ❌ и ⚠️-кластеров §4 в **11 различимых пробелов**. Каждый —

### 14.1 Логика создания мультиагентной системы глазами консультантов по внедрению ИИ (кейс БА Mango)

Оптика раздела — **группа бизнес-консультантов по внедрению ИИ** (сводит Э1, Э4,
Э8). Ключевой сдвиг рамки: заказчик просит «автоматизировать бизнес-аналитика», а
консультант переформулирует задачу как **индустриализацию когнитивной функции** —
превращение работы БА в гибридную команду человек+ИИ с воспроизводимыми артефактами.
Не «один большой промпт-ассистент», а **организованная система** маленьких,
стандартизованных, маршрутизируемых и снабжённых человеческими воротами единиц.

**Логическая цепочка консультанта (заземлена на спок):**

1. **Декомпозиция функции.** Разложить работу БА на **когнитивные операции** (13
   операций: `ingestion`, `understanding`, `validation`, `modeling`,
   `solution_design`, `documentation`, `quality`, `research`, `governance` +
   `impact_analysis`, `reverse_requirements`, `risk_analysis`, `release_readiness`)
   и **процессы** (9 процессов БА) — `mango_ba_prompts/docs/taxonomy.md`. Это язык,
   на котором дальше говорит вся система.
2. **Привязка к отраслевому своду знаний.** Операции маппятся на **BABOK Guide v3** и
   ISO — `docs/taxonomy.md §4`. Система не «ad-hoc», а укоренена в профессиональной
   норме БА (аналог индустриального бенчмарка §3, но на предметном уровне).
3. **Паттерны для классов задач.** Для каждого воспроизводимого класса — **паттерн**
   (переиспользуемый метод, 8 обязательных полей) — `mango_ba_prompts/patterns/` +
   `standards/pattern-standard.md`. Паттерн отвечает на «как решать этот класс».
4. **Исполнимые промпты по режимам запуска.** Каждый паттерн материализуется
   **промптами** в режимах `stepwise` / `oneshot` / `legacy` —
   `mango_ba_prompts/prompts/` + `standards/prompt-standard.md`. Промпт — «агент» в
   узком смысле: исполнимая инструкция LLM с `temperature`, `id`, `status`.
5. **Маршрутизация.** **Центральный роутер** выбирает цепочку процесс→паттерн→промпт —
   `mango_ba_prompts/docs/ba-processes/00-index.md` (+ executable-companion). Это
   прямой аналог Anthropic **Routing** (§3.4) и 5-шагового роутера Хаба (H1).
6. **Человеческие ворота.** Операции класса `человек`/`гибрид` не закрываются без
   evidence и human gate — «gate rules» в `00-index.executable.md`. Инвариант
   Человек > Команда Q > Исполнитель (§2.2) держится на предметном уровне.
7. **Операционная память.** Каждый прогон фиксируется — `mango_ba_prompts/runs/` +
   `standards/runs-contract-standard.md`. Это тот самый слой наблюдаемости, который
   Хаб осознанно отложил (A2, §11): спок его **уже имеет**.
8. **Research-якорь.** Всё опирается на research-базу Хаба через единственный мост —
   `mango_ba_prompts/docs/hub-research-dependencies.md` (детально — §14.4).

**Вывод консультанта.** Автоматизация БА ≠ большой промпт; это **оркестраторная
мультиагентная топология** (§3.4, H9): `00-index` — оркестратор/роутер, `prompts/` —
исполнители-воркеры, `runs/` — наблюдаемость, `standards/` — контракты, человек —
владелец ворот. Ценность консультанта не в «написать промпты», а в том, чтобы задать
**систему координат** (таксономия→паттерн→промпт→прогон), в которой промпты
воспроизводимы, маршрутизируемы и проверяемы.

- **Язык индустрии (§3.4).** «State graph / nodes» (LangGraph), «roles / crews»
  (CrewAI), «skills» (Semantic Kernel → Microsoft Agent Framework), различение
  **Workflows** vs **Agents** (Anthropic). «Агент» — это версионируемый артефакт
  (узел графа / скилл / роль), а не запущенный сервер.
- **В споке.** Агенты «живут» как единицы `(паттерн + промпт)` в
  `mango_ba_prompts/patterns/` и `prompts/`, адресуемые реестром
  `docs/ba-processes/00-index.md`. Единица-«агент» = исполнимый промпт, привязанный к
  паттерну и операции таксономии, с frontmatter (`id`, `temperature`, `status`).
- **Вывод для обучаемого.** «Где живут агенты» в отраслевом языке = **субстрат
  версионируемых артефактов**, а не работающий процесс в памяти. У Mango это
  Markdown-файлы в `prompts/` под git — их можно ревьюить, версионировать, откатывать.

#### Шаг B. «Как агенты запускаются»

- **Язык индустрии (§3.4).** Entrypoint/boot; planner/router выбирает путь; MCP
  подключает инструменты/контекст; Anthropic **Routing** — «classifies an input and
  directs it to a specialized followup task».
- **В споке.** Запуск = boot (`governance/agent-onboarding-protocol.md`) → готовый
  handover-промпт (`AI_SESSION_HANDOVER_PROMPT.executable.md`) → выбор маршрута в
  роутере (`00-index` route selection) → режим запуска (`stepwise`/`oneshot`/`legacy`) →
  **каскадная загрузка контекста** (`standards/cascading-context-loading-standard.md`:
  сначала `executable`-слой, эскалация к `full` только по триггеру).
- **Вывод для обучаемого.** Запуск = `boot(правила) → route(задача→процесс→цепочка
  промптов) → load(минимальный исполнимый контекст) → run → record(runs/)`. Это тот же
  boot-конвейер, что онбординг Хаба (H19), но доведённый до предметного маршрута.

#### Шаг C. «Как агенты работают по правилам»

- **Язык индустрии (§3.3, §3.5).** Guardrails, constitutional constraints, fail-closed,
  least-privilege (OWASP LLM Top-10:2025, Google SAIF, Anthropic Constitutional AI).
- **В споке — правила слоятся:**
  - **Governance-правила** (кто решает, границы) — `AI_GOVERNANCE.md` (роли, правила 1–9).
  - **Исполнимые правила** (что агент делает по шагам) — `AI_QUICK_RULES.md`
    («ПРИМЕНЯЙ, НЕ ПЕРЕСКАЗЫВАЙ»; **Structured = fail-closed**, Creative = обоснованный
    обход с фиксацией в PR).
  - **Ворота** (где обязателен человек) — «gate rules»: без evidence + human gate нельзя
    ставить `covered`/`validated`/`approved`/`released`.
  - **Жёсткие запреты** (никогда): нет `research/` в HTOM-команде, нет секретов,
    нет несанкционированной публикации/удаления.
- **Вывод для обучаемого.** «Работа по правилам» = **слоёная** система: governance
  (кто решает) / исполнимые правила (что делать сейчас) / ворота (где подписывает
  человек) / жёсткие запреты (черта, которую не переходят). Всё держится на инварианте
  Человек > Команда Q > Исполнитель (§2.2).

#### Шаг D. «Баланс между документированием процессов и практикой без детального описания»

Это **ключевое** напряжение обучения. Слишком много документации — агент тонет и
галлюцинирует; слишком мало — теряется воспроизводимость и ворота.

- **Язык индустрии (§3.4).** Anthropic: «start with the simplest solution … add
  complexity only when needed»; 12-factor **own-your-prompts** (ext-002); anti-inflation.
- **Как спок решает баланс — три механизма:**
  1. **Каскадная загрузка контекста** (`standards/cascading-context-loading-standard.md`):
     каждый документ разбит на **Level 1 `executable`** (тонкий слой практики — маршруты,
     правила, триггеры; грузится всегда первым) + **Level 2 `full`** (полный процесс для
     человека — rationale, история, таблицы; грузится **только** по сработавшему
     `Escalation trigger`) + **Level 3 `raw`** (первоисточники). Правило жёсткое: «LLM не
     выбирает слой свободно; если триггер не сработал — не грузи более тяжёлый файл “на
     всякий случай”».
  2. **Anti-Inflation:** артефакт/документ создаётся только под доказанную операционную
     боль (`AI_GOVERNANCE.md` правило 7).
  3. **Честная маркировка пробелов:** где промпта ещё нет, реестр **не** имитирует
     покрытие, а помечает «Выполняется вручную / Требуется разработка промпта»
     (`00-index.executable.md`).
- **Вывод для обучаемого.** Баланс — не «документировать всё» и не «без документации», а
  **ярусный**: по умолчанию система работает на тонком исполнимом слое практики;
  разворачивает полный документированный процесс **только** при проверяемом триггере; и
  никогда не плодит документацию без реальной боли. Документация существует, чтобы к ней
  *эскалировать*, а не чтобы читать её на каждом прогоне.

### 14.6 Скелет учебного курса для фаундера: агентная система как бизнес-процесс

Оптика — Э7 (обучение/UX) + Э4 (процессы). Цель issue #398 — дать фаундеру
возможность вести **создание агентной системы как бизнес-процесс**:
этапы→процессы→подпроцессы→операции→артефакты. Ниже — скелет курса, где каждый модуль
= этап §14.2, разложенный до артефакта (по образцу онтологии Артефакт↔Процесс↔Операция

| --- | --- | --- | --- |
| M0. Геном и режимы | Установить правила игры | Определить роли; выбрать режимы (Structured/Creative/Hybrid+Deep Think); задать fail-closed и ворота | `AI_GOVERNANCE.md`, `AI_QUICK_RULES.md` |
| M1. Декомпозиция функции | Разложить работу | Выделить операции; сгруппировать в процессы; привязать к BABOK/ISO | `docs/taxonomy.md`, `standards/ba-ontology.md` |
| M2. Research-мост | Подключить знания | Завести единственный реестр `research_dep`; запретить локальный `research/` | `docs/hub-research-dependencies.md` |
| M3. Контракты | Задать форму | Написать `pattern-`/`prompt-`/`runs-`контракты (RFC 2119) | `standards/*.md` |
| M4. Паттерны | Описать методы | Для каждого класса задач — паттерн (8 полей) | `patterns/*/README.md` |
| M5. Промпты | Сделать исполнимое | Реализовать промпты по режимам запуска; executable-companion | `prompts/*.md`, `*.executable.md` |

## 15. До-исследование (issue #400): Хаб вне границ проектов — индустриальные практики, обучение и готовность репозитория к миграции репо→агент

### 15.0 Мандат issue #400: критическая ошибка границ и как §15 её исправляет

**Диагноз (почему задача #398 возвращена в до-исследование).** §14 выполнен тем же
методом Deep Think, но с *заземлением на один спок* — `mango_ba_prompts`. Это дало
точную и полезную проекцию, **но нарушило видение Хаба**. Vision (L1) фиксирует, что
Хаб **собирает международные практики AI-работы** и **не ограничивается границами
проектов**:

> «Хаб анализирует связанные проекты, выделяет общие базисы, **собирает международные
> практики** AI-работы в команде и возвращает в проекты исследования и структурированные
> практики» ([docs/vision.md](../../docs/vision.md), раздел *Vision*).
>
> «Хаб предоставляет набор практик под проекты и обеспечивает обмен опытом: **внешний —
> из международных источников**, внутренний — между проектами экосистемы»
> ([docs/vision.md](../../docs/vision.md), раздел *Core Innovation*).
>
> «Хаб **не заменяет проекты и не ограничивает их локальные решения**; он помогает
> переводить опыт из одного мира в другой так, чтобы смысл не терялся»
> ([docs/vision.md](../../docs/vision.md), раздел *Метафора «Связь миров»*).

Из этого следует **критическая ошибка §14**: бо́льшая часть до-исследования была проведена
*применительно к проекту* Mango, то есть Хаб рассуждал **внутри** границ спока, тогда как
его роль — действовать **вне** границ: сперва собрать индустриальные/международные практики
как самостоятельный актив, и **только затем** проверить их релевантность и применимость к
проектам. issue #400 квалифицирует применённые границы как критическую ошибку рамки и
требует **переставить порядок оптики**: сначала мир → потом экосистема → потом проект.

**Что делает §15 (корректирующее до-исследование).** §15 **не удаляет** §14 (проекция на
спок остаётся полезным входом), а **надстраивает** над ним недостающий внешний контур и
разводит слои по отдельным разделам, как требует issue #400:

1. **отдельно** исследует индустриальные практики создания ИИ-агентов **вне границ проекта**
   (§15.1);
2. **отдельно** исследует возможную структуру обучающих материалов по созданию ИИ-агентов
   (§15.2);
3. **отдельно** проверяет, насколько прежнее исследование закрыло корреляцию (готовность)
   репозитория Mango к миграции в проект-агент (§15.4);
4. **отдельно** анализирует эволюционные варианты миграции репозитория в ИИ-агента —
   с приоритетом достижения результатов, двумя параллельными треками (§15.6).

Плюс **обязательные по правилу issue #400** отдельные разделы: индустриальные практики
(§15.1), корреляция с экосистемой (§15.3), применимость в Mango (§15.5), маршрут фаундера
(§15.7). Всё — на уровне бизнес-логики **и** с погружением в техническую специфику, чтобы
фаундер-новичок мог сам пройти путь и сделать Агента.

**Режим, метод, команда (наследуются).** Режим — `Creative` + `Deep Think` (§1.1); метод —
пятишаговый Deep Think (§1.2); команда — симуляция панели из **8 экспертов** (§1.3). Каждая
находка помечена оптикой-источником (Э1…Э8). Токены не экономятся (требование issue #400):
объём отчёта увеличен более чем на 70%.

**Границы (Rule 4/5/6).** §15 — **исследование, не решение**: даёт глубокий анализ и
варианты с рейтингованием и рекомендациями лучших **для перехода к RFC**, но ни один
вариант не становится нормой без RFC/ADR и слова человека
([AI_GOVERNANCE.md](../../AI_GOVERNANCE.md) правило 4). Индустриальные факты уровня дат/версий

документированной практики** и там, где требуется точная сверка, помечены `[требует
верификации]`. Счётчики §5/§6 базового отчёта **не переписываются**; §15 вводит собственные
прикладные гипотезы **ПГ-4…ПГ-6** и **варианты миграции V1…V4** с собственными

### 15.1 Индустриальные практики создания ИИ-агентов (вне границ проекта) [обязательный раздел]

> **Оптика раздела:** Э1 (архитектор агентных систем), Э8 (индустриальный аналитик), с
> подключением Э2/Э3/Э4 по темам рисков, галлюцинаций и процессов. **Заземление —
> международное**: это самостоятельный внешний контур, читаемый *до* и *независимо* от
> экосистемы Хаба. Проекция на Хаб/Mango вынесена в §15.3 и §15.5, чтобы не повторить
> ошибку границ §14.

**Установка для фаундера (бизнес-логика).** «Создать ИИ-агента» в индустрии **не** значит
«написать один умный промпт». Это значит — **построить систему**, у которой есть: цель и
границы автономии, память, инструменты, маршрутизация, ограждения (guardrails), оценка
качества и наблюдаемость прогонов, и человек в петле. Ниже — карта того, что мир уже
выработал по каждому из этих элементов, с рейтингом «что брать новичку первым».

#### 15.1.1 Что индустрия называет «жизненным циклом создания агента»

Публичные материалы ведущих вендоров и сообществ сходятся на цикле из шести ступеней
(агрегировано; частные формулировки различаются, суть — одна):

| Ступень | Вопрос, на который отвечает | Публичный ориентир |
| --- | --- | --- |
| 1. Use-case & autonomy scoping | «Какую задачу автоматизируем и сколько автономии даём?» | Anthropic «Building Effective Agents» (workflow vs agent); EU AI Act risk-tiering (§3.2) |
| 2. Design (архитектура + паттерн) | «Один агент или несколько? Какой паттерн оркестрации?» | §3.4 (Anthropic patterns; LangGraph/CrewAI/MAF) |
| 3. Build (промпты, инструменты, память) | «Из чего собран агент как исполнимый артефакт?» | 12-factor agents (ext-002); MCP (§3.4) |
| 4. Evaluate (offline evals) | «Как измерить, что агент решает задачу, до релиза?» | OpenAI evals; LLM-as-judge; agent benchmarks |
| 5. Deploy & observe (AgentOps) | «Как видеть прогоны, стоимость, дрейф в проде?» | OpenTelemetry GenAI semantic conventions `[требует верификации версии]`; tracing-платформы |
| 6. Govern & improve (петля) | «Как безопасно улучшать по статистике прогонов?» | NIST AI RMF Manage/Measure (§3.5); Evaluator-optimizer (§3.4) |

**Вывод для обучаемого.** Цикл — **петля, а не водопад**: ступени 4–6 возвращают на 2–3.
Ключевая мысль для новичка: *агент существует настолько, насколько его прогоны измеряются и
наблюдаются* — без ступеней 4–5 «агент» остаётся демо, а не системой. Это ровно тот слой
(`runs/`/наблюдаемость), который Хаб осознанно отложил (§2.3, A2), а спок Mango уже имеет
(§14.1 п.7).

#### 15.1.2 Референс-архитектура одного агента (канонические компоненты)

Свод §2.3 (7 компонентов агентной системы) в индустриальном языке, с добавлением того, чего
в §2.3 не было развёрнуто (память-эпизоды, планировщик, наблюдаемость):

| Компонент | Индустриальный термин | Что делает | Мин. форма для новичка |
| --- | --- | --- | --- |
| Цель/политика | system prompt + policy | задаёт роль, границы, режим автономии | 1 системный промпт + список «нельзя» |
| Планировщик | planner / router | разбивает задачу, выбирает следующий шаг | 5-шаговый роутер (§H1) или явный граф |
| Рабочая память | context window / scratchpad | держит текущий ход рассуждения | сама сессия LLM |
| Долговременная память | knowledge store / RAG | факты, документы, прошлые решения | папка Markdown под git (как у Mango) |
| Инструменты | tools / function calling / MCP | действия во внешнем мире | 1–3 функции с чёткой сигнатурой |
| Ограждения | guardrails / policy enforcement | не дают выйти за границы | fail-closed + human gate |
| Наблюдаемость | tracing / run log | фиксирует каждый прогон и его исход | `runs/RUN-XXXX/` (как у Mango) |

**Вывод для обучаемого.** Новичку **не** нужно сразу строить все семь на максимуме. Правило
индустрии (Anthropic, дословно, §3.4): «начни с простейшего решения; добавляй сложность
только когда нужно». Минимальный жизнеспособный агент = *системный промпт + 1–3 инструмента +
fail-closed + лог прогонов*. Всё остальное — по доказанной боли (Anti-Inflation, §H7).

#### 15.1.3 Паттерны проектирования: workflow → agent (спектр автономии)

Индустрия различает не «агент/не агент», а **спектр автономии**. Слева — предсказуемые
workflow (LLM в заранее заданных путях кода), справа — автономные агенты (LLM сам ведёт
процесс и выбор инструментов). Каталог паттернов (расширяет §3.4):

| Паттерн | Автономия | Когда применять | Риск | Рейтинг для 1-го агента |
| --- | --- | --- | --- | --- |
| **Prompt chaining** | низкая | задача раскладывается на фиксированные шаги | почти нет | ⭐⭐⭐ (старт) |
| **Routing** | низкая-средняя | вход надо классифицировать и направить | ошибка классификации | ⭐⭐⭐ (старт) |
| **Parallelization (sectioning/voting)** | средняя | независимые под-задачи / нужен консенсус | стоимость×N | ⭐⭐ |
| **Orchestrator-workers** | средняя-высокая | под-задачи неизвестны заранее | сложность отладки | ⭐⭐ (2-й шаг) |
| **Evaluator-optimizer** | средняя | есть чёткий критерий качества для петли | зацикливание | ⭐⭐ (для зрелости) |
| **ReAct (reason+act)** | высокая | нужны итеративные вызовы инструментов | галлюцинация действий | ⭐ (позже) |
| **Plan-and-execute** | высокая | длинные многошаговые цели | накопление ошибки | ⭐ (позже) |
| **Reflection / self-critique** | высокая | нужна самопроверка чернового ответа | стоимость, ложная уверенность | ⭐ (позже) |

**Вывод для обучаемого.** Дорога новичка идёт **слева направо**: начни с *chaining+routing*
(предсказуемо, дёшево, легко отлаживать), переходи к *orchestrator-workers* и
*evaluator-optimizer* только когда задача этого требует. Полная автономия (ReAct,
plan-and-execute) — **не** начало пути: она максимизирует и ценность, и класс рисков
(Excessive Agency, LLM06, §3.3). Это прямо согласуется с fail-closed-дефолтом Хаба (§H16).

#### 15.1.4 Память и знания: RAG, эпизоды, дрейф

- **Working vs long-term** (§2.1 уже вводит это разделение). Индустрия добавляет тип
  долговременной: **семантическая** (факты/документы, обычно RAG поверх векторного индекса)
  и **эпизодическая** (журнал прошлых прогонов/решений — у Mango это `runs/`).
- **RAG (retrieval-augmented generation)** — базовый способ заземлить агента на корпус без
  дообучения модели. Ключевой риск — LLM04 (data poisoning) и LLM08 (vector weaknesses),
  §3.3. Для новичка важно: RAG нужен **не всегда** — если корпус мал и помещается в
  контекст (как онтология БА Mango), можно обойтись каскадной загрузкой файлов (§14.3.D)
  вместо векторного слоя.
- **Дрейф знаний.** Практика «research по ссылке, не копией» (мост
  `hub-research-dependencies.md`, §14.4) — индустриальный ответ на дрейф: единый источник +
  SHA-пиннинг вместо расходящихся копий.

**Вывод для обучаемого.** Память — это **выбор дешевизны против свежести**: контекст-окно
дёшево, но забывчиво; RAG свеж, но добавляет инфраструктуру и класс рисков. Новичку:
начни с файлов-под-git + каскадной загрузки; вводи векторный RAG только под доказанную боль
«корпус не помещается / поиск по смыслу обязателен».

#### 15.1.5 Инструменты и интеграции: function calling, MCP, least-privilege

- **Function/tool calling** — модель возвращает структурированный вызов, среда исполняет.
  Дизайн инструмента (чёткое имя, схема аргументов, узкие права) важнее самого промпта.
- **MCP (Model Context Protocol)** — открытый стандарт подключения инструментов/контекста;
  передан в Agentic AI Foundation под Linux Foundation (дек. 2025, §3.4). Для новичка — способ
  не изобретать интеграции: инструмент, обёрнутый в MCP-сервер, переиспользуется разными
  агентами.
- **Least-privilege / SAIF** (§3.6, Google SAIF): агенту дают **минимум** прав; write/delete
  и внешние действия — под human gate. Это прямая проекция инварианта Человек > Команда Q >
  Исполнитель (§2.2) на слой инструментов.

**Вывод для обучаемого.** «Дать агенту инструмент» = «дать право действовать в мире».
Правило: сначала **read-only** инструменты (поиск, чтение), действия с побочными эффектами —
позже и за воротами. У Mango это уже так: промпты производят *черновики артефактов*, а
статусы `covered/validated/approved/released` ставит человек (gate rules, §14.3.C).

#### 15.1.6 Оценка качества (evals): офлайн, LLM-as-judge, онлайн

Мир сходится: **без evals агент неуправляем**. Три слоя:

| Слой | Что делает | Инструментальный ориентир | Аналог у Mango |
| --- | --- | --- | --- |
| **Offline (датасет)** | прогон на фиксированном наборе кейсов до релиза | OpenAI evals; golden sets | ручные приёмочные кейсы паттерна |
| **LLM-as-judge** | модель-судья оценивает ответ по рубрике | rubric-based scoring | Команда Q как совещательный критик (§2.2) |
| **Online (прод-метрики)** | качество/стоимость/латентность на живых прогонах | tracing-платформы, run stats | `runs/` (журнал прогонов) |

- **Риск LLM-as-judge** — судья наследует смещения модели; митигируется рубрикой,
  калибровкой на человеческих метках и правилом «судья — вход, не решение» (§2.2).
- **Agent benchmarks** (публичные наборы для tool-use/веб-задач) полезны как *язык
  сравнения*, но редко покрывают предметную БА-специфику — их роль для Mango вторична.

**Вывод для обучаемого.** Eval — это **тест агента**, как unit-тест для кода. Минимум для
первого агента: 5–10 «золотых» кейсов процесса + рубрика приёмки + правило «`draft→canonical`
только после ≥1 зафиксированного прогона» (Mango уже это делает, §14.4 ПГ1.3). Именно evals
превращают «изменения по статистике прогонов» из лозунга в механизм (см. Трек A, §15.6).

#### 15.1.7 Наблюдаемость и AgentOps/LLMOps

- **Трейсинг прогона** — записать каждый шаг агента (промпт, вызовы инструментов, ответы,
  стоимость, латентность). Складывается индустриальный стандарт семантики трейсов —
  **OpenTelemetry GenAI semantic conventions** `[требует верификации текущего статуса/версии]`.
- **Управление промптами** — промпт как версионируемый артефакт (SPDD ext-011, 12-factor
  own-your-prompts ext-002, §3.1): id/status/version/temperature во frontmatter (Mango и Хаб
  уже так делают).
- **Run statistics** — агрегаты по прогонам (доля успешных, стоимость, где падает) — топливо
  петли улучшения (NIST Measure/Manage, §3.5).

**Вывод для обучаемого.** AgentOps = «DevOps для агентов»: без журнала прогонов нельзя
ответить «почему промпт дал такой результат» и нельзя менять библиотеку **по статистике**.
Для Mango слой уже заложен (`runs/`, `runs-contract-standard.md`) — задача миграции не
*создать* его, а *наполнить и начать считать* (Трек A, §15.6).

#### 15.1.8 Ограждения и безопасность

Свод §3.3 (OWASP LLM Top-10:2025), §3.6 (SAIF, Constitutional AI, RSP) в практическую
чек-лист-форму для новичка:

| Класс защиты | Мин. практика | Норма |
| --- | --- | --- |
| Вход (prompt injection) | не доверять внешнему контексту как инструкции | LLM01 |
| Выход (bad output) | валидатор формы перед применением | LLM05; `tools/validate-*` |
| Автономия (excessive agency) | fail-closed + human gate на действия | LLM06; §2.2 |
| Секреты | hard ban на секреты/приватные данные | LLM02; Rule 6 |
| Происхождение (supply chain) | проверять источник импортируемых практик/данных | LLM03 |
| Расход (unbounded) | token/step budget | LLM10; skill-token-budget |

**Вывод для обучаемого.** Безопасность агента — **слоёная** (§14.3.C), а не один «фильтр».
Новичку хватит стартового набора: не-доверяй-входу, валидируй-выход, действия-за-воротами,
никаких-секретов, бюджет-на-прогон. Пробел, вскрытый §3.3/H25: у Хаба нет **единого**
OWASP-LLM-чек-листа — это кандидат-вход для RFC (§15.8).

#### 15.1.9 Фреймворки-инструменты: сравнение и «что брать новичку»

Расширяет §3.4. Важно (правило Хаба): фреймворк — **язык описания и опция спока, не норма
Хаба**. Сравнение как ориентир, не как предписание:

| Фреймворк | Сильная сторона | Кривая входа | Когда брать |
| --- | --- | --- | --- |
| **Anthropic patterns** (не фреймворк, а гайд) | минимализм, «code-first» | низкая | всегда — как мышление |
| **LangGraph** | явный граф состояний, контроль | средняя | когда нужен детерминизм и ветвление |
| **CrewAI** | роли/команды агентов «из коробки» | низкая-средняя | быстрый multi-agent прототип |
| **Microsoft Agent Framework** (AutoGen+SK, GA 1.0 апр. 2026, §3.4) | enterprise, skills, интеграции | средняя | .NET/Azure-контур |
| **OpenAI Agents SDK / Assistants** | тесная связка с моделями OpenAI | низкая | если стек уже OpenAI |
| **LlamaIndex** | сильный RAG/данные | средняя | data-heavy агенты |
| **DSPy** | программирование промптов, авто-оптимизация | высокая | когда evals — центр, нужна оптимизация |
| **Pydantic AI** | типобезопасность выходов (Python) | низкая | Python-стек, строгие схемы |

**Вывод для обучаемого.** Для первого агента фреймворк **не обязателен**: Anthropic-паттерны
+ прямые вызовы модели + Markdown-артефакты (стек Mango: `stack: python`, §14.2) закрывают
80% задачи. Фреймворк вводится под конкретную боль (детерминизм → LangGraph; multi-agent →
CrewAI; строгие схемы → Pydantic AI), а не «на старте».

#### 15.1.11 Международные плейбуки создания агентов: сравнение подходов (вне границ)

issue #400 требует «собрать международные практики». Ниже — сопоставление **публичных
плейбуков** ведущих вендоров и сообществ: не «кто лучше», а **какую доктрину каждый несёт** и
что из неё берёт новичок. (Точные названия документов/дат — уровень публичной практики; где
критично — наследуется из §3, уже «проверено».)

| Источник плейбука | Ядро доктрины | Что берёт новичок | Что НЕ навязывает |
| --- | --- | --- | --- |
| **Anthropic — «Building Effective Agents»** (§3.4) | «workflow vs agent»; начинай с простейшего; композиция паттернов, а не фреймворк | мышление «спектр автономии», 5 паттернов | конкретный фреймворк/SDK |
| **OpenAI — Agents/Preparedness** (§3.1, §3.6) | pre-release evals + post-release feedback; system cards; tracked risk categories | дисциплину «evals до релиза» | привязку к одной модели как норму |
| **Google — SAIF** (§3.6) | безопасность агентных инструментов/данных/действий; least-privilege | security-контроль на слое инструментов | тяжёлый enterprise-процесс на старте |
| **Microsoft — Agent Framework** (§3.4) | skills/plugins, enterprise-интеграции, оркестрация | словарь «skills + orchestration» | .NET/Azure как обязательность |
| **LangChain/LangGraph** (§3.4) | явный граф состояний; контроль потока как код | детерминизм и ветвление, когда нужны | граф там, где хватает chaining |
| **CrewAI** (§3.4) | роли/команды агентов «из коробки» | быстрый multi-agent прототип | сложную оркестрацию на старте |
| **HumanLayer — 12-factor agents** (ext-002) | владей промптами/контекстом/control-flow как кодом; человек в петле | «промпт-как-код», HITL | «магию» скрытых абстракций |
| **Hugging Face — Agents Course** (§15.2.1) | открытая практика на ноутбуках | воспроизводимость, руки | один «правильный» стек |
| **NIST/ISO/EU** (§3.2, §3.5) | риск-профиль, management-system, tiering по риску | язык управления и рисков | сертификацию как условие старта |

**Наблюдение (Э8).** Плейбуки **сходятся** в четырёх точках, несмотря на разных авторов:
(1) начинай с минимальной автономии; (2) промпт — версионируемый артефакт; (3) evals и
observability — не опция, а условие управляемости; (4) человек держит ворота на действиях.
Эти **четыре инварианта** и есть «международная практика», которую Хаб призван собрать и
вернуть в проекты — и которую §2–§13 уже независимо реконструировал внутри репозитория (§7).

**Наблюдение (Э2, риски).** Плейбуки **расходятся** там, где начинается автономия и внешние
действия: SAIF/NIST/EU нагружают этот участок контролем, тогда как «build-first»-курсы часто
его пропускают. Для новичка вывод: **бери мотивацию из build-first, но границы автономии — из
security-плейбуков**; это ровно гибрид B×C из §15.2.2.

#### 15.1.12 Стоимость, латентность и маршрутизация моделей (экономика агента)

Тема, которую §14 не затрагивал, но без которой «агент как система» неполон.

- **Стоимость прогона** = токены×цена модели × число шагов/вызовов. Паттерны с высокой
  автономией (ReAct, plan-execute) множат вызовы → множат стоимость; отсюда рейтинг §15.1.3
  «слева направо» экономически обоснован, не только по риску.
- **Латентность** — длинные цепочки/оркестрация увеличивают время ответа; для интерактивной
  БА-валидации это UX-фактор.
- **Model routing** — практика направлять простые под-задачи на дешёвую/быструю модель, а
  сложные — на сильную. Прямой аналог роутинга задач (§H1), но на слое **выбора модели**, не
  процесса. Для новичка — **не** первый шаг (Anti-Inflation), но мощная оптимизация под
  доказанную боль «дорого/медленно».
- **Token/step budget** — ограничитель расхода (LLM10, §3.3; skill-token-budget) — единственный
  экономический контроль, который стоит вводить **сразу** (дёшево, предотвращает runaway).

**Вывод для обучаемого.** Экономика — часть дизайна, а не «потом»: выбор паттерна (§15.1.3)
задаёт и риск, **и** стоимость. Минимум для первого агента — **budget на прогон**; model
routing и кэширование — оптимизации второй волны.

#### 15.1.13 Координация нескольких агентов (когда одного мало)

Расширяет H9 (orchestrator-workers) индустриальным языком — как ориентир для **будущего** L4,
не для старта.

| Механизм координации | Что решает | Риск | Индустр. ориентир |
| --- | --- | --- | --- |
| **Оркестратор + воркеры** | центральный делит задачу, синтезирует | узкое место оркестратора | Anthropic Orchestrator-workers (§3.4) |
| **Роли/команда (crew)** | специализация агентов по ролям | рост стоимости, координационный шум | CrewAI (§3.4) |
| **Граф состояний** | детерминированные переходы между агентами | сложность графа | LangGraph (§3.4) |
| **A2A-протоколы** | агент-агент обмен | **нет принятого стандарта** (§3.4) | A2A/AgentID — «ещё не применимо» |
| **Blackboard/общая память** | агенты пишут в общий контекст | гонки, need-to-know (H10) | классический AI-паттерн |

**Вывод для обучаемого.** Multi-agent — это **умножение** и ценности, и стоимости, и
поверхности рисков. Правило §15.2.3 (L2→L4): один агент с прогонами должен быть закреплён
прежде, чем вводить координацию. Для Mango это означает: сначала V2 (один процесс), и лишь
затем — оркестрация по нескольким процессам (§15.6.3 V3 как *будущее*, не старт).

#### 15.1.14 Каталог типовых отказов и анти-паттернов создания агентов (чек-лист «чего не делать»)

Индустрия описывает создание агента не только «как надо», но и через **повторяющиеся отказы**.
Их полезно собрать явно: для новичка это самая быстрая защита (Э7), для проекта — карта рисков
(Э2). Каждый отказ дан с ранним признаком и противоядием, уже присутствующим в геноме Хаба.

| Отказ / анти-паттерн | Как проявляется | Ранний признак | Противоядие (уже в экосистеме) |
| --- | --- | --- | --- |
| **Big-prompt («один промпт на всё»)** | нечитаемый, нетестируемый мега-промпт | промпт растёт, а `runs/` пусты | декомпозиция процесс→паттерн→промпт (§6, M1) |
| **Autonomy-first** | агенту сразу дан широкий доступ/автономия | нет списка «нельзя», нет ворот | Structured/fail-closed на старте (§2, §15.1.3) |
| **Eval-last** | качество меряют «на глаз», в конце | нет golden-set, нет рубрики | evals как ранний трек (§15.1.6, M7) |
| **Orchestration-too-early** | мульти-агент до отладки одного | графы/роли без прогонов одного агента | V2 thin slice, L2→L4 (§15.1.13, §15.2.3) |
| **Knowledge-copy (дрейф)** | research скопирован в проект | локальный `research/` в споке | knowledge по ссылке (§15.1.4, ПГ3.5) |
| **Excessive Agency** | агент сам ставит финальный статус | автопростановка `validated` | human gate (LLM06, §2.2, Ш5) |
| **Prompt-injection незакрыт** | внешний текст управляет агентом | нет фильтра входа/least-privilege | OWASP-LLM чек-лист (§15.1.8, RFC-D) |
| **Silent-drift** | качество тихо деградирует | метрики не собираются | observability/`runs/` (§15.1.7, RFC-G) |
| **Folders-on-grow** | структура «на вырост» без боли | пустые каталоги/слои | Anti-Inflation + ADR (§7, §15.6.4) |
| **Optimize-on-noise** | правки промпта по 1–2 прогонам | изменения без статистики | правило ≥N прогонов (Трек A, M9) |
| **Version-drift промптов** | нет id/version, воспроизводимость потеряна | правки поверх без версии | frontmatter-контракт (§15.1, CI) |
| **Hallucinated-confidence** | уверенная выдумка на неоднозначности | нет «стоп-и-спроси» | fail-closed + RAG (§H12, §15.1.4) |

**Как этим пользоваться (Ф1).** Для фаундера это **пред-полётный чек-лист**: перед запуском
среза V2 пройти 12 строк и убедиться, что ни один анти-паттерн не активен. Для RFC — это карта
рисков реализации, дополняющая §15.8.1.

**Стресс-тест каталога (Э2 adversarial).** Возражение: «это универсальные банальности, не
привязанные к Mango». Проверка: 8 из 12 строк напрямую бьют по разрывам ПГ-4 (eval-last,
orchestration-too-early, knowledge-copy, excessive-agency, silent-drift, optimize-on-noise,
version-drift, autonomy-first) — то есть каталог **операционализирует** именно ту незакрытую
операционно-измеримую готовность, что вскрыл §15.4. Возражение снято: привязка конкретна.

**Вывод раздела 15.1.** Индустрия за 2023–2026 сошлась на **governance-first, минимализм,
измеримость**: начинай с малой автономии, версионируй промпты, логируй прогоны, держи
человека на воротах, наращивай сложность под доказанную боль. Четыре инварианта §15.1.11
(минимальная автономия / промпт-как-артефакт / evals+observability / человек на воротах)
образуют переносимое ядро «международной практики». Это **не** случайное совпадение с
подходом Хаба/Mango — это тот же вектор (§7), полученный снаружи. Корреляция разобрана
отдельно в §15.3.

### 15.2 Структура обучающих материалов по созданию ИИ-агентов (вне границ проекта) [цель Ц2]

> **Оптика раздела:** Э7 (UX/онбординг как boot), Э8 (индустриальный аналитик), Э5
> (архитектор стандартов — прогрессия компетенций как «лестница»). Как и §15.1, раздел —
> **внешний контур**: сначала как учит мир, потом (в §15.7) — маршрут конкретно для фаундера.

**Установка для фаундера (бизнес-логика).** issue #400 требует **отдельно** исследовать,
как *может быть устроена* обучающая программа по созданию ИИ-агентов — чтобы фаундер-новичок
прошёл путь и **закрепил навык**, а не просто прочитал теорию. Ниже — обзор того, как это
делает индустрия, затем три каркаса программы с рейтингом.

#### 15.2.1 Как индустрия учит созданию агентов (обзор форматов)

Публичные образовательные форматы 2024–2026 (агрегировано; конкретные названия/провайдеры —
`[требует верификации точных наименований и дат]`, но сами форматы устойчивы):

| Формат | Что даёт | Сильная сторона | Слабость для новичка |
| --- | --- | --- | --- |
| **Короткие курсы «build-first»** (напр. DeepLearning.AI short courses) | собрать работающего агента за часы | быстрый успех, «первая победа» (Vision) | легко получить демо без понимания рисков |
| **Практические курсы сообществ** (напр. Hugging Face Agents Course) | руки на фреймворках, ноутбуки | воспроизводимость, открытость | привязка к конкретному стеку |
| **Cookbook’и вендоров** (Anthropic, OpenAI) | канонические паттерны от первоисточника | авторитетность, актуальность | фрагментарность (рецепты, не система) |
| **Академии фреймворков** (LangChain/LangGraph Academy) | глубина по одному инструменту | системность внутри инструмента | риск «инструмент = мышление» |
| **Нормативные материалы** (NIST AI RMF playbook, ISO, OWASP) | язык рисков и управления | заземляет governance | сухо, не «руками» |
| **Университетские/книжные курсы** | теория агентов, планирование, память | фундамент | отстаёт от практики на 1–2 года |

**Вывод для обучаемого.** Ни один формат не самодостаточен: build-first даёт *успех без
понимания*, нормативные — *понимание без успеха*. Эффективная программа **чередует** «сделал
руками» и «понял почему», добавляя governance-слой **рано**, а не в конце (иначе новичок
привыкает к небезопасным дефолтам).

#### 15.2.2 Три каркаса учебной программы (варианты с рейтингом)

| Каркас | Логика | Плюс | Минус | Рейтинг для фаундера-новичка |
| --- | --- | --- | --- | --- |
| **A. Concept-first** (теория → практика) | сперва компоненты/паттерны/риски, потом сборка | прочный фундамент, мало «карго-культа» | медленная первая победа; риск бросить | ⭐⭐ |
| **B. Build-first** (сборка → обобщение) | сразу собрать простого агента, затем осмыслить | ранний успех, мотивация (Vision «первая победа рождает доверие») | риск закрепить небезопасные дефолты | ⭐⭐⭐ (с governance-вставками) |
| **C. Ops/governance-first** (правила → сборка → прогоны) | сперва геном/режимы/ворота, потом агент, сразу с evals/прогонами | безопасно by design, ложится на модель Хаба | психологически тяжелее для старта | ⭐⭐⭐ (лучший для экосистемы) |

**Рекомендация (для перехода к RFC).** Гибрид **B×C**: *build-first по мотивации, но с
governance-каркасом с первого модуля* — собрать простого агента на предметном срезе, но сразу
в рамке «режим→ворота→прогон». Это точно ложится на онбординг-протокол Хаба (boot-first, H19)
и на этапность Mango (§14.2). Именно этот гибрид развёрнут в §15.7 как маршрут фаундера.

#### 15.2.3 Прогрессия компетенций (лестница уровней)

Индустриальная «лестница» навыка, независимая от вендора (проекция на §14.6 модули):

| Уровень | Навык | Критерий «сдал» | Модуль Mango-курса (§14.6) |
| --- | --- | --- | --- |
| L0 | Промпт как артефакт | промпт с id/status/version, воспроизводим | M0–M1 |
| L1 | Инструменты и границы | агент вызывает 1–3 инструмента, действия за воротами | M3–M5 |
| L2 | Один агент end-to-end | процесс от boot до `runs/` на golden-кейсах | M5–M7 |
| L3 | Оценка и петля | evals + изменения по статистике прогонов | M7 + Трек A (§15.6) |
| L4 | Оркестрация | несколько агентов/воркеров под роутером | M6 (в масштабе) |
| L5 | Governance в масштабе | мульти-спок, sync, риск-профиль | M8, §15.6 Трек B |

**Вывод для обучаемого.** Компетенция строится **снизу вверх**: нельзя «перепрыгнуть» в L4
(оркестрация), не закрепив L2 (один агент с прогонами). Это дисциплинирует и рейтинг практик
§15.1.10, и выбор варианта миграции §15.6 (сначала один процесс до `runs/`, потом масштаб).

#### 15.2.4 Рекомендация структуры обучающих материалов (для RFC)

- **Формат:** гибрид B×C (build-first + governance-first), 11 модулей = M0–M10 (§14.6),
  но с добавлением **сквозного evals/observability-трека** (L3) как отдельной линии, а не
  «в конце».
- **Принцип каждого модуля:** «термин индустрии → материализация в артефакте → что делает
  обучаемый руками → критерий сдал» (формат уже опробован в §14.3).
- **Каноническая «капстоун»-работа:** провести **один** реальный процесс БА end-to-end от
  boot до `runs/` (эталонный прогон, §14.6) — это и есть точка, где фаундер «делает Агента
  сам» (требование issue #400).
- **Открытый вопрос для RFC:** где живут учебные материалы — в `education/` Хаба (уже есть
  каталог) как международно-заземлённый курс, versus как спок-специфичный гайд в Mango.
  Vision тянет к Хабу (материалы — переиспользуемый актив экосистемы). См. §15.8.

#### 15.2.5 Детальная программа: модуль за модулем (для перехода к RFC-F)

Разворачивает §14.6 (M0–M10) в **учебную** форму: у каждого модуля — цель обучения, что
делает обучаемый руками, типичная ошибка новичка и критерий сдачи. Это скелет курса, а не
курс (Rule 4): наполнение и размещение — предмет RFC-F (§15.8).

| Модуль | Цель обучения (навык) | Практика руками | Типичная ошибка новичка | Критерий «сдал» |
| --- | --- | --- | --- | --- |
| **M0. Геном и режимы** | понять «кто решает» и что такое режим автономии | выписать роли и правило Человек>Q>Исполнитель; выбрать Structured для первого агента | сразу дать агенту полную автономию | сформулированы роли, режим, fail-closed |
| **M1. Декомпозиция функции** | разложить работу на операции/процессы | взять таксономию БА, отметить 1 процесс для агента | «один большой промпт на всё» | процесс выделен, привязан к операции |
| **M2. Research-мост** | подключать знания **по ссылке**, не копией | завести `research_dep` на нужное исследование | скопировать research в проект (дрейф) | knowledge — по ссылке, локального `research/` нет |
| **M3. Контракты формы** | задать требования к артефактам (RFC 2119) | прочитать `prompt-`/`pattern-`/`runs-`стандарты | писать промпт без формы | промпт соответствует контракту |
| **M4. Паттерны** | описать переиспользуемый метод | заполнить 8 полей паттерна процесса | смешать метод и конкретный промпт | паттерн валиден |
| **M5. Промпты** | сделать исполнимое | написать промпт `stepwise` с frontmatter | забыть id/version/temperature | промпт запускается, воспроизводим |
| **M6. Роутер** | собрать выбор «процесс→паттерн→промпт» | зарегистрировать промпт в `00-index` | дублировать маппинг в промпте | маршрут резолвится централизованно |
| **M7. Прогоны + evals** | включить память и измеримость | собрать golden-set, прогнать, записать в `runs/` | менять промпт «на глаз», без метрик | есть `runs/` + % успеха по рубрике |
| **M8. Ворота** | не дать агенту решать за человека | включить human gate на финальный статус | автопростановка `validated` | статус ставит человек |
| **M9. Петля улучшения** | менять библиотеку по статистике | правка промпта по ≥N прогонам | «оптимизация по шуму» (мало данных) | ≥1 изменение обосновано статистикой |
| **M10. Дисциплина роста** | наращивать сложность под боль | ADR под каждый новый слой | «папки на вырост», ранняя оркестрация | новый слой оправдан болью+ADR |

**Сквозной трек (не «в конце»): evals & observability.** M7 и M9 не приделываются в финале —
это **отдельная линия**, идущая параллельно M3–M10 (§15.2.4). Индустриальная причина: без
измеримости остальные модули производят *недоказуемое качество* (§15.1.6). Это и есть разрыв,
вскрытый ПГ-4 (ПГ4.3), поэтому в курсе он поднят рано.

**Капстоун курса.** Провести **один** процесс («Валидация ФТ/ТЗ», §15.6.2) от M0 до M9 —
эталонный прогон, на котором фаундер «делает Агента сам» (§15.7). Капстоун = точка, где
учебная структура §15.2 и вариант миграции V2 (§15.6.3) **совпадают**: обучение и есть
первый шаг миграции.

**Вывод раздела 15.2.** Структура обучающих материалов, которую индустрия делает
разрозненно (курсы, cookbook’и, нормы), в экосистеме Хаба собирается в **единый
governance-first + build-first курс** с ранним evals-треком и капстоуном-миграцией.
Рекомендация к RFC-F: разместить курс в `education/` Хаба как международно-заземлённый актив,
переиспользуемый любым споком (Vision: практика — переиспользуемый актив).

### 15.3 Насколько до-исследование коррелирует с экосистемой Хаба [обязательный раздел, П1]

> **Оптика раздела:** Э6 (синхронизация экосистемы), Э1 (архитектор), Э8 (индустрия). Задача —
> **явно** сопоставить внешний контур (§15.1–§15.2) с видением и артефактами Хаба, и показать,
> что §15 восстанавливает нарушенную §14 рамку «сначала вне границ, потом в границы».

#### 15.3.1 Корреляция с Vision (L1): восстановление рамки

| Тезис Vision | §14 (как было) | §15 (как исправлено) |
| --- | --- | --- |
| Хаб собирает **международные** практики | практики читались *через* призму спока | §15.1–§15.2 — самостоятельный внешний контур |
| Хаб **не ограничивается** границами проектов | заземление на один спок Mango | §15.1/§15.2 заземлены на индустрию, спок — потребитель |
| Хаб **возвращает** в проекты релевантное | смешано с проекцией на спок | §15.5 — отдельный слой «что применимо в Mango» |
| Хаб не заменяет и не ограничивает проекты | риск подмены Хаба спок-логикой | §15 разводит слои мир→экосистема→проект |

**Вывод.** Ошибка §14 была не в *содержании* (проекция на спок ценна), а в **порядке и
границах оптики**: Хаб рассуждал изнутри проекта. §15 возвращает канонический порядок Vision:
*внешние источники → research/ → practices/ → проекты импортируют* (цикл обмена, docs/vision.md).

что §2.3 и §11, но полученный **снаружи внутрь** (от индустрии к Хабу), а не наоборот — то
есть §15 подтверждает выводы §2–§13 **независимым** путём, что усиливает их (триангуляция).

#### 15.3.3 Корреляция с реестром пробелов §11 (без нового параллельного реестра)

§15 **не** заводит новый реестр — все ⚠️/❌ выше сворачиваются в существующие пробелы §11:
наблюдаемость → A2/R4; отсутствие CI-заземления и evals-стандарта → D1/R6; OWASP-чек-лист →
G2 (H25); enforced-границы/теги → G4/R7; обратный sync → F1/R8, F2/R11. Новизна §15 — не новые
пробелы, а **внешнее подтверждение приоритетов**: индустрия ставит evals и observability в
центр, что повышает приоритет A2/R4 и D1/R6 как входов для RFC (§15.8).

| ПГ4.4 | Есть ли **проектный план первого агента**? Нет как артефакта: §14.6 лишь называет кандидат-процесс; выбор с критериями и план инфраструктуры отсутствуют. (Э5) | ❌ |
| ПГ4.5 | Не переоценена ли готовность из-за границ §14? Да, риск есть: заземляясь только на спок, §14 мерил готовность «изнутри», без внешнего benchmark зрелости agent-систем (§15.1.1 цикл). (Э8) | ⚠️ |
| ПГ4.6 | Согласуется ли частичная готовность с индустрией? Да: индустрия считает систему «готовой к агенту», когда замкнуты evals+observability (§15.1.6–7); у Mango они заложены, но не активированы. (Э2) | ⚠️ |

**Итог ПГ-4.** ⚠️ **Гипотеза подтверждена частично.** Прежнее исследование **закрыло
структурную** корреляцию готовности (язык, форма, исполнимые единицы, маршрутизация, ворота,
разделение слоёв — всё ✅), но **не закрыло операционно-измеримую**: (1) evals как класс
артефактов не исследованы (ПГ4.3 ❌); (2) петля «прогон→статистика→изменение библиотеки» не
замкнута/не наполнена (ПГ4.2 ⚠️); (3) проектный план первого агента отсутствует как артефакт
(ПГ4.4 ❌). Эти три разрыва — **прямая повестка §15.6** (варианты миграции) и §15.7 (маршрут
фаундера). Разрывы наследуют пробелы §11 (A2/R4 наблюдаемость, D1/R6 заземление/evals), а не
образуют новый реестр.

#### 15.5.1 Разбор «как выглядит недостающий артефакт» — evals-набор (для фаундера, Ф1)

Три P0/P1-пробела (`runs/`-статистика, golden-sets+рубрики, петля) абстрактны для новичка.
Ниже — **иллюстрация формы** недостающих артефактов на процессе «Валидация ФТ/ТЗ». Это
учебный эскиз (Rule 4): реальная форма — предмет RFC-A, здесь показан только *облик*.

- **Golden-set (что это и зачем).** Набор реальных входов «требование/ФТ» + эталонного
  вердикта валидации. Роль: без «правильного ответа» run-статистика неинтерпретируема
  (§15.1.6). Минимум для старта — 5–10 кейсов, покрывающих типовые дефекты требований
  (неоднозначность, непроверяемость, противоречие, неполнота).

- **Рубрика приёмки (эскиз критериев).** Для каждого кейса — да/нет по колонкам, дающим % успеха:

  | Критерий рубрики | Что проверяет | Провал = |
  | --- | --- | --- |
  | Дефект найден | агент выявил заложенную проблему | пропуск дефекта (ложный «ок») |
  | Тип верен | классификация дефекта совпала | неверная категория |
  | Обоснование корректно | ссылка на конкретное место требования | «галлюцинация» причины |
  | Нет ложных срабатываний | не помечены валидные требования | шум/over-flagging |
  | Формат вердикта | вывод соответствует контракту промпта | невалидная форма |

- **Запись прогона `runs/` (какие поля).** Минимальный след: `id прогона`, ссылка на кейс
  golden-set, версия промпта/паттерна, выход агента, отметки рубрики, итог «сдал/не сдал».
  Роль: это память, из которой считается % успеха и на которой видно дрейф между версиями
  (§15.1.7).

- **Петля (как замыкается).** Когда по ≥N прогонам одной версии промпта рубрика показывает
  повторяющийся провал одной колонки (например, «Обоснование корректно» проседает), это —
  **обоснованный данными** повод к правке промпта/паттерна (evaluator-optimizer, §3.4), а не
  правка «на вкус». Так библиотека меняется по статистике (Трек A), а не по шуму (§15.1.14).

**Стресс-тест эскиза (Э3 анти-галлюцинация, Э2).** Возражение: «рубрика поощряет придирки —
агент начнёт помечать валидные требования ради „находок“». Контрмера уже в рубрике: колонка
«Нет ложных срабатываний» штрафует over-flagging, а колонка «Обоснование корректно» требует
привязки к месту требования (нет привязки → провал). Баланс «находит дефекты ∥ не шумит»
встроен в рубрику. Возражение снято на уровне формы; калибровка порогов — вход RFC-A.

**Вывод раздела 15.5.** Применимое к Mango из мирового контура сводится к **измеримости и
петле улучшения** (evals + run-статистика + evaluator-optimizer) — ровно к тем разрывам,
что вскрыла ПГ-4. Структурно Mango уже готов; миграция — это **активация операционного
контура**, а не стройка с нуля. Эскиз §15.5.1 показывает: недостающие артефакты **малы по
форме** (golden-set + рубрика + поля `runs/`), но именно они переводят «качество» в число.

### 15.6 Эволюционные варианты миграции репозитория → ИИ-агент [цель Ц4]

**Рамка issue #400 (дословно переведена в конструкцию).** Приоритет — достижение результатов.

- **Трек A (обязательный):** закрыть проект **как библиотеку промптов** с контролем версий и

(«Валидация ФТ/ТЗ»), параллельно (а) наполнять его `runs/` и вводить evals (Трек A на этом
срезе) и (б) собрать на нём первого Агента (Трек B). Обоснование: максимум результата рано
(эталонный прогон + измеримая под-библиотека), минимум риска (один низкорисковый процесс с
воротами), точное соответствие Vision (внешние практики §15.1 → применены в проекте) и
Anti-Inflation (слой под доказанную боль), и это же — «капстоун» обучения фаундера (§15.7).
V1 — безопасная, но медленная альтернатива; V3/V4 — анти-паттерны (нарушают start-simplest и
порядок компетенций L2→L4).

**Явная оговорка (Rule 4).** V1–V4 и выбор процесса — **варианты и рекомендация для RFC**, а
не решение. Финальный выбор — за человеком; §15 лишь готовит вход RFC с рейтингом и рисками.

#### 15.6.4 Фазовая раскладка рекомендованного V2 (карта для RFC, не план работ)

Чтобы рекомендация V2 (§15.6.3) была проверяема и не превратилась в «большой скачок», ниже —
её разложение на **фазы с выходами и воротами**. Это карта для RFC (что за чем логически
следует и как понять «фаза закрыта»), а **не** директивный план и не сроки (Rule 4). Фазы
идут по нарастанию доказанной боли (Anti-Inflation): следующий слой открывается, только когда
предыдущий дал измеримый выход.

| Фаза | Что закрывает | Выход (артефакт) | Ворота перехода дальше | Треки A/B |
| --- | --- | --- | --- | --- |
| **Ф0. Срез выбран** | зафиксировать один процесс + режим автономии | ADR «первый агент = Валидация ФТ/ТЗ, Structured» | человек утвердил срез и «нельзя» | B (рамка) |
| **Ф1. Форма закрыта** | промпт/паттерн среза соответствуют контрактам | валидный `pattern-`/`prompt-` с frontmatter | CI (frontmatter/naming) зелёный | A (форма) |
| **Ф2. Измеримость включена** | golden-set + рубрика для среза | `evals/` + рубрика % успеха | есть ≥1 базовый прогон с числом | A∥B (evals) |
| **Ф3. Прогоны идут** | наблюдаемость: `runs/` наполняется | ≥N записей `runs/` по кейсам | % успеха посчитан, дрейф виден | A (память) |
| **Ф4. Ворота стоят** | человек на финальном статусе | включённый human-gate на `validated` | нет автопростановки статуса | B (guardrails) |
| **Ф5. Петля замкнута** | изменения библиотеки по статистике | ≥1 правка, обоснованная run-данными | изменение прошло рубрику (не по шуму) | A (улучшение) |
| **Ф6. Обобщение** | извлечь переиспользуемый паттерн для др. процессов | обновлённый `pattern-` + запись в research (по ссылке) | Q-ревью подтвердило переносимость | A∥B |

**Что осознанно НЕ входит в V2 (границы среза).** Мульти-агентная оркестрация (§15.1.13),
автономные режимы выше Structured, второй БА-процесс, собственный `research/` в споке — всё это
**отложено до доказанной боли** и последующих RFC. Явная фиксация «чего не делаем» — это
анти-паттерн-предохранитель (§15.1.3: не строить оркестратор, пока не выжат один агент).

**Стресс-тест фазовой карты (Э6 sync, Э2 risk).** Попытка опровергнуть: «фазы — это скрытый
водопад». Ответ: нет — каждая фаза даёт **самостоятельный измеримый выход** (Ф2 даёт число ещё
до Ф5), и любой срез можно остановить на Ф3, уже имея пользу (измеримая под-библиотека Трека A).
Порядок здесь — не сроки, а **зависимости по данным** (нельзя замкнуть петлю Ф5 без чисел Ф2–Ф3).
Возражение снято: карта деградирует до полезного состояния на каждой фазе, водопадом не является.

### 15.7 Маршрут фаундера: сделать первого Агента самому [формат Ф1, сквозной]

> **Оптика раздела:** Э7 (обучение/UX), Э4 (процессы), Э1 (архитектор). Формат issue #400:
> **команда консультантов учит фаундера-новичка** — на уровне бизнес-логики **и** с
> погружением в технику, чтобы он **сам** прошёл путь и закрепил навык. Ниже — маршрут-оверлей
> поверх рекомендованного варианта **V2** и процесса «Валидация ФТ/ТЗ».

**Как читать этот маршрут.** Каждый шаг = «зачем (бизнес) → что делаешь руками (техника) →
как понять, что получилось (критерий)». Маршрут кладётся на действующую инфраструктуру (Хаб +
спок Mango), а не «с нуля» (§14.6): у фаундера уже есть геном, стандарты и роутер.

| Шаг | Зачем (бизнес-логика) | Что делаешь руками (техника) | Критерий «сделал» |
| --- | --- | --- | --- |
| Ш0. Рамка автономии | решить, сколько свободы у агента | выбрать режим `Structured` (fail-closed) для валидации; записать «нельзя» | зафиксирован режим + список запретов |
| Ш1. Выбрать процесс | сфокусироваться на одной ценности | взять `fr-validation-stepwise` из `00-index` | процесс выбран, покрытие проверено |
| Ш2. Собрать golden-set | «на чём проверяем, что агент прав» | 5–10 реальных кейсов валидации ФТ + ожидаемый результат | golden-set лежит в `evals/`/паттерне |
| Ш3. Прогнать промпт | увидеть агента в деле | запустить промпт по каждому кейсу, записать прогон | каждый кейс → запись в `runs/` |
| Ш4. Оценить по рубрике | отделить «работает» от «кажется» | сравнить выход с ожиданием; отметить успех/провал | рубрика заполнена, есть % успеха |
| Ш5. Поставить ворота | не дать агенту решать за человека | статус `validated` ставит человек, не агент | gate на `validated` включён |
| Ш6. Замкнуть петлю | менять библиотеку **по данным**, не по вкусу | по статистике ≥N прогонов — правка промпта/паттерна | одно изменение обосновано run-статистикой |
| Ш7. Обобщить | понять, что построил | описать срез как эталонный прогон курса | фаундер объясняет систему своими словами |

**Погружение в технику (для новичка, без экономии деталей).**

- **«Агент» здесь** = связка `паттерн (метод) + промпт (исполнимая инструкция) + роутер
  (выбор) + runs (память прогонов) + ворота (человек)`. Это не сервер и не «ИИ, который сам
  всё решает», а **версионируемая, маршрутизируемая, измеримая единица** (§15.1.2).
- **Почему Structured/fail-closed на старте** — при неоднозначности агент **останавливается и
  спрашивает**, а не выдумывает (анти-галлюцинация, §H12); для валидации требований цена
  ложного «ок» высока (ПГ6.5).
- **Почему golden-set до прогонов** — без «правильных ответов» run-статистика неинтерпретируема;
  evals — это и есть перевод «качества» в число (§15.1.6).
- **Почему человек на воротах** — инвариант Человек > Команда Q > Исполнитель (§2.2) и LLM06
  (Excessive Agency): агент **готовит** вердикт, **подписывает** человек.
- **Что фаундер получает на выходе** — не «demo», а L2-компетенцию (один агент end-to-end с
  прогонами, §15.2.3) и работающий эталонный срез, на котором дальше растёт оркестрация (L4).

**Вывод раздела 15.7.** Маршрут даёт фаундеру **воспроизводимый путь** от рамки автономии до
замкнутой петли улучшения на одном реальном процессе. Это одновременно закрывает формат issue
#400 (учим новичка сделать Агента самому) и служит капстоуном учебной структуры §15.2.

### 15.9 Мини-глоссарий индустрии агентов (для фаундера-новичка) [формат Ф1]

Формат issue #400 требует **технического погружения** новичка. Ниже — рабочий словарь
терминов §15.1–§15.7 «на человеческом языке», с привязкой к тому, где термин уже живёт в
экосистеме. Это учебный аппарат курса (§15.2), а не новые нормы (Rule 4).

| Термин | По-простому (для фаундера) | Где в экосистеме / индустрии |
| --- | --- | --- |
| **Агент** | версионируемая+измеримая связка «метод+инструкция+выбор+память+ворота», а не «ИИ, решающий сам» | §15.1.2; спок Mango |
| **Autonomy / режим** | сколько свободы у агента; на старте — минимум (Structured) | §2, §15.1.3 |
| **Boot / геном** | стартовые правила: кто он, что можно/нельзя | genome спока, §15.1.2 |
| **Router / роутер** | «диспетчер», выбирающий процесс→паттерн→промпт | `00-index`, §15.1.2 |
| **Pattern / паттерн** | переиспользуемый **метод** (как делать), отдельно от конкретной инструкции | `pattern-*`, §6 |
| **Prompt / промпт** | исполнимая инструкция агенту с метаданными (id/version/temperature) | `prompt-*`, §15.1 |
| **Golden-set** | набор «правильных ответов», по которым меряем агента | §15.1.6, Ш2 |
| **Evals** | перевод «качества» в число (офлайн-проверки, рубрики) | §15.1.6, RFC-A |
| **Rubric / рубрика** | критерии, по которым прогон считается «сдал/не сдал» | §15.1.6, Ш4 |
| **Runs / прогоны** | журнал того, что агент реально делал (память для улучшений) | `runs/`, §15.1.7 |
| **Observability / AgentOps** | видимость: трассы, метрики, стоимость, дрейф | §15.1.7, RFC-G |
| **Guardrails / ограждения** | что агенту запрещено; фильтры входа/выхода | §15.1.8, §3.3 |
| **Human gate / ворота** | человек подписывает финальный статус, агент лишь готовит | §2.2, Ш5 |
| **RAG** | подтягивание знаний по запросу, чтобы не «выдумывать» | §15.1.4 |
| **MCP** | стандарт подключения инструментов к агенту (least-privilege) | §15.1.5, ext |
| **Hallucination / галлюцinация** | уверенная выдумка; лечится fail-closed + evals + RAG | §H12, §15.1.6 |
| **Drift / дрейф** | расхождение знаний/поведения со временем | §15.1.4, R4 |
| **Anti-Inflation** | не строить слой, пока боль не доказана | §7, §15.6.4 |
| **Orchestration** | координация нескольких агентов; **не** на старте | §15.1.13, V3=анти-паттерн |
| **Thin vertical slice** | один процесс end-to-end вместо «всего сразу» | V2, §15.6.3 |

**Зачем глоссарий в исследовании (а не в курсе).** Он фиксирует **общий словарь** между
международным контуром (§15.1) и экосистемой Хаба, снимая риск, что при переходе к RFC/курсу
термины «поплывут» (Э6 sync). Наполнение курса этими карточками — задача RFC-F.

**Adversarial-финал (Э8, попытка опровергнуть всё §15).** Три самых сильных возражения и
ответы:

1. *«§15 снова рассуждает внутри Mango, ошибка §14 не исправлена».* Ответ: контур §15
   стартует с международного контура (§15.1/§15.10) и лишь в §15.5 **возвращается** в проект —

2. *«Рекомендация V2 — это скрытое решение, а не исследование».* Ответ: V2 дана как
