# Reports: индустриальные нормы и scope стандартизации

> **Режим:** `Research` + `Deep Think`. Документ **не создаёт** стандарт Reports,
> **не выполняет** инвентаризацию текущих Reports-артефактов, **не меняет**
> структуру каталогов и **не переводит** документы между статусами. Он фиксирует
> индустриальные нормы для Reports-артефактов, проверяет гипотезы H1–H4 о scope
> стандартизации, разбирает граничные кейсы Reports ↔ Analysis/Audit и даёт
> рекомендацию по варианту стандартизации (A/B/C) для будущего human decision.
> Источник задачи —
> [issue #307](https://github.com/G-Ivan-A/hybrid-Intelligence-lab/issues/307).

> **EN abstract.** Source-backed research on industry norms for "Reports"
> artifacts (audit / statistical / general). It benchmarks Diátaxis, DITA,
> ISO/IEC/IEEE, ANSI/NISO Z39.18, ГОСТ 7.32 / 19.101, NIST AI RMF, the EU AI Act
> and how major OSS governance repositories (Kubernetes, CNCF, OpenTelemetry)
> structure reports. It tests four hypotheses about whether Reports is a separate
> artifact type, whether subtypes need separate standards, whether one Report
> standard with subtypes suffices, and where Reports overlaps Analysis/Audit. It
> recommends a hybrid scope (Variant C): one base Report standard plus light
> subtype profiles, A-shaped now and B-capable later. It deliberately does not
> create a standard or inventory current reports.

## 1. Введение

### 1.1. Зачем нужно это исследование

В видении фаундера §3
([2026-06-23-repository-structure-concept.md](2026-06-23-repository-structure-concept.md))
каталог `docs/report/` указан как базовый подкаталог `docs/` наравне с
`analysis/`, `rfc/`, `adr/`, с пометкой «(аудит, отчёт, статистика)». При этом
Reports-артефакты ещё не сверялись с индустриальными нормами, а scope их
стандартизации не определён: нужен ли один стандарт Reports с подтипами или
отдельные стандарты для каждого подтипа, и где проходит граница с Analysis и
Audit.

Это исследование готовит проверяемую основу для будущего решения. Оно
параллельно цепочке `Analysis -> RFC -> Standard` для Research/Analysis/Audit,
зафиксированной в
[2026-06-28-research-analysis-audit-inventory.md](2026-06-28-research-analysis-audit-inventory.md)
и в задачах B-024..B-034 ([governance/backlog.md](../../governance/backlog.md)),
но смотрит на **четвёртый** тип артефакта — Report.

### 1.2. Рабочие термины

«Report» здесь понимается широко: документ, фиксирующий **результат** —
что произошло, что измерено, что проверено. В разных контекстах он называется
report, отчёт, audit report, statistical report, technical report, conformity
assessment report, model card, annual report. Термин «Report-like» применяется
там, где артефакт выполняет функцию отчёта, но называется иначе.

Три подтипа из постановки задачи:

| Подтип | Рабочее определение | Индустриальный аналог |
| --- | --- | --- |
| **Аудит** (audit report) | Фиксация результата проверки на соответствие норме/контракту | ISO 19011 audit report, ISA 700 auditor's report, EU AI Act conformity assessment |
| **Отчёт** (general/technical report) | Фиксация результата выполнения задачи/работы | ANSI/NISO Z39.18, ГОСТ 7.32 (отчёт о НИР), EU AI Act Annex IV technical documentation |
| **Статистика** (statistical report) | Фиксация данных/метрик/измерений | SDMX (ISO 17369), DDI, model cards, datasheets, NIST AI RMF MEASURE outputs |

### 1.3. Что не входит в документ

Документ не создаёт `standards/report-standard.md` и не предлагает финальный
шаблон. Он не инвентаризирует текущие Reports-артефакты Хаба/споков (это
отдельная задача — backlog B-038). Он не вводит и не переименовывает каталоги.
Он даёт **рекомендацию по scope** и trade-offs; выбор и создание стандарта —
отдельная цепочка RFC → ADR → Standard с human decision.

## 2. Результаты исследования

### 2.1. BLUF

1. **Reports — отдельный тип артефакта по жанру, но почти всегда терминальный
   выход процесса.** Индустрия имеет самостоятельные нормы для отчётов
   (ANSI/NISO Z39.18, ГОСТ 7.32, ISO 19011), но эти отчёты — результат
   audit/research/measurement-процессов. Бинарная формулировка «Report — не
   output surface» опровергается: audit report **является** выходом аудита.
   Точная формулировка: Report — это **самостоятельный класс по форме** (record
   of results), который **обычно производится** как финал процесса.
2. **Единого «стандарта Reports вообще» в индустрии нет.** Нормы для отчётов
   всегда **доменно-специфичны**: аудит (ISO 19011, ISA 700), технический отчёт
   (ANSI/NISO Z39.18, ГОСТ 7.32), статистика (SDMX, DDI). Они различаются по
   обязательным секциям, evidence-модели и lifecycle.
3. **Но эти доменные стандарты — тяжёлые.** Они рассчитаны на регуляторов,
   сертификацию и межорганизационный обмен. Механический перенос трёх полных
   стандартов в небольшой knowledge-репозиторий нарушает Anti-Inflation principle
   ([governance/repo-model.md](../../governance/repo-model.md)).
4. **Доминирующий масштабируемый паттерн — «базовая структура + специализация».**
   DITA строит документ-типы через специализацию одного базового topic; ГОСТ 7.32
   задаёт единую структуру общего отчёта; governance-репозитории применяют один
   шаблон отчёта на каждое вхождение (`annual-report-YYYY.md`). Это поддерживает
   гибрид, а не три независимых стандарта.
5. **Diátaxis не содержит Reports.** Его четыре типа (tutorial, how-to,
   reference, explanation) делят **пользовательскую документацию по нужде
   пользователя**, а отчёт — это запись результата, не product-документация.
   Вывод: Reports лежат **вне** оси Diátaxis, и его нельзя «впихнуть» в один из
   четырёх типов без потери смысла.
6. **Граница Reports ↔ Analysis/Audit определяется намерением, а не каталогом.**
   Report фиксирует «что» (descriptive), Analysis объясняет «почему» (causal),
   Audit оценивает «соответствует ли норме» (normative). Этот критерий совпадает
   с content-over-path подходом из issue #288.
7. **Рекомендация: Вариант C (гибрид), реализованный как «A сейчас, B потом».**
   Один базовый стандарт Report (frontmatter, naming, lifecycle, evidentiary
   stance) + лёгкие профили подтипов как секции, с явным триггером выделения
   отдельного стандарта подтипа только при накопленной операционной потребности.

### 2.2. Сводная матрица бенчмарка

| Источник / фреймворк | Есть ли «Reports» как класс | Один стандарт или несколько | Сигнал для scope |
| --- | --- | --- | --- |
| Diátaxis | ❌ Нет (4 типа docs по нужде) | — | Reports вне оси документации → отдельный класс |
| DITA (OASIS) | ⚠️ Через специализацию базового topic | Один базовый + специализации | Сильный сигнал за гибрид (Вариант C) |
| ISO 19011 / ISA 700 | ✅ Audit report | Отдельный стандарт аудита | Подтип «аудит» имеет свою норму (H2) |
| ANSI/NISO Z39.18 | ✅ Technical report (general) | Единый стандарт общего отчёта | Подтип «отчёт» покрывается одним стандартом (H3) |
| ГОСТ 7.32 / 19.101 | ✅ Отчёт о НИР (7.32) отдельно от ЕСПД (19.101) | Единая структура общего отчёта | Reports ≠ документация; единая структура отчёта |
| SDMX (ISO 17369) / DDI | ✅ Statistical report/data | Отдельные стандарты данных/метаданных | Подтип «статистика» имеет свою норму (H2) |
| NIST AI RMF | ✅ Measurement/impact records, model cards | Профили + шаблоны (model card, datasheet) | Подтипы как шаблоны под общим каркасом |
| EU AI Act | ✅ Conformity / technical / incident reports | Отдельные обязательства по annex/article | H2 на уровне регуляции, общий каркас |
| Kubernetes / CNCF / OTel | ✅ Annual/SIG/health reports | Один шаблон на вхождение, date-stamped | Один шаблон + per-occurrence файл |

## 3. Метод и воспроизводимость

### 3.1. Корпус

Исследование опирается на три слоя:

1. **Фреймворки документации** — Diátaxis и DITA как нормы организации
   контента.
2. **Нормативные стандарты отчётов** — ISO/IEC/IEEE, ANSI/NISO, ГОСТ, SDMX/DDI,
   а также регуляторные требования NIST AI RMF и EU AI Act.
3. **Практика репозиториев** — как крупные governance/OSS-проекты физически
   хранят и шаблонируют отчёты.

### 3.2. Метод проверки гипотез

Для каждой гипотезы H1–H4 (раздел 9) применяется правило из постановки задачи:

- найти минимум 3 индустриальных примера за и против;
- если доля опровержений > 20 % — зафиксировать опровержение (или уточнение
  формулировки);
- если гипотеза выживает — принять с явными trade-offs.

Бенчмарк качественный, а не количественный scan: в отличие от issue #278/#288,
здесь нет GitHub tree-scan эксперимента, потому что вопрос — про **нормы
оформления отчётов**, а не про path signals в чужих репозиториях. Source-backed
выводы строятся на первоисточниках стандартов и официальной документации
фреймворков (раздел 14). Это сознательное ограничение метода (раздел 13).

## 4. Что индустрия считает «Report»

Сквозной признак отчёта во всех просмотренных источниках — **фиксация
результата для последующего использования и аудита**. Отчёт:

- описывает то, что уже произошло/измерено/проверено (descriptive, ретроспектива);
- адресован читателю, которому нужен факт результата, а не обучение или решение;
- имеет дату/период и автора как обязательные атрибуты (traceability);
- обычно является **выходом** процесса (исследования, аудита, измерения,
  выполнения работы), а не процессом сам по себе.

Это отделяет Report от соседних классов:

| Класс | Главный вопрос | Темпоральность | Стойка |
| --- | --- | --- | --- |
| **Report** | Что произошло / что измерено? | Прошлое (фиксация) | Descriptive |
| **Analysis** | Почему так произошло? | Прошлое → объяснение | Causal/interpretive |
| **Audit** | Соответствует ли норме? | Прошлое → вердикт | Normative |
| **RFC/ADR** | Что предлагаем / что решили? | Будущее (решение) | Prescriptive |
| **Docs (Diátaxis)** | Как пользоваться / понять? | Вне времени (продукт) | Instructive |

Важно: один документ может содержать несколько стоек (отчёт с разделом анализа),
но **доминирующая стойка** определяет тип. Это совпадает с принципом
classification-by-content из issue #288.

## 5. Бенчмарк фреймворков документации: Diátaxis и DITA

### 5.1. Diátaxis

Diátaxis (Daniele Procida / Divio) делит **пользовательскую документацию** на
четыре типа по двум осям (изучение ↔ работа, практика ↔ теория):

| Тип | Нужда пользователя |
| --- | --- |
| Tutorial | Учусь, на практике |
| How-to guide | Работаю, на практике |
| Reference | Работаю, нужна информация |
| Explanation | Изучаю, нужна теория |

В Diátaxis **нет** типа «Report». Отчёт не обслуживает ни одну из четырёх нужд:
он не учит, не помогает выполнить задачу, не является справочником API и не
объясняет концепцию. Ближайший по форме — `reference` (информация), но reference
описывает **продукт/систему** (машинно-стабильное знание), а отчёт описывает
**событие/результат** (датированная запись). Подгонять Report под Diátaxis
значит размывать ось «нужда пользователя».

**Вывод (Diátaxis):** Reports — артефакт **другого порядка**, чем product docs.
Он не должен жить в Diátaxis-таксономии. Это совпадает с §8-R4 видения фаундера:
`docs/analysis|report` — деление по «жанру/процессу», ортогональное Diátaxis по
функции. Поэтому отдельный класс Report **оправдан** и не конкурирует с
documentation taxonomy.

### 5.2. DITA

DITA (OASIS) задаёт три базовых информационных типа — concept, task, reference
(плюс troubleshooting в DITA 1.3) — и, главное, механизм **специализации**:
новые document/topic types создаются как сужения базового topic с наследованием
правил. «Report» в DITA — это не отдельный из base-типов, а **специализация**:
например, отчёт оформляется как reference-topic или как доменная специализация
с собственными обязательными секциями.

**Вывод (DITA):** ключевой архитектурный паттерн — **«один базовый контракт +
специализации»**, а не N независимых стандартов. Это самый прямой индустриальный
аргумент за Вариант C (гибрид): базовый Report-контракт + профили подтипов.

## 6. Бенчмарк нормативных стандартов: ISO/IEC/IEEE, ANSI/NISO, ГОСТ

### 6.1. Документация для пользователей ≠ отчёты

ISO/IEC/IEEE 26514 (design and development of information for users) и
ISO/IEC/IEEE 26511 (management of information for users) нормируют
**пользовательскую документацию и процесс её ведения**, а не отчёты о
результатах. То есть ISO разводит «документацию» и «отчёт» так же, как Diátaxis.
Прямого «ISO standard for reports вообще» нет.

### 6.2. Технический/общий отчёт: ANSI/NISO Z39.18 и ГОСТ 7.32

- **ANSI/NISO Z39.18** (Scientific and Technical Reports — Preparation,
  Presentation, and Preservation) — канонический стандарт **общего технического
  отчёта**: единая структура (front matter, body, back matter), правила
  оформления и сохранности. Один стандарт покрывает весь жанр «technical report».
- **ГОСТ 7.32** («Отчёт о научно-исследовательской работе. Структура и правила
  оформления») — российский аналог: единая фиксированная структура (титульный
  лист, реферат, содержание, введение, основная часть, заключение, список
  источников, приложения).
- **ГОСТ 19.101-77** (ЕСПД) перечисляет **виды программных документов** — это про
  документацию ПО, не про отчёты; он сосуществует со СИБИД (ГОСТ 7.x), где живёт
  отчёт. Это подтверждает: отчёт нормируется **отдельной** линией стандартов, а
  не внутри документации.

**Вывод:** для подтипа «отчёт» (general) индустрия использует **один** стандарт
с единой структурой (H3-сигнал для этого подтипа).

### 6.3. Аудиторский отчёт: ISO 19011 и ISA 700

- **ISO 19011** (guidelines for auditing management systems) задаёт содержание и
  структуру **audit report** отдельно от технических отчётов: scope, criteria,
  findings, conclusions, nonconformities.
- **ISA 700 series** (auditor's report) — отдельный жёсткий формат аудиторского
  заключения с обязательным мнением (opinion).

**Вывод:** подтип «аудит» имеет **собственную** норму, отличную от общего отчёта
(H2-сигнал). Аудиторский отчёт обязательно содержит **вердикт о соответствии** —
это и есть отличие Audit от чистого Report.

### 6.4. Статистический отчёт: SDMX и DDI

- **SDMX** (Statistical Data and Metadata eXchange, ISO 17369:2013) — стандарт
  обмена статданными и метаданными.
- **DDI** (Data Documentation Initiative) — описание данных, переменных,
  методологии сбора.

**Вывод:** подтип «статистика» снова имеет **свою** норму, ориентированную на
данные/метаданные, а не на нарратив (H2-сигнал). Обязательные поля здесь —
период, методология, источник данных, единицы измерения.

### 6.5. Промежуточный синтез по стандартам

| Подтип | Канонический стандарт | Обязательное ядро |
| --- | --- | --- |
| Отчёт (general) | ANSI/NISO Z39.18, ГОСТ 7.32 | Реферат, тело, выводы, источники, приложения |
| Аудит | ISO 19011, ISA 700 | Scope, criteria, findings, **вердикт о соответствии** |
| Статистика | SDMX (ISO 17369), DDI | Период, **методология**, источник данных, единицы |

Подтипы различаются **обязательным ядром секций**, но делят общий каркас
(дата/автор/scope/источники). Это ровно структура «общая база + специфичные
секции» — снова Вариант C.

## 7. Бенчмарк AI governance: NIST AI RMF и EU AI Act

### 7.1. NIST AI RMF

NIST AI RMF (Govern/Map/Measure/Manage) не вводит «report standard», но его
функция **MEASURE** производит измерительные записи и метрики, а экосистема
вокруг — структурированные отчётные шаблоны: **model cards** (Mitchell et al.),
**datasheets for datasets** (Gebru et al.). Это фактически **подтип-специфичные
шаблоны отчётов** (статистика/данные/модель) под общим governance-каркасом.

**Вывод:** AI-индустрия нормирует отчёты как **шаблоны под зонтиком**, а не как
самостоятельные тяжёлые стандарты — поддержка гибрида.

### 7.2. EU AI Act

EU AI Act требует разных отчётных артефактов по разным статьям/приложениям:

| Артефакт | Норма | Подтип |
| --- | --- | --- |
| Conformity assessment | Annex VI/VII | Аудит |
| Technical documentation | Annex IV | Отчёт (general) |
| Record-keeping / logs | Art. 12 | Статистика/события |
| Post-market monitoring / serious incidents | Art. 72/73 | Отчёт/статистика |

Разные приложения = разные «под-стандарты», но **в рамках одного регуляторного
каркаса** (общие определения, lifecycle, ответственность). Это снова гибрид: один
каркас, несколько профилей.

**Вывод:** и NIST, и EU AI Act дают сигнал, что подтипы нужны как профили под
общим контрактом, а не как изолированные стандарты.

## 8. Бенчмарк репозиториев: Kubernetes, CNCF, OpenTelemetry

В отличие от RFC/KEP (issue #278), формальный каталог `reports/` как
first-class artifact type **редок**. Наблюдаемая практика:

| Проект | Reports-практика | Паттерн |
| --- | --- | --- |
| kubernetes/community | SIG annual reports (`sig-*/annual-report-YYYY.md`) по общему шаблону | Один шаблон, per-occurrence файл, co-located с владельцем |
| cncf/toc | Annual reviews, due-diligence/project health документы | Datestamped reports, общий формат на тип |
| open-telemetry | Reports реже как каталог; чаще release/notes и issues | Слабый сигнал отдельного reports-контура |

**Вывод:** governance-репозитории не плодят отдельные стандарты под каждый тип
отчёта. Они держат **один шаблон на жанр** и создают датированный файл на каждое
вхождение (`YYYY-...-report.md`), co-located с владеющей сущностью. Это прямо
совпадает с тем, как Хаб уже ведёт `research/<domain>/YYYY-MM-DD-topic.md`.

## 9. Проверка гипотез H1–H4

### 9.1. H1 — Reports есть отдельный тип артефакта (не output surface)

**Подтверждения:** Diátaxis/ISO разводят документацию и отчёт (§5.1, §6.1);
ANSI/NISO Z39.18, ГОСТ 7.32, ISO 19011 — самостоятельные нормы именно для
отчётов; §8 показывает отчёты как отдельные датированные артефакты.

**Опровержения:** audit report **является** выходом аудита; conformity report —
выход conformity assessment; model card — выход measurement. Доля «отчёт = output
процесса» заметно > 20 %.

**Вердикт:** ⚠️ **подтверждена с уточнением формулировки.** Report — отдельный
класс **по форме/жанру** (record of results с собственным lifecycle), но по
происхождению **почти всегда output процесса** (audit/research/measurement/work).
Бинарное «не output surface» **опровергнуто**; корректно: «Report — это
самостоятельно адресуемый артефакт-результат, который обычно производится как
финал процесса».

### 9.2. H2 — Подтипы требуют отдельных стандартов

**Подтверждения:** аудит (ISO 19011, ISA 700), статистика (SDMX, DDI), общий
отчёт (Z39.18, ГОСТ 7.32) имеют **разные** обязательные ядра и нормы (§6).
EU AI Act разносит обязательства по annex/article (§7.2).

**Опровержения:** эти стандарты доменно-тяжёлые и предназначены для
регуляции/сертификации; для knowledge-репозитория они избыточны (Anti-Inflation).
DITA, NIST-шаблоны и governance-репозитории показывают **один каркас +
специализации** вместо N стандартов (§5.2, §7, §8).

**Вердикт:** ⚠️ **подтверждена на уровне индустрии, опровергнута для масштаба
Хаба.** Подтипы действительно различны по содержанию, но «отдельные **полные**
стандарты» оправданы только при регуляторном/межорганизационном масштабе. Для
Хаба различия лучше выразить **профилями подтипов**, а не тремя стандартами.

### 9.3. H3 — Один стандарт Reports с подтипами достаточен

**Подтверждения:** ГОСТ 7.32 и Z39.18 — единые стандарты целого жанра; DITA-
специализация и AI-шаблоны строятся на одном базовом контракте; governance-
репозитории используют один шаблон на жанр (§5.2, §6.2, §7, §8).

**Опровержения:** аудиторский вердикт (opinion) и статистическая методология
настолько специфичны, что «совсем одинаковый» стандарт без подтиповых секций
потеряет обязательное ядро (§6.3–6.4).

**Вердикт:** ✅ **подтверждена в гибридной форме.** Один стандарт **достаточен,
если** он несёт общий каркас **и** обязательные секции на подтип. Чистый «один
стандарт без подтиповой специфики» — недостаточен.

### 9.4. H4 — Reports пересекается с Analysis/Audit и требует чётких границ

**Подтверждения:** issue #288 уже зафиксировал перегрузку `report/analysis`
naming; §6.3 показывает, что audit report = report + вердикт; статистический
отчёт с трактовкой трендов уходит в analysis (§10). Минимум 3 примера
пересечений.

**Опровержения:** не найдено значимых — все источники подтверждают наличие
серой зоны.

**Вердикт:** ✅ **подтверждена.** Граница нужна и формулируется через
доминирующую стойку (descriptive/causal/normative), а не через каталог.

## 10. Граничные кейсы Reports ↔ Analysis/Audit

| # | Кейс | Где проходит граница | Критерий разведения |
| --- | --- | --- | --- |
| BC-1 | Статистический отчёт с трактовкой трендов | Report → Analysis | Появляется «почему» (причинность, гипотезы), а не только «что» |
| BC-2 | Отчёт о выполнении с оценкой соответствия плану | Report → Audit | Появляется сравнение с нормой/контрактом и **вердикт** |
| BC-3 | Audit report (audit как процесс → report как форма) | Audit ⊃ Report | Audit report — это Report-форма с обязательным вердиктом; тип = Audit по доминирующей стойке |
| BC-4 | Analysis с executive summary | Analysis ⊃ Report-секция | Summary — output surface внутри Analysis, не самостоятельный Report |
| BC-5 | Report-генерация из CI/measurement (model card, metrics) | Report (статистика) | Чистая фиксация данных без причинной трактовки = Report |

**Сводные критерии разведения (совместимы с issue #288):**

- **Report** = фиксация результата, доминирующая стойка descriptive («что
  произошло / что измерено»), самостоятельный датированный артефакт.
- **Analysis** = объяснение контекста, доминирующая стойка causal («почему»).
- **Audit** = проверка по норме, доминирующая стойка normative («соответствует
  ли») с обязательным вердиктом.
- **Output surface vs артефакт:** если «отчёт» — терминальная секция/рендер
  внутри Analysis/Audit без отдельного lifecycle, это **output surface**, не
  самостоятельный Report. Самостоятельный Report имеет свой frontmatter, имя и
  статус.

## 11. Варианты scope стандартизации

### 11.1. Вариант A: один стандарт Reports с подтипами

| Поле | Форма |
| --- | --- |
| Артефакт | `standards/report-standard.md` с секциями для audit/stat/general |
| Общее | frontmatter, naming `docs/report/YYYY-MM-DD-name.md`, lifecycle, evidentiary stance |
| Специфика | секции обязательного ядра на подтип в одном файле |
| Плюсы | Единая точка входа; минимум артефактов (Anti-Inflation) |
| Минусы | Файл растёт; подтипы связаны версией одного документа |

### 11.2. Вариант B: отдельные стандарты на подтип

| Поле | Форма |
| --- | --- |
| Артефакты | `audit-report-standard.md`, `statistical-report-standard.md`, `general-report-standard.md` |
| Плюсы | Чистая изоляция подтипов; прямое соответствие индустриальным доменным стандартам |
| Минусы | Дублирование общего каркаса; три lifecycle; over-engineering для текущего масштаба |

### 11.3. Вариант C: гибрид (базовый стандарт + профили подтипов)

| Поле | Форма |
| --- | --- |
| Артефакты | базовый `report-standard.md` (каркас) + лёгкие профили подтипов (секции или отдельные профильные файлы при росте) |
| Плюсы | Соответствует DITA-специализации, NIST-шаблонам и repo-практике; масштабируется; не нарушает Anti-Inflation |
| Минусы | Нужен явный триггер выделения подтипа в отдельный стандарт, иначе риск «расползания» секций |

### 11.4. Критерий выбора (из постановки задачи)

| Если индустрия… | …то вариант |
| --- | --- |
| имеет единый стандарт жанра | A |
| имеет разные стандарты на подтип | B |
| имеет общий базис + специфику | C |

Бенчмарк показал **третье**: общий каркас отчёта (Z39.18/ГОСТ 7.32 для жанра,
DITA-специализация, AI-шаблоны, repo-шаблоны) **плюс** доменная специфика
обязательного ядра подтипов (ISO 19011 для аудита, SDMX/DDI для статистики). →
Критерий указывает на **Вариант C**.

## 12. Рекомендация и критический анализ

### 12.1. Рекомендация

**Вариант C (гибрид), реализованный как «A сейчас, B потом».**

Конкретно для будущей цепочки RFC → ADR → Standard (вне scope этого документа):

1. Один базовый стандарт Report фиксирует **общий контракт**: frontmatter
   (`status`/`version`/`updated`/`temperature` + класс-поля), naming
   `docs/report/YYYY-MM-DD-name.md`, lifecycle (`draft → reviewed → published/
   canonical → superseded`), evidentiary stance (descriptive), обязательные
   дата/период/автор/источники.
2. Подтипы (аудит/статистика/отчёт) входят как **профили-секции** с обязательным
   ядром каждого (аудит → вердикт о соответствии; статистика → методология и
   источник данных; отчёт → реферат/выводы).
3. **Триггер B** (Anti-Inflation): подтип выделяется в отдельный стандарт только
   когда накопит достаточно собственных правил/повторяющихся кейсов — по тому же
   принципу, по которому Хаб откладывает `product-profile`/`education-profile` до
   повторяющейся операционной потребности.

Это даёт минимальную поверхность сейчас (как A) и путь к разделению потом (как
B), сохраняя индустриально доминирующий паттерн «база + специализация» (как C).

### 12.2. Стресс-тест рекомендации

- **Что если индустрия ошибается (тяжёлые доменные стандарты избыточны)?** Хаб и
  так не копирует их целиком; берётся только общий каркас + минимальное ядро.
  Риск низкий.
- **Где Вариант C ломается?** Если один подтип (вероятнее аудит) станет
  доминирующим и обрастёт правилами — секция станет узким местом. Решение
  предусмотрено триггером B (выделение в отдельный стандарт).
- **Trade-off A vs C:** A проще, но при росте аудиторской специфики придётся
  ретроактивно резать файл; C закладывает шов разделения сразу.
- **Trade-off B vs C:** B чище изолирует, но сегодня даёт тройное дублирование
  каркаса и три lifecycle при малом числе реальных отчётов — преждевременно.
- **Риск размывания границ (H4):** без явных критериев разведения (§10) Report-
  секции «утекут» в Analysis/Audit. Будущий стандарт должен включить
  decision-tree по доминирующей стойке.

### 12.3. Связь с экосистемой

Рекомендация согласуется с:

- видением фаундера §3 (`docs/report/` как базовый подкаталог) — Report
  подтверждён как отдельный класс (H1);
- issue #288 (content-over-path классификация) — критерии §10 переиспользуют
  доминирующую стойку;
- цепочками B-024..B-034 — Report-стандарт логично запускать **четвёртой**
  цепочкой `Analysis(inventory B-038) -> RFC -> ADR -> Standard` после/параллельно
  Research/Analysis/Audit, чтобы не закреплять текущую перегрузку `report`.

## 13. Ограничения исследования

1. Бенчмарк качественный и source-backed, без GitHub tree-scan эксперимента:
   вопрос про нормы оформления, а не про path signals. Это снижает
   количественную воспроизводимость по сравнению с issue #278/#288.
2. Доменные стандарты (ISO 19011, ANSI/NISO Z39.18, ГОСТ 7.32, SDMX) частично
   платные/закрытые; выводы опираются на их публичные scope-описания и
   официальные аннотации, а не на полный нормативный текст.
3. Repo-практика (Kubernetes/CNCF/OTel) описана по наблюдаемым паттернам
   каталогов и шаблонов, а не по исчерпывающему scan на конкретный commit.
4. Документ даёт рекомендацию по scope, **не** финальный стандарт и **не**
   инвентаризацию (это backlog B-038 и будущая RFC→ADR→Standard цепочка).
5. Граничные критерии (§10) — гипотезы для будущего стандарта, требующие
   валидации на реальном корпусе Reports при инвентаризации.
