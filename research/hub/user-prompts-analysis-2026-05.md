# Анализ пользовательских промптов

Версия: 0.1 | Дата: 2026-05-26

## 🔹 Краткий итог (для быстрого чтения)

Анализ применяет матрицу из
`research/hub/prompts-classification-standard-2026-05.md` к 18 пользовательским
промптам из issue #85.

Главный вывод: все 18 промптов относятся к `Structured`-уровню (`PT-2`) и
ориентированы на человека в чате (`SC-H`). Это сильная сторона набора: почти
везде есть роль, правила, формат или порядок работы. Агентских (`PT-3`) и
multi-agent (`PT-4`) промптов нет.

Основной риск: набор содержит несколько эволюционных дублей. Особенно это
заметно в ASR-промптах (`#1`/`#2`) и в генерации ФТ (`#4`-`#11`). Их лучше не
переносить в репозиторий как 18 отдельных активных файлов. Практичнее выделить
4-5 устойчивых prompt families: ASR post-editing, генерация ФТ, контекст
разделов 1-2, детализация раздела 7, meeting-to-requirements и вопросы
заказчику.

Что уже покрыто репозиторием: генерация User Story, пошаговый Use Case и
статистика классов Mango. Прямого готового промпта для ASR, генерации ФТ,
резюме встреч, сопроводительного письма или списка вопросов заказчику пока нет.

Приоритет для интеграции: сначала адаптировать `#18` как prompt для уточняющих
вопросов заказчику и `#6`/`#10` как основу генератора ФТ. Они ближе всего к
Mango workflow и дополняют существующие `user-story-generator_*` и
`usecase-stepwise-generator_*`.

## 🔹 Таблица классификации 18 промптов

Обозначения соответствуют стандарту:

- `PT-2` - Structured prompt.
- `KA-S` - Standalone: знания встроены в промпт или ожидаются от модели.
- `KA-R` - RAG-enabled: промпт явно требует внешнюю базу знаний или выдержку.
- `DM-1` - Single-shot; `DM-M` - Multi-turn.

- `M2-lite` - та же зона M2, но без полноценного диалога или с single-shot

## 🔹 Признаки устаревших паттернов (если найдены)

| Признак | Где видно | Почему это риск | Как адаптировать |
|---------|-----------|-----------------|------------------|
| Версионные дубли вместо одного активного промпта | #1/#2, #4/#5/#6, #9/#10/#11 | Пользователь не понимает, какая версия актуальна; поведение расползается | Оставить одну активную версию, старые версии пометить как архивные в анализе |
| Слишком длинные инструкции без явной иерархии | #2, #5, #6 | Модель хуже удерживает приоритеты; пользователю трудно копировать | Перевести в структуру `# РОЛЬ` / `# ВХОДЫ` / `# КАК РАБОТАЕМ` / `# ПРАВИЛА` / `# ФОРМАТ` / `# НАЧНЕМ?` |
| Ссылки на TM Forum/eTOM/ODA без предоставленного источника | #4-#11, #13, #18 | Модель будет опираться на общие знания, а не на согласованную team KB | Для точных проверок сделать `_exp`/RAG-вариант и запрашивать выдержку из ODA/eTOM/Mango taxonomy |
| Смешение задач в одном промпте | #4, #5, #6, #14 | Валидация, генерация, редактирование и согласование конкурируют за внимание | Разделить на prompt family: validate FT, generate FT, summarize meeting, finalize requirements |
| Неуправляемое "согласование по умолчанию" | #5, #6 | Если пользователь молчит, результат не должен автоматически считаться согласованным | Заменить на явный stop-gate: "СТОП: жду подтверждение или правки" |
| Настройки модели внутри prompt body | #1, #2, #4 | "Температура" - параметр запуска, а не надежное поведенческое правило в тексте | Заменить на поведенческое правило: "работай консервативно, не меняй смысл без подтверждения" |
| Конфликт "без Markdown" и "выделять жирным" | #4-#6 | Жирное выделение обычно требует Markdown или форматирование редактора | Явно выбрать формат: plain text без выделений или Markdown с `**изменениями**` |
| Недостаточный output contract у быстрых версий | #7-#11, #17 | Результат может отличаться между сессиями | Добавить короткий формат вывода: статус, ФТ/резюме/письмо, вопросы, допущения |

## 🔹 План интеграции

Интеграцию лучше делать отдельными небольшими PR, чтобы не раздуть
`projects/mango/prompts/` и не сломать текущий контракт каталога.

| Приоритет | Что интегрировать | Источник | Целевой формат | Почему |
|-----------|-------------------|----------|----------------|--------|
| P1 | Промпт уточняющих вопросов заказчику | #18 | `requirement-questions_simple-2026-05.md` | Закрывает этап до User Story и снижает риск домыслов |
| P1 | Генератор ФТ | #6 + #10 | `ft-generator_simple-2026-05.md` и позже `_exp` | Самая частая рабочая задача, не покрыта текущими prompt assets |
| P2 | Контекст разделов 1-2 | #12 | `ft-context-sections_simple-2026-05.md` | Узкий, понятный артефакт для подготовки ФТ |
| P2 | Meeting summary для требований | #15 + #16 | `meeting-requirements-summary_simple-2026-05.md` | Связывает ASR/встречи с формализацией требований |
| P3 | Раздел 7 "Список доработок" | #13 | `implementation-changes-list_simple-2026-05.md` | Полезно для разработки, но выше риск уйти в неподтвержденные UI/API детали |
| P3 | ASR post-editor | #1 + #3 | Отдельный universal prompt или Mango prompt после решения PO | Ценно, но не является core Mango taxonomy/requirements prompt |

Минимальная схема адаптации для каждого нового prompt asset:

1. Добавить frontmatter по образцу существующих `projects/mango/prompts/*`.
2. Использовать структуру `# РОЛЬ`, `# КОНТЕКСТ` или `# ВХОДЫ`,
   `# КАК РАБОТАЕМ`, `# ПРАВИЛА`, `# ФОРМАТ ВЫВОДА`, `# НАЧНЕМ?`.
3. Для `_simple` держать standalone-логику без ссылок на репозиторий.
4. Для `_exp` явно запрашивать выдержки `research/mango/classification.md`,
   ODA/eTOM map или другой утвержденный KB-фрагмент.
5. Не переносить архивные версии #2, #4, #5, #9, #11 как активные файлы.
