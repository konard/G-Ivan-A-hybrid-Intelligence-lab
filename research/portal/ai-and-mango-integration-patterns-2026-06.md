## Часть 1. Слой AI-провайдера (Yandex GPT)

### 1.1. Паттерн вызова

| Паттерн | Где живёт ключ | Безопасность | Вердикт |
| --- | --- | --- | --- |
| **Прямой вызов с клиента** | В браузере | 🔴 Ключ утекает в bundle/DevTools | Запрещено (AI_GOVERNANCE: секреты не на клиенте). |
| **Serverless-proxy** | На сервере (env/secret-store) | 🟢 Ключ не покидает сервер; можно добавить rate-limit, аудит, маскирование | 🟢 **Выбрать**. |
| **Свой backend-сервис** | На сервере | 🟢 Гибко | 🟡 Преждевременно для Phase 1 (Anti-Inflation). |

**Вывод 1.1:** все вызовы Yandex GPT — через **serverless-функцию-proxy**
(Cloudflare Worker / Netlify Function). Клиент шлёт запрос на свой эндпоинт
портала; функция добавляет ключ из secret-store, вызывает
`llm.api.cloud.yandex.net`, возвращает ответ. Это закрывает правило «AI API keys
must never be exposed client-side».

### 1.2. Провайдеро-агностичный слой

Чтобы не привязываться жёстко к одному вендору, proxy реализует тонкий
интерфейс `generate(promptId, variables, options)` с адаптером под Yandex GPT.
Смена/добавление провайдера (например, локальная модель) не затрагивает клиент.
Трассировка: open-source-first + отсутствие vendor-lock как принцип.

```text
[Браузер/остров BA Prompts]
        │ POST /api/ai/generate {promptId, variables}   (без ключа)
        ▼
[Serverless-proxy]  ── secret: YANDEX_API_KEY, FOLDER_ID
        │ собирает промпт из каталога + переменные
        │ (опц.) rate-limit, аудит-лог без PII
        ▼
[Yandex GPT 5 REST]  llm.api.cloud.yandex.net
```

## Часть 2. Интеграция `mango_ba_prompts` (критично)

### 2.1. Что подключаем

`mango_ba_prompts` — проект автоматизации бизнес-анализа Mango (промпты для
Yandex GPT). По модели Хаба это **отдельный spoke** (production-код, своя
команда), который портал **витринирует и связывает**, а не поглощает.

| Сценарий Phase 1 (из issue) | Что нужно порталу |
| --- | --- |
| Пользователь **вызывает промпты для скрининга/копирования** | Каталог промптов (версионируемый контент) + UI-просмотр/копирование + (опц.) запуск через proxy. |
| Пользователь **загружает результаты для отладки** | Форма загрузки результата → сравнение/комментирование (debug-loop) **без** записи приватных данных в репозиторий. |

### 2.2. Паттерны подключения каталога промптов

| Паттерн | Суть | Плюсы | Минусы | Вердикт |
| --- | --- | --- | --- | --- |
| **Git submodule / синк** | Промпты тянутся из репозитория `mango_ba_prompts` | Один источник правды, версионирование | Связность сборки, права доступа | 🟡 Если репозиторий доступен порталу. |
| **Контент-коллекция (копия санитизированных промптов)** | Портал хранит **публичные/шаблонные** промпты как контент | Просто, контролируемо, безопасно | Нужен процесс синхронизации | 🟢 **Выбрать для Phase 1** (только санитизированные шаблоны). |
| **Прямой API к приватному хранилищу промптов** | Портал ходит в чужое хранилище | Свежесть | 🔴 Приватность, связность, права | 🔴 Не для Phase 1. |

**Вывод 2.2:** Phase 1 — портал хранит **санитизированные/шаблонные** промпты
как версионируемую контент-коллекцию (в `projects/ba-prompts/`), с явной
ссылкой на исходный spoke. **Реальные клиентские промпты, секреты и
несанитизированные production-промпты не коммитятся** (AI_GOVERNANCE, правило 6).

### 2.3. Debug-loop (загрузка результатов)

- Результат, загружаемый пользователем, обрабатывается **в сессии/кабинете**, не
  пишется в git и не попадает в публичную аналитику с PII.
- В Phase 1 (без кабинета) — обработка клиентская/эфемерная; персистентное
  хранение результатов появляется в Phase 2 вместе с кабинетом и маскированием.

## Часть 3. Аутентификация (по фазам)

| Фаза | Потребность | Паттерн | Источник |
| --- | --- | --- | --- |
| **Phase 1** | Публичный просмотр, копирование, эфемерный запуск | Без логина (или лёгкий rate-limit по IP/токену на proxy) | serverless-first, Anti-Inflation |
| **Phase 2** | Личный кабинет, история, сохранённые результаты | **Supabase Auth** (email/OAuth), RLS на уровне строк | Иссл. 2 (BaaS = Supabase) |
| **Phase 3** | Агенты, действия от имени пользователя | Сервисные роли + согласия + аудит | RFC §9 |

**Вывод 3:** auth вводится **в Phase 2** (не раньше — нет данных для защиты в
Phase 1). Open-source-first → Supabase Auth / Auth.js.

## Часть 4. Маскирование данных (по фазам)

| Приём | Что делает | Когда |
| --- | --- | --- |
| **Маскирование на клиенте до отправки** | Пользователь/правила удаляют PII (имена, телефоны, ID) до отправки в proxy/LLM | Phase 2 (ключевой механизм) |
| **Аудит-лог без PII** | Proxy логирует факт вызова и метрики, не содержимое с PII | Phase 2 |
| **Изоляция результатов (RLS)** | Данные пользователя видны только ему | Phase 2 |
| **Согласия и retention-политика** | Явное согласие, сроки хранения | Phase 2–3 |

**Вывод 4:** маскирование — **сердце Phase 2** (личный кабинет + маскирование из
варианта фаундера). Принцип: приватные данные минимизируются на клиенте **до**
выхода за периметр; портал и LLM видят как можно меньше.

## Карта рисков приватности

| Риск | Митигирование |
| --- | --- |
| Утечка API-ключа | Только serverless-proxy; secret-store; ключ не в bundle. |
| Коммит реальных клиентских промптов/результатов | Только санитизированные шаблоны в git; CI-проверка на секреты; правило 6 AI_GOVERNANCE. |
| PII в LLM-запросе | Клиентское маскирование (Phase 2) до отправки. |
| PII в аналитике | PostHog без PII; аудит-лог обезличен. |
| Связность с приватным репозиторием | Phase 1 — копия санитизированных шаблонов, не прямой доступ. |
