# Исследование: оптимальный формат исполнимых контрактов

## 1. Введение

**Причина.** Issue #165, Блок 2: «Модель и человек сразу должны понимать, как
запускается и что надо делать». Зафиксированный антипаттерн (см. инцидент
«анализ вместо исполнения» в
[governance/rfc/contract-executability-rfc.md](../../governance/rfc/contract-executability-rfc.md)):
агент читает исполнимый документ как описание и начинает *анализировать* вместо
*исполнять*. Нужен формат, в котором **первая строка — команда к действию, а не
заголовок**.

**Цель.** Исследовать лучшие практики оформления исполнимых документов

## 2. Результаты исследования

### 2.1. Главный вывод

Исполнимый контракт строится по схеме **директива → EXECUTION → EXPLANATION**,
где **первая видимая строка после frontmatter — императивная команда**, а не
заголовок и не контекст. Это «action-first» / «command-first» интерфейс: документ
ведёт себя как `SYNOPSIS` в man-странице или как system prompt — инструкция стоит
раньше пояснения.

Формат подтверждает и **усиливает** утверждённое фаундером «Видение 3»
(«Директива сверху, контекст под катом») трёхслойной защиты из RFC. Расхождений
с RFC нет; стандарт лишь придаёт ему нормативную (RFC 2119) форму.

### 2.2. Рекомендации

| # | Рекомендация | Куда фиксируется |
| --- | --- | --- |
| R1 | Маркер `executable: true` (опц. `entrypoint: true`) во frontmatter — машиночитаемый сигнал типа документа. | `executable-contract-standard.md` |
| R2 | Сразу после frontmatter — **директивный блок** (🚦 blockquote): стоп-сигнал «не анализируй — выполняй» + роль/цель (User-Story) + явный запрет анализа. | стандарт |
| R3 | Далее — **▶️ EXECUTION**: первая строка = императивная команда; шаги по порядку, без «контекста». | стандарт |
| R4 | Ниже — **ℹ️ EXPLANATION**: контекст, обоснование, краевые случаи (всё, что *не* нужно для немедленного исполнения). | стандарт |
| R5 | Человеческий слой навигации: README-легенда 🚦 объясняет смысл значка исполнимости. | стандарт, §навигация |
| R6 | Применить формат к `governance/agent-onboarding-protocol.md` как к эталону; шаблоны спока (`AI_QUICK_RULES.md`, `AI_SESSION_HANDOVER_PROMPT.md`) уже несут 🚦. | стандарт, §миграция |

### 2.3. Открытые вопросы

- **Степень дублирования директивы.** Директивный блок не должен копировать весь
  Handover Prompt — только указывать на исполнимую часть (канон остаётся
  единственным). Решение: директива — указатель, а не копия (см. §3.4 и GLOSSARY
  «Директивный блок»).
- **Смешанные документы.** Документ, частично исполнимый, частично справочный, —
  помечается `executable: false` и не выдаёт ложного сигнала; исполнимой делается
  только выделенная процедура. См. §3.5.

### 2.4. Учёт мнения команды Q

Команда Q по Блоку 2 отдельной рекомендации не давала. «Входные данные» здесь —
ранее утверждённое фаундером «Видение 3» из RFC. Мы относимся к нему как к
входу, а не догме, и проверяем независимо (§3): практики system-prompt и runbook
подтверждают «команда сверху». Совпадение усиливает решение; отклонять «Видение 3»
оснований нет.

## 3. Детализация

### 3.1. Почему «первая строка — команда»

Модель (LLM) и человек строят гипотезу о *типе* документа по его **началу**.
Если первым идёт заголовок «# Протокол онбординга» и абзац контекста, и человек,
и агент по умолчанию переходят в режим *чтения/анализа*. Если первой идёт
императивная директива («ВЫПОЛНИ следующие шаги сейчас; не анализируй»), активируется
режим *исполнения*. Это прямое следствие того, что инструкция в начале
контекста сильнее влияет на поведение, чем та же инструкция в конце (общее
наблюдение prompt-engineering: ключевые инструкции размещают в начале и явно).

### 3.2. Сравнение практик императивного интерфейса

| Практика | Где «команда» | Что под катом | Сигнал «исполняй, не читай» |
| --- | --- | --- | --- |
| **System prompt (LLM)** | Инструкции в начале, императив | примеры, ограничения ниже | роль + «You must…» |
| **SRE Runbook** | Пронумерованные действия сверху | rationale/диагностика рядом или ниже | «Steps to mitigate» |
| **man-страница** | `SYNOPSIS` (вызов) до `DESCRIPTION` | подробное описание ниже | структурный порядок секций |
| **README Quick Start** | команда установки/запуска в начале | архитектура/детали ниже | «Quick start» заголовок |
| **Makefile / CLI** | цель = действие (`make build`) | комментарии/доки рядом | сам факт исполняемости |
| **Хаб «Видение 3»** | 🚦 директивный блок + ▶️ EXECUTION | ℹ️ EXPLANATION ниже | стоп-сигнал «не анализируй — выполняй» |

Вывод: все императивные интерфейсы ставят **вызов/действие раньше описания**.
«Видение 3» — точная реализация этого принципа для Markdown-документа,
читаемого ИИ-агентом.

### 3.3. Три слоя защиты (наследуются из RFC)

1. **Frontmatter-маркер** — `executable: true` (`entrypoint: true` для точки
   входа). Машиночитаемо, ортогонально `Operating Mode` (см.
   [standards/glossary.md](../../standards/glossary.md)).
2. **Директивный блок** (🚦) — in-band «системная роль» для документа, которой у
   обычного Markdown нет: стоп-сигнал + User-Story (роль/цель агента) + запрет
   анализа. Стоит **до** любого пояснения.
3. **EXECUTION → EXPLANATION** — тело: сначала команда и шаги, затем контекст.

Слои независимы: даже если агент проигнорировал frontmatter, он встретит 🚦-блок;
даже если пропустил 🚦, структура EXECUTION-first удержит его в режиме исполнения.
Это «defense in depth».

### 3.4. Директива — указатель, не копия

Директивный блок **не** копирует канонический Handover Prompt целиком: он
коротко указывает «исполняй процедуру ниже». Канон (например, сам Handover Prompt)
остаётся в единственном экземпляре, иначе появляются рассинхронизированные копии
(анти-DRY). См. определение «Директивный блок» в
[standards/glossary.md](../../standards/glossary.md).

### 3.5. Краевые случаи

- **Справочный документ.** `executable: false` или отсутствие поля → обычный
  режим чтения; директивный блок не ставится (ложный сигнал хуже отсутствия
  сигнала).
- **Частично исполнимый.** Документ помечается `executable: false`, а исполнимой
  оформляется только выделенная процедура (или она выносится в отдельный
  `executable: true` файл).
- **Точка входа.** `entrypoint: true` — единственный «первый» исполнимый документ
  (в Хабе это `governance/agent-onboarding-protocol.md`); валидатор frontmatter допускает
  значение только `true`.

### 3.6. Зеркальность с форматом исследований

| Признак | Исполнимый контракт | Исследование |
| --- | --- | --- |
| Цель | действие сейчас | знание для решения |
| Что сверху | **команда** | **выводы** |
| Маркер | `executable: true` | `executable: false`/нет |
| Режим читателя | исполнять | читать/решать |

Оба формата подчинены одному метапринципу Хаба: **самое важное — наверх**. Для
исполнимого документа важнейшее — команда; для исследования — вывод.
