# Жизненный цикл требования Mango: неопределенность, декомпозиция и международная практика

Документ исследует движение требования на доработку системы или новую фичу
Mango от сырого сигнала заказчика до согласованного бизнес-слоя (этапы `0 → 1`
уточненного потока) и фокусируется на одном вопросе, который пока не закрыт
другими исследованиями `research/mango/`: **как процесс БА должен вести себя
при неопределенности входящего требования**. Цель — дать основу для будущего
стандарта процесса БА Mango и для оркестрации промпт-цепочек уровня
автоматизации 2 (системный промпт + артефакты).

## Область и границы исследования

| Параметр | Значение |
| --- | --- |
| Operating Mode | `creative` — моделируем кейсы, предлагаем варианты структуры стандарта, адаптируем под телеком-контекст Mango (см. [standards/glossary.md](../../standards/glossary.md)). |
| В фокусе | Запросы на доработку системы и новые фичи; декомпозиция составных требований; обработка неопределенности (этапы `0.1 → 0.5`, бизнес-слой `1`). |
| Вне фокуса | Валидация готовых тендерных ТЗ и расчет tender-fit — это отдельный процесс, уже описанный в [2026-05-22-requirements-flow.md](2026-05-22-requirements-flow.md). |
| Опора | Уточненный поток требования и 4 уровня автоматизации из issue #55; Unified Capability Taxonomy из [2026-05-26-taxonomy-concept.md](2026-05-26-taxonomy-concept.md). |
| Принцип | Anti-Inflation ([governance/repo-model.md](../../governance/repo-model.md)): не плодим новый стандарт и не дублируем существующие исследования; этот файл — research draft, а не canonical правило. |

### Уточненный поток требования (рабочая модель из issue #55)

Для удобства ссылок ниже фиксируем этапы, на которые опирается все
исследование. Это рабочая модель обсуждения, а не утвержденный стандарт.

| Этап | Подпроцесс | Артефакт этапа |
| --- | --- | --- |
| `0. Разбор сырого` | `0.1` Signal Extract → `0.2` Ambiguity Map → `0.3/0.4` Интервью / Offline Q&A (опционально) → `0.5` Normalization | `Normalized Requirement` |
| `1. Бизнес-слой` | User Story → Use Case → ФТ (высокий уровень, без техники) | `Business Alignment Pack` |
| `2. Gap-анализ` | Проверка OOTB / реализуемости относительно `kb/product-matrix.md` | `Gap Analysis Report` |
| `3. Технический слой` | ФТ (детально) + НФТ | `Technical Specification` |
| `4. Документирование` | Сборка спецификации | `Final Doc / Task` |
| `5. Трассировка` | Связь с задачей / тестом / релизом | `RTM entry` |

Этапы `0.3–0.4` опциональны: при низкой неопределенности требование идет по
пути `Offline Q&A`, БА выступает оркестратором промпт-цепочек. Настоящий
документ детально прорабатывает именно `0 → 1`, потому что здесь рождается и
гасится неопределенность; этапы `2 → 5` затрагиваются только в части связей.

## 1. Бенчмарк международной практики

Цель раздела — посмотреть, как четыре зрелые практики проводят требование от
спроса до решения и какие их паттерны устойчивы к неопределенности. Для каждой
практики фиксируем: что работает, что не переносится в Mango напрямую, и какой
именно anti-uncertainty паттерн стоит заимствовать.

### 1.1 TM Forum ODA / eTOM (отраслевой телеком-слой)

eTOM (Business Process Framework) и ODA задают сквозной язык
`Customer → Product → Service → Resource` и lifecycle-процессы
Strategy/Concept → Fulfillment → Assurance. Это тот же доменный слой, на
котором уже стоит [2026-05-26-taxonomy-concept.md](2026-05-26-taxonomy-concept.md).

- **Что работает.** Жесткое разделение слоев Customer/Product/Service/Resource
  не дает смешать «что хочет клиент» и «чем мы это реализуем». Это прямой
  анти-паттерн для преждевременных технических решений на этапе `1`.
- **Что не переносить.** Полный Frameworx как обязательная структура процессов
  и команд избыточен для уровня автоматизации 2; берем язык доменов и
  lifecycle, а не всю модель.
- **Anti-uncertainty паттерн.** Декомпозиция требования по слоям ODA до
  классификации: неопределенность локализуется на конкретном слое (неясна
  Service-граница vs неясен Resource), а не «требование непонятное целиком».

### 1.2 ITIL 4 Demand Management + Service Request Management

ITIL 4 разделяет управление спросом (понять и сформировать спрос) и обработку
сервисных запросов (предсказуемое исполнение типовых запросов). Concept of
value streams показывает требование как поток ценности от спроса к результату.

- **Что работает.** Явное различение «запрос-проект» (нужен анализ и discovery)
  и «типовой запрос» (есть готовый путь). Это основа для маршрутизации
  `One-to-One` vs составное требование в кейсах раздела 3.
- **Что не переносить.** Тяжелый каталог сервисных запросов и формальные
  request models оправданы при больших объемах; для пилота достаточно правила
  маршрутизации по уровню неопределенности.
- **Anti-uncertainty паттерн.** Triage спроса на входе: до глубокого анализа
  оценивается, типовой это запрос или требует discovery. В потоке Mango этому
  соответствует ранний `0.2 Ambiguity Map`.

### 1.3 Cisco (vendor-практика requirements/solution discovery)

Публичная инженерная практика Cisco (lifecycle PPDIOO — Prepare, Plan, Design,
Implement, Operate, Optimize — и solution validation в DevNet) строит решение
от бизнес-требований к техническим через явную фазу валидации.

- **Что работает.** Отдельная фаза Plan/Design с проверкой требований до
  реализации; требование не уходит в дизайн, пока не подтверждено бизнес-
  намерение. Это совпадает с правилом Mango «согласование с заказчиком ДО
  технической оценки» (этап `1` перед `3`).
- **Что не переносить.** PPDIOO заточен под сетевые инфраструктурные проекты;
  для SaaS-фич избыточны фазы Operate/Optimize в каждом мелком требовании.
- **Anti-uncertainty паттерн.** Gate между Plan и Design: переход дальше
  разрешен только при закрытых открытых вопросах. В Mango — критерий перехода
  `0.5 → 1` и `1 → 2`.

### 1.4 МТС (российский телеком-контекст)

Публичные материалы МТС о продуктовой и agile-трансформации описывают переход к
продуктовым командам, discovery и работе с гипотезами при неполных данных. Для
Mango это ближайший рыночный и регуляторный контекст (РФ, телеком, ПДн, услуга
связи).

- **Что работает.** Продуктовый discovery как способ работать с неполным
  требованием: гипотеза + быстрая проверка вместо блокировки до полной ясности.
  Прямо ложится на принцип issue #55 «не блокируй прогресс при
  неопределенности».
- **Что не переносить.** Масштаб продуктовых команд и dual-track agile МТС
  избыточен для пилота из 2 продуктов; берем идею гипотез и discovery-петли.
- **Anti-uncertainty паттерн.** Гипотеза с явным уровнем уверенности и быстрым
  способом проверки вместо ожидания полной спецификации. В Mango — атрибут
  `uncertainty: low/medium/high` и опциональные `0.3/0.4`.

### 1.5 Сводка паттернов, устойчивых к неопределенности

| Источник | Что работает | Что не переносить | Anti-uncertainty паттерн для Mango |
| --- | --- | --- | --- |
| TM Forum ODA/eTOM | Слои Customer/Product/Service/Resource, lifecycle. | Полный Frameworx как обязательная структура. | Декомпозиция по слоям локализует неопределенность. |
| ITIL 4 Demand Mgmt | Различение спрос-проект vs типовой запрос. | Тяжелый каталог request models. | Ранний triage спроса (`0.2 Ambiguity Map`). |
| Cisco PPDIOO / validation | Валидация требования до дизайна. | Полный lifecycle на каждое мелкое требование. | Gate Plan→Design = критерии перехода между этапами. |
| МТС product discovery | Гипотезы при неполных данных. | Масштаб dual-track agile. | Гипотеза + `uncertainty` вместо блокировки. |

### 1.6 Развилка бенчмарка: как формализовать неопределенность

По требованию issue #55 «при неопределенности в бенчмарке — предложи 2-3
варианта с плюсами/минусами». Главная открытая развилка: **как именно мерить и
кодировать неопределенность требования**.

| Вариант | Суть | Плюсы | Минусы |
| --- | --- | --- | --- |
| A. Дискретная шкала `low/medium/high` | Один атрибут уверенности на требование (как в taxonomy-concept). | Просто для промпта и для БА; согласуется с существующей таксономией. | Грубо: не видно, в чем именно неопределенность (термин, объем, источник). |
| B. Ambiguity Map по категориям | Матрица типов неясности: термин / объем / источник данных / интеграция / приемка. | Точно показывает, какой вопрос задать на `0.3/0.4`; управляемый clarification backlog. | Дороже в заполнении; требует словаря категорий неясности. |
| C. Гибрид: шкала + теги категорий | Общий уровень `low/medium/high` + 1-3 тега из Ambiguity Map. | Баланс простоты и точности; шкала для триажа, теги для вопросов. | Нужно договориться о минимальном наборе тегов, чтобы не перегрузить. |

Рекомендация для пилота — **вариант C**: он переиспользует атрибут
`uncertainty` из [2026-05-26-taxonomy-concept.md](2026-05-26-taxonomy-concept.md) и
добавляет ровно столько структуры, сколько нужно для генерации вопросов на
`0.3/0.4`. Решение оставляем на согласование с PO (см. раздел «Вопросы»).

## 2. Сравнение с планируемым процессом Mango

Таблица сопоставляет международную практику с уточненным потоком Mango: где
совпадает, где нужна адаптация, где шаги Mango сильнее или слабее.

| Аспект | Международная практика | Наш флоу (Mango) | Совпадение / адаптация / сила-слабость |
| --- | --- | --- | --- |
| Точка входа требования | ITIL Demand Mgmt: triage спроса на входе. | `0.1 Signal Extract` + `0.2 Ambiguity Map`. | **Совпадает.** Наш шаг сильнее: явно отделяет извлечение сигнала от карты неясностей. |
| Обработка неполноты | МТС/discovery: гипотеза при неполных данных. | `0.3/0.4` опциональные интервью/Offline Q&A. | **Совпадает.** Адаптация: у нас два режима (sync интервью и async Q&A) с явным выбором по уровню неопределенности. |
| Нормализация языка | ODA/SID + ISO/IEC/IEEE 29148: единый язык и атомарность. | `0.5 Normalization` к Unified Capability Taxonomy. | **Совпадает.** Сила: привязка к 7 доменам и capability ID (`MGO-XX-...`). |
| Разделение бизнес/техника | Cisco: бизнес-намерение до дизайна; ODA слои. | Этап `1` (бизнес-слой) строго до этапа `3` (техника). | **Совпадает.** Сила: согласование с заказчиком зафиксировано как обязательный gate. |
| Декомпозиция составного требования | ODA: разнесение по слоям Service/Resource. | Декомпозиция Cross-Product на capability разных доменов. | **Адаптация.** Слабость: пока нет явного правила связывания подтребований в один пакет (см. кейс В). |
| Проверка реализуемости | Cisco validation; ITIL request models. | Этап `2` Gap-анализ против `kb/product-matrix.md`. | **Совпадает.** Слабость: матрица фич как RAG-навигатор пока не создана. |
| Критерии перехода между этапами | Cisco gate Plan→Design. | Пока неявные; этот документ их предлагает (раздел 6). | **Слабее.** Адаптация требуется: формализовать gate-критерии. |
| Уровень автоматизации | Vendor-инструменты, тяжелые платформы. | Уровень 2: системный промпт + артефакты, путь к n8n. | **Сильнее по экономичности.** Слабее по готовым инструментам; компенсируется промпт-цепочками. |
| Роль человека | Product owner / solution architect утверждает. | БА — оркестратор промптов; человек подтверждает статус. | **Совпадает** с принципом human ownership taxonomy-concept. |
| Метрики | ITIL: время обработки, доля типовых; discovery: validated learning. | Пока не зафиксированы; предложены в разделе 6. | **Слабее.** Требуется задать метрики обработки неопределенности. |

Вывод: поток Mango концептуально согласован с международной практикой и сильнее
в части ранней работы с неопределенностью (`0.1/0.2`) и разделения слоев.
Главные пробелы — формальные критерии перехода между этапами, правило сборки
составного требования и метрики. Их закрывает раздел 6.

## 3. Моделирование кейсов входящих требований

Для каждого кейса показываем прохождение `0.1 → 1`, уровень и тип
неопределенности, нормализацию к таксономии и примеры `User Story` и
`Use Case`. Используются домены и поля из
[2026-05-26-taxonomy-concept.md](2026-05-26-taxonomy-concept.md) (домены `VN/UC/CC/AI/AQ/IP/GC`,
`capability_id`, `uncertainty`, `coverage_status`).

### Кейс А — четкое полное требование (One-to-One)

Сырой сигнал заказчика: «Нужно, чтобы входящие звонки распределялись на
свободного оператора с нужным навыком, а не по кругу».

| Этап | Действие | Результат |
| --- | --- | --- |
| `0.1` Signal Extract | Вычленен сигнал: маршрутизация по навыкам. | 1 атомарный сигнал. |
| `0.2` Ambiguity Map | Неясностей нет: capability стандартная, домен `CC`. | `uncertainty: low`, тегов нет. |
| `0.3/0.4` | Пропускаются (низкая неопределенность). | Путь Offline Q&A не нужен. |
| `0.5` Normalization | Привязка к capability `MGO-CC-ROUTING-001` (skill-based routing), домен `CC`, `feature_type: core-feature`. | `Normalized Requirement`. |
| `1` Бизнес-слой | Генерация User Story → Use Case → ФТ (высокий уровень). | `Business Alignment Pack`. |

**Normalized Requirement.** «Система должна распределять входящий вызов на
свободного оператора, обладающего требуемым навыком, вместо кругового
распределения, с фиксацией результата маршрутизации».

**User Story.**
> Как супервайзер контакт-центра, я хочу, чтобы входящие вызовы направлялись на
> свободного оператора с подходящим навыком, чтобы клиент быстрее попадал к
> компетентному специалисту и сокращалось время решения.
>
> Capability: `MGO-CC-ROUTING-001` · Домен: `CC` · `uncertainty: low`

**Use Case.**

| Поле | Значение |
| --- | --- |
| Название | UC-A: Skill-based маршрутизация входящего вызова |
| Актор | Входящий вызов / супервайзер (настройка) |
| Предусловие | Заданы навыки операторов и правила соответствия. |
| Основной сценарий | 1. Поступает вызов. 2. Система определяет требуемый навык. 3. Находит свободного оператора с навыком. 4. Соединяет. 5. Логирует решение маршрутизации. |
| Альтернативы | Нет свободного оператора с навыком → постановка в очередь навыка / callback. |
| Постусловие | Вызов соединен или поставлен в очередь; решение зафиксировано. |
| Coverage (предв.) | `Покрывается` — стандартная capability CCaaS (подтверждается на этапе `2`). |

### Кейс Б — неоднозначное неполное требование (нужны `0.3/0.4`)

Сырой сигнал: «Хотим автоматически оценивать качество работы операторов».

| Этап | Действие | Результат |
| --- | --- | --- |
| `0.1` Signal Extract | Сигнал: автоматическая оценка качества. | 1 сигнал, но размытый. |
| `0.2` Ambiguity Map | Неясности: что оценивать (речь/чек-лист/тональность), источник (запись/реалтайм), охват (все/выборка), критерий приемки. | `uncertainty: high`, теги: `объем`, `источник данных`, `приемка`. |
| `0.3/0.4` | Запускается Offline Q&A (4 вопроса по тегам Ambiguity Map). | Ответы заказчика: оценка по настраиваемому чек-листу, источник — записи звонков, охват — все звонки, критерий — отчет с баллом по чек-листу. |
| `0.5` Normalization | После ответов: capability `MGO-AQ-QM-001` (Quality Management), домен `AQ`, теги `compliance` (записи), `NFR: accuracy`. | `Normalized Requirement` + снижение `uncertainty` до `low`. |
| `1` Бизнес-слой | Генерация User Story → Use Case. | `Business Alignment Pack`. |

**Вопросы `0.3/0.4` (сгенерированы из Ambiguity Map).**
1. По каким критериям оценивать: готовая речевая аналитика, настраиваемый
   чек-лист или тональность? *(тег: объем)*
2. Источник оценки — записи звонков или оценка в реальном времени?
   *(тег: источник данных)*
3. Оцениваются все звонки или выборка? *(тег: объем)*
4. Что считается результатом: балл по чек-листу, отчет, флаг нарушения?
   *(тег: приемка)*

**Normalized Requirement (после Q&A).** «Система должна выполнять
автоматическую оценку всех записанных звонков операторов по настраиваемому
чек-листу с сохранением результата оценки (балл и отметки по пунктам) и
доступом к отчету».

**User Story.**
> Как руководитель контроля качества, я хочу автоматически оценивать записанные
> звонки операторов по настраиваемому чек-листу, чтобы получать объективную
> оценку качества без ручного прослушивания.
>
> Capability: `MGO-AQ-QM-001` · Домен: `AQ` · `uncertainty: low` (после `0.4`) ·
> Compliance-trigger: `запись разговоров / ПДн`

**Use Case.**

| Поле | Значение |
| --- | --- |
| Название | UC-Б: Автоматическая оценка звонков по чек-листу |
| Актор | Руководитель QM / система оценки |
| Предусловие | Настроен чек-лист; доступны записи звонков; есть правовое основание обработки. |
| Основной сценарий | 1. Завершается звонок и сохраняется запись. 2. Система применяет чек-лист. 3. Формирует балл и отметки. 4. Сохраняет результат и делает доступным в отчете. |
| Альтернативы | Запись недоступна → пометка «не оценено»; спорный пункт → ручная проверка QM. |
| Постусловие | Оценка сохранена и доступна; compliance-флаг проставлен. |
| Coverage (предв.) | `Требует уточнения` до `0.4`, далее `Покрывается / Частично` — проверяется на этапе `2`. |

Кейс Б показывает ключевую ценность процесса: без `0.2/0.4` требование ушло бы
в работу с `uncertainty: high` и дало бы нестабильный вывод; карта неясностей
превращает один размытый сигнал в проверяемое требование.

### Кейс В — составное требование (Cross-Product: ЛК Mango + CRM)

Сырой сигнал: «Нужна статистика по звонкам в личном кабинете Mango и
автоматическая передача записей разговоров в нашу внешнюю CRM».

| Этап | Действие | Результат |
| --- | --- | --- |
| `0.1` Signal Extract | Выделены 2 сигнала: (1) статистика звонков в ЛК; (2) авто-передача записей в CRM. | Признак составного требования. |
| `0.2` Ambiguity Map | Подтребование 1 — `uncertainty: low`. Подтребование 2 — `uncertainty: medium`, теги: `интеграция` (какая CRM, формат), `источник данных`, `приемка` (момент передачи). | Карта по каждому подтребованию. |
| `0.3/0.4` | Offline Q&A только по подтребованию 2: целевая CRM (например, amoCRM/Bitrix24), формат (вебхук/API), момент (по завершении звонка). | Ответы: amoCRM, передача по API по завершении звонка. |
| `0.5` Normalization | Декомпозиция на capability: подтребование 1 → `MGO-AQ-STAT-001` (домен `AQ`); подтребование 2 → `MGO-IP-CRM-001` (домен `IP`, integration). Оба связываются в один `Business Alignment Pack` через общий `request_id`. | 2 нормализованных требования + связь. |
| `1` Бизнес-слой | Генерация одной User Story с двумя Use Case (по одному на capability). | Составной `Business Alignment Pack`. |

**Правило декомпозиции (предложение).** Составное требование разносится на
атомарные подтребования по доменам таксономии, каждое получает свой
`capability_id` и `uncertainty`, но все связываются общим `request_id` и
маршрутизируются как пакет, чтобы не потерять зависимость «статистика + передача
во внешнюю систему».

**User Story.**
> Как руководитель отдела продаж, я хочу видеть статистику по звонкам в личном
> кабинете Mango и автоматически получать записи разговоров в amoCRM, чтобы
> анализировать нагрузку и хранить записи рядом с карточкой клиента.
>
> Пакет: `request_id: REQ-CMP-001` · Подтребования: `MGO-AQ-STAT-001` (`AQ`),
> `MGO-IP-CRM-001` (`IP`) · `uncertainty: low / medium`

**Use Case 1.**

| Поле | Значение |
| --- | --- |
| Название | UC-В1: Статистика по звонкам в ЛК Mango |
| Актор | Руководитель отдела продаж |
| Capability | `MGO-AQ-STAT-001` · Домен `AQ` |
| Основной сценарий | 1. Пользователь открывает раздел статистики в ЛК. 2. Задает период/фильтры. 3. Система показывает агрегаты по звонкам. |
| Coverage (предв.) | `Покрывается` — стандартная аналитика ЛК. |

**Use Case 2.**

| Поле | Значение |
| --- | --- |
| Название | UC-В2: Авто-передача записей звонков в amoCRM |
| Актор | Система интеграции |
| Capability | `MGO-IP-CRM-001` · Домен `IP` · integration · Compliance-trigger `ПДн` |
| Предусловие | Настроено подключение к amoCRM (API-ключ, маппинг полей). |
| Основной сценарий | 1. Звонок завершен, запись готова. 2. Система формирует событие. 3. Передает запись и метаданные в amoCRM по API. 4. Фиксирует статус доставки. |
| Альтернативы | CRM недоступна → очередь повторной отправки; ошибка маппинга → пометка на ручную проверку. |
| Coverage (предв.) | `Частично` / `Требует уточнения` — зависит от наличия готового коннектора amoCRM (проверяется на этапе `2`). |

Кейс В показывает, что декомпозиция и связывание — обязательная часть обработки
составного требования: разные домены, разный уровень неопределенности, но один
бизнес-пакет и одна точка согласования с заказчиком.

## 6. Рекомендации для стандарта процесса БА Mango

Рекомендации сформулированы как кандидаты для будущего стандарта. По
Anti-Inflation principle сам стандарт (`standards/...`) создается только после
пилота и подтвержденной операционной боли; пока это research-предложение.

### 6.1 Правила процесса

1. **Раннее картирование неопределенности.** Каждое требование проходит
   `0.2 Ambiguity Map` до классификации; результат — `uncertainty` + теги
   категорий (вариант C, раздел 1.6).
2. **Опциональность `0.3/0.4` по порогу.** Если `uncertainty: low` и тегов нет
   — интервью/Q&A пропускаются. Если `medium/high` — обязателен хотя бы
   асинхронный Offline Q&A по тегам.
3. **Бизнес-слой до техники.** Этап `1` (User Story → Use Case → ФТ высокого
   уровня) согласуется с заказчиком **до** этапа `3`; технические решения на
   этапе `1` запрещены (паттерн ODA/Cisco).
4. **Декомпозиция составного требования.** Cross-Product требование
   разносится по доменам таксономии, связывается общим `request_id` и
   маршрутизируется как пакет (кейс В).
5. **Evidence и человек-владелец.** AI предлагает классификацию и статус;
   финальное решение и изменение таксономии подтверждает БА/owner (наследуется
   из taxonomy-concept).
6. **Иерархия терминов.** Разрешение терминов: `kb/glossary.md` → внутренняя
   документация → документация внешней системы; отсутствующий термин помечается
   `⚠️` и уходит в clarification.

### 6.2 Критерии перехода между этапами (gate-критерии)

| Переход | Критерий перехода (gate) |
| --- | --- |
| `0.2 → 0.3/0.4` | `uncertainty ≥ medium` или есть ≥1 тег Ambiguity Map. |
| `0.3/0.4 → 0.5` | Все вопросы clarification закрыты или явно помечены «принято с допущением». |
| `0.5 → 1` | У требования есть `capability_id`, `domain`, `feature_type`, `requirement_type`; `uncertainty` снижен до `low/medium`. |
| `1 → 2` | Есть согласованный с заказчиком `Business Alignment Pack` (User Story + Use Case). |
| `2 → 3` | Известен `coverage_status`; для `Частично/Не покрывается` есть решение PO о доработке. |

### 6.3 Метрики контроля качества

| Метрика | Что измеряет | Целевой ориентир пилота |
| --- | --- | --- |
| `uncertainty resolution rate` | Доля требований, у которых `uncertainty` снижен до `low` после `0.4`. | ≥ 80 % требований доходят до `0.5` как `low`. |
| `clarification round count` | Сколько раундов Q&A нужно на требование. | Медиана ≤ 1 раунд. |
| `ambiguity detection recall` | Доля реальных неясностей, пойманных на `0.2` (а не всплывших позже). | Минимизировать поздние возвраты с этапа `2→0`. |
| `output stability` | Стабильность нормализованного требования при повторном прогоне промпта. | Идентичная классификация при повторе (см. термины issue). |
| `composite linking integrity` | Доля составных требований без потерянных подтребований. | 100 % подтребований связаны `request_id`. |
| `manual override rate` | Доля исправлений классификации/статуса человеком. | < 25 % (выше — процесс неустойчив). |

### 6.4 Точки обязательного ручного вмешательства

- `0.4` решение «принять с допущением» вместо ответа заказчика — фиксирует БА.
- Любое требование с compliance-trigger (`ПДн / реклама / услуга связи / КИИ`)
  — review БА и при необходимости Legal/ИБ.
- Составное требование с конфликтом доменов/владельцев — решает БА + затронутые
  PO.
- `output stability` нарушена (повторный прогон дал другую классификацию) —
  ручная фиксация эталона и запись в журнал отладки промптов.

### 6.5 Привязка к оркестрации промптов и n8n

| Этап процесса | Промпт-цепочка (уровень 2) | Узел будущего n8n |
| --- | --- | --- |
| `0.1` | Промпт Signal Extract → выходной JSON-контракт. | Node: Extract → ветвление по `is_composite`. |
| `0.2` | Промпт Ambiguity Map (теги + `uncertainty`). | Node: Classify uncertainty → switch по порогу. |
| `0.3/0.4` | Промпт Clarification Question Set (из тегов). | Node: Q&A (sync/async ветка) с human-in-the-loop. |
| `0.5` | Промпт Normalization (словари glossary + taxonomy как RAG). | Node: Normalize → запись в registry. |
| `1` | Три промпта: User Story, Use Case, ФТ высокого уровня с чек-листами. | Node: Generate Business Pack → approval gate. |

БА остается оркестратором: на уровне 2 он запускает цепочку вручную и
подтверждает gate-переходы; n8n постепенно автоматизирует ветвления и
human-in-the-loop точки, не убирая обязательные точки ручного контроля (6.4).

## Вопросы для согласования с PO/Founder

1. **Жесткая vs гибкая автоматизация.** Какие этапы требуют жесткой
   автоматизации сразу (`0.1 Signal Extract`, `0.5 Normalization`), а какие
   оставить гибкими (`0.3/0.4` интервью)? Предложение: жестко автоматизировать
   `0.1/0.2/0.5`, оставить гибкими `0.3/0.4` и `1`.
2. **Метрики обработки неопределенности.** Какую метрику считать главной для
   эффективности обработки неопределенности: `uncertainty resolution rate`,
   `clarification round count` или `output stability`? Нужно выбрать 1–2
   ведущих метрики для пилота (раздел 6.3).
3. **Готовность артефактов к `kb/` в спринте 1.** Какие артефакты из раздела 4
   внедряем в `kb/` первыми? Предложение: `kb/glossary.md` (#1), Ambiguity Map
   (#5), Clarification Question Set (#6), Normalized Requirement Contract (#7).
4. **Формализация неопределенности.** Принимаем ли вариант C (шкала + теги
   категорий) из раздела 1.6, и какой минимальный набор тегов Ambiguity Map
   фиксируем, чтобы не перегрузить процесс?
5. **Правило составного требования.** Утверждаем ли `request_id`-связывание
   подтребований как обязательное правило декомпозиции (кейс В)?
