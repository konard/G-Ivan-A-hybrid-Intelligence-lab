# RFC: Оптимизация потребления токенов при работе AI с репозиторием `mango_ba_prompts` / Token Consumption Optimization for AI Working with the `mango_ba_prompts` Repository

## 1. Введение / Introduction

**Причина / Pain.** В споке `mango_ba_prompts` присутствует системная **токеновая
инфляция (token inflation)**: документация создаётся как `Full` (полная, для
чтения человеком), но AI-агенты и пользовательские LLM используют её как
`Executable` (краткая, для загрузки в контекст). Один и тот же файл несёт и
исполнимые инструкции, и пояснения/историю/метаданные. При загрузке в LLM
токены тратятся на «шум», не нужный для выполнения задачи. Эталонный пример —
[`AI_SESSION_HANDOVER_PROMPT.md`](https://github.com/G-Ivan-A/mango_ba_prompts/blob/main/AI_SESSION_HANDOVER_PROMPT.md):
файл весит **14.6 KB (~9000 токенов)**, но для онбординга агента нужен только
блок `EXECUTION` (~3000 токенов). Перерасход — **~6000 токенов на каждый
онбординг**.

оптимизации потребления токенов: провести аудит сценариев и точек кипения

## 2. Результаты исследования / Findings

### 2.1. Главный вывод / Key Finding

Корень токеновой инфляции в `mango_ba_prompts` — **не объём знаний, а
отсутствие разделения «представления» (representation) и «носителя» (carrier)**.
Один markdown-файл одновременно играет три роли: (1) исполнимый контракт для
LLM, (2) объяснение для человека, (3) запись истории/синхронизации. AI вынужден
загружать все три, хотя для задачи нужна одна. Усугубляет проблему **дублирование
одного знания в 3–4 представлениях** (`ADR → Standard → Transfer → Audit`),
создающее retrieval noise.

Ключевая нюансировка: **часть «дублирования» намеренная**. Пара `ADR ↔ Standard`
— это сознательная практика Хаба: обязательства (нормы) живут в `standards/`,
обоснование (rationale) — в `docs/adr/` (подтверждено frontmatter
`related_standard`/`related_artifacts` и текстом «обязательства сверху,
обоснование — в ADR»). Значит цель не «склеить всё в один файл», а **дать LLM
загружать ровно тот слой, который нужен задаче**, не теряя историю решений.

## 3. Детализация / Details

### 3.0. Метод оценки токенов / Token Estimation Method

Точные токены зависят от токенайзера конкретной модели; для RFC используется
прозрачный сравнительный эвристик, калиброванный по единственному известному
замеру (фаундер: `AI_SESSION_HANDOVER_PROMPT.md` 14.3 KB ≈ 9000 токенов):

```text
tokens_estimate ≈ bytes × 0.6
```

Для русскоязычного markdown в UTF-8 это даёт верную **порядковую** оценку
(кириллица в UTF-8 занимает 2 байта/символ и тяжелее токенизируется, поэтому
ratio к байтам ниже, чем привычные ~0.25 tok/byte для английского). Все цифры
RFC — **сравнительные оценки** для приоритизации, а не биллинг. Точный baseline
собирается отдельно реальным токенайзером целевой модели (R6, OQ1).

### 3.2. ФТ-2. Разделение Full vs Executable / Splitting Representations

Сравнение подходов (плюсы/минусы, сложность, эффект):

| Подход | Суть | Плюсы | Минусы | Сложность | Эффект на токены |
| --- | --- | --- | --- | --- | --- |
| **A. Разделение файлов** (`*.exec.md` / `*.full.md`) | Исполнимое и полное — разные файлы | Чистая загрузка только `exec`; явный контракт | Риск рассинхрона двух файлов; больше файлов | Средняя | Высокий (−50–70%) |
| **B. Слои внутри файла** (`EXECUTION` сверху, `EXPLANATION` ниже) | Один файл, исполнимый блок первым | Нет рассинхрона; уже частично есть в handover | Агент всё равно может прочитать весь файл | Низкая | Средний (−30–50%) |
| **C. Ссылки/трансклюзия** (canonical + `@include`) | Тело — в одном месте, остальные ссылаются | Идеальный SSOT | Нет нативной трансклюзии в md; стоимость retrieval (OQ3) | Высокая | Высокий, но зависит от retrieval |
| **D. Динамическая загрузка** (selective/RAG) | Грузить фрагмент по запросу | Минимум контекста | Нужна инфраструктура (embeddings/index) | Высокая | Очень высокий |
| **E. Frontmatter-разметка слоёв** (`layer`, `tokens_estimate`) | Метаданные + инструмент-загрузчик | Совместимо с A/B/C/D; машиночитаемо | Требует инструмента-потребителя | Низкая (разметка) / Средняя (инструмент) | Усиливает любой из A–D |

**Рекомендация архитектора:** **B + E сейчас, A для hot-файлов, C/D — фазой 2.**
Начать с дешёвого и совместимого: исполнимый блок сверху (B, уже есть в
handover — нужно лишь, чтобы пояснения не «обволакивали» его) + frontmatter
`layer`/`tokens_estimate` (E). Для самых горячих файлов (H1) — вынести
`exec`-слой в отдельный артефакт (A). C/D (трансклюзия, RAG) — когда появится
загрузчик и Private/Object Storage из PR #254. Это **не ломает** существующие
промпты: `Full`-представление остаётся, меняется лишь порядок/адресация.

### 3.2.1. Паттерн: Каскадная загрузка контекста / Cascading Context Loading

Подходы A–E (§ 3.2) отвечают на вопрос «**как** разделить представления». Этот
паттерн отвечает на вопрос «**когда** грузить более тяжёлый слой» — и тем самым
превращает разделение из *возможности* в *исполнимый контракт*. Он усиливает
рекомендацию B+E+A и предложен по итогам ревью PR #256 (вариант с явными
триггерами эскалации).

**1. Принцип / Principle — иерархия доступа (access hierarchy).** Вместо
параллельного существования `executable` и `full`, где LLM сам решает, что
читать, вводится **строгий порядок загрузки**:

```text
Уровень 1 — Executable: загружается ВСЕГДА первым (минимальный исполнимый слой)
Уровень 2 — Full:       загружается ТОЛЬКО при срабатывании триггера эскалации
Уровень 3 — Raw:        загружается ТОЛЬКО при явной необходимости в исходнике
```

**2. Триггеры эскалации / Escalation triggers (детерминированные).** Ключевая
проблема паттерна — слово «эскалация» неоднозначно. Без явных триггеров LLM либо
*никогда* не грузит `full` (экономит токены, теряет качество), либо *всегда*
грузит (теряет экономию). Поэтому эскалация определяется как **закрытый список
детерминированных условий**, а не как усмотрение модели:

| Триггер эскалации / Escalation trigger | Пример / Example |
| --- | --- |
| Задача не выполнима по `executable` | «Не нашёл нужный раздел в краткой версии» |
| В `executable` отсутствует требуемый раздел | «Раздел "Бизнес-правила" не описан в executable» |
| Требуется детализация по **явному запросу** пользователя | «Покажи полное описание онтологии» |
| `executable`-слой возвращает ошибку/неопределённость | «Не могу определить уровень требования» |

Любое условие вне списка — эскалация **запрещена** (агент работает на текущем
слое). Это делает поведение **предсказуемым** и проверяемым на ревью.

**3. Контракт исполнения / Execution contract.** Промпт (или frontmatter `layer`
из подхода E) формулирует правило загрузки явно и императивно — пример шаблона:

```text
Начни с <executable>. Эскалация в <full> ТОЛЬКО при выполнении одного из
триггеров (см. список). Эскалация в <raw> ТОЛЬКО если нужны исходные данные.
Иначе — не загружай более тяжёлый слой.
```

**4. Примеры применения / Examples** (оценка токенов — по эвристику § 3.0):

| Артефакт | Уровень 1 (exec) | Уровень 2 (full) | Уровень 3 (raw) | Экономия при типовом запросе |
| --- | --- | --- | --- | --- |
| `standards/ba-ontology` | `ba-ontology.executable.md` ~5 KB | `ba-ontology.md` ~63 KB | — | до ~35 000 токенов (нет эскалации) |
| `kb/` фрагмент | `fragments/mango-api.executable.md` ~2 KB | `fragments/mango-api.md` ~20 KB | `sources/mango-api.pdf` | до ~11 000 токенов |
| `AI_SESSION_HANDOVER_PROMPT.md` | блок `EXECUTION` ~3 KB | блок `EXPLANATION` ~6 KB | блок `METADATA` ~5 KB (только человек) | **~11.3 KB ≈ ~7 000 токенов / онбординг** |

В примере handover `METADATA` не грузится агентом **никогда** (это слой для
человека) — крайний случай каскада: «уровень с запретом загрузки».

**5. Экономия / Savings.** Эффект гарантирован именно благодаря детерминизму:
при типовом запросе (триггер не сработал) агент остаётся на `executable`, и
перерасход H1/H2 (§ 2.2) не возникает вовсе. При срабатывании триггера качество
не страдает — `full`/`raw` догружаются ровно тогда, когда нужны. Это закрывает
риск OQ3 (мнимая экономия): дочитывание происходит не «на всякий случай», а по
явному условию.

**6. Совместимость / Compatibility (PR #254).** Паттерн ложится на Вариант B:
`executable`-слой живёт в Public `mango_ba_framework` (быстрый онбординг),
`full`/`raw` — где уместно по уровню ИБ (Public/Private/Object Storage), при этом
контракт загрузки един и не зависит от физического размещения. Каскад **не
ломает** существующие промпты (НФТ-4): `full` остаётся доступным, меняется лишь
условие его загрузки; и **не снижает** качество (НФТ-3), т.к. эскалация
детерминирована, а не отключена.

### 3.3. ФТ-3. Устранение дублирования / Deduplication (Token Inflation)

Стратегии (без потери истории решений):

| Стратегия | Что делает | Сохраняет историю? | Риск |
| --- | --- | --- | --- |
| **S1. Canonical carrier + ссылки** | Один носитель тела концепта; копии → ссылки | Да (ADR остаётся как rationale) | Стоимость retrieval (OQ3) |
| **S2. Слоевое разделение ролей** | ADR = почему, Standard = что, Transfer = мост к Хабу — без повторения тела | Да | Требует дисциплины авторов |
| **S3. Summary-указатель** | Короткий индекс-саммари + ссылка на полное | Да | Саммари может устареть |
| **S4. Карта артефактов как реестр SSOT** | `governance/artifact-map.md` фиксирует «где источник истины» | Да | Нужна актуализация |

**Single Source of Truth для LLM:** объявить в `governance/artifact-map.md`
спока для каждого концепта **один** `canonical`-носитель и пометить остальные
`status: derived` + ссылка. Агент по контракту грузит canonical; derived —
только при явной нужде в обосновании/истории. Намеренный split `ADR ↔ Standard`
сохраняется как S2 (роли разные, тело не дублируется). Начать с кластера
`ba-ontology` (H2) как пилота: проверить, сколько из 63.4 KB — реально разное
знание, а сколько — повтор (OQ4).

### 3.4. ФТ-4. Контентные и процессные стратегии / Content & Process

**Контентные / Content:**

- Исполнимый блок — **первым**, императивно, без преамбул («Сделай по шагам:
  …»). Пояснения, история, метаданные — ниже или в `full`-слое.
- **Вынос деталей**: длинные обоснования и исторические справки — в ADR/`full`,
  из исполнимого слоя убрать.
- **Гайдлайн для авторов**: «пиши executable-слой так, чтобы LLM могла исполнить
  без чтения остального файла».

**Процессные / Process:**

- **Валидация размера/бюджета** в CI (R4): для `executable: true` — warn при
  превышении порога (например, 4 000 токенов). По образцу существующего
  `tools/validate-frontmatter.sh` (warn-only, не блокирует).
- **Правило написания документов с учётом токенов** — в `CONTRIBUTING.md` спока.
- **Чек-лист ревью**: «есть ли в файле `exec`-слой, измерим ли он, не дублирует
  ли он canonical».

### 3.5. ФТ-5. Технические механизмы / Technical Mechanisms

| Механизм | Применимость к стеку | Сложность | Эффект | Когда |
| --- | --- | --- | --- | --- |
| **Prompt/Context caching** (кэш системного контекста модели) | Высокая для повторяющегося bootstrap (Сценарий A) | Низкая (на стороне модели/SDK) | Снижает стоимость повторов, не размер | Сейчас |
| **Selective loading по frontmatter `layer`** | Высокая, нужен загрузчик-обёртка | Средняя | Высокий | Фаза 1–2 |
| **Embeddings/RAG-индекс** (`kb/` уже имеет pipeline `sources/→processed/→fragments/`) | Средняя — фундамент уже заложен | Высокая | Очень высокий для H4/H5 | Фаза 2 |
| **Токен-счётчик в CI** (tiktoken/аналог) | Высокая | Низкая | Включает метрики/R4 | Сейчас |

Стек уже частично готов: `kb/` с hybrid-pipeline и `scripts/kb/` — основа для
RAG (D из § 3.2). Самый дешёвый старт — токен-счётчик в CI (включает R4 и
baseline) и context caching на стороне SDK.

### 3.6. ФТ-6. Метрики успеха и Baseline / Success Metrics & Baseline

| Метрика | Определение | Как собрать baseline | Целевой показатель (предлагаемый) |
| --- | --- | --- | --- |
| **Tokens/operation** | токены на онбординг / на типовой промпт | прогон реальным токенайзером по 4 сценариям | −50% на онбординг (H1, R5) |
| **Стоимость $/operation** | tokens × цена модели | из tokens/operation × тариф | пропорционально токенам |
| **Retrieval noise** | доля загруженных, но неиспользованных токенов | разметка `layer` + лог чтений агента | <20% «шума» |
| **Время отклика** | latency до первого полезного действия | замер сессии (Сценарий A) | снижение за счёт меньшего контекста |
| **Bootstrap cost** | токены «прогрева» сессии (§ 2.2) | сумма bootstrap-набора | с ~46k до ~15–20k |

**План сбора baseline (R6):** (1) выбрать эталонный токенайзер (OQ1); (2)
посчитать токены текущих hot-файлов и bootstrap-набора; (3) зафиксировать как
снимок (дата + SHA дерева) в `runs/` или `governance/` спока; (4) повторять
замер после каждой оптимизации для сравнения. Без baseline эффект недоказуем —
поэтому baseline идёт **до** любой реализации.

### 3.7. Совместимость и безопасность изменений / Compatibility & Safety

- **НФТ-3 (качество).** Разделение слоёв не удаляет знание — `Full`-слой
  сохраняется; меняется адресация. Качество не падает, т.к. агент по-прежнему
  может дочитать обоснование при необходимости.
- **НФТ-4 (совместимость).** Существующие промпты и ссылки продолжают работать:
  подход B (слои внутри файла) и E (frontmatter) аддитивны; A (отдельный файл)
  вводится с сохранением старого как `full`.
- **НФТ-6 (структура).** Все предложения совместимы с PR #254 (§ 2.5).
- **Откат.** Изменения атомарны и обратимы (порядок блоков, frontmatter-поля,
  ссылки), что снижает риск регресса.
