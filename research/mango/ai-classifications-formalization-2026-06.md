### 1. Инжиниринг промптов (Prompt Engineering)

**Описание.** Подпроцесс создания, тестирования и валидации промптов для
AI-компонентов. Промпт здесь — **интерфейс к Tool** (LLM), а инжиниринг промптов —
**подпроцесс**, дополняющий классическое «Документирование требований».

**Операции:**

| # | Операция (RU / EN) | Вход | Действие | Выход |
| --- | --- | --- | --- | --- |
| 1.1 | Анализ задачи для промпта / Prompt task analysis | бизнес-требование, контекст | определение цели промпта, входных/выходных данных | спецификация задачи для промпта |
| 1.2 | Написание черновика / Draft authoring | спецификация задачи | создание первой версии промпта | черновик промпта |
| 1.3 | Тестирование промпта / Prompt testing | черновик, тестовые данные | запуск промпта, анализ результатов | результаты тестирования |
| 1.4 | Итерация промпта / Prompt iteration | результаты тестирования | улучшение промпта по результатам | улучшенная версия промпта |
| 1.5 | Валидация промпта / Prompt validation | улучшенная версия | финальная проверка соответствия требованиям | валидированный промпт |

**Артефакты:**

- **Spec промпта / Prompt specification** (descriptive): назначение, входные данные,
  ожидаемый выход, контекст (какие знания использует), критерии качества, примеры
  использования, связь с бизнес-требованием. Создаётся **только для критичных**
  (продакшен) промптов (см. ОВ-3).
- **Исполнимый промпт / Executable prompt** (executable): текст промпта для LLM.

**Связь с классикой.** Дополняет «Документирование требований» (Вигерс, Гл. 10):
spec промпта — это требование к AI-компоненту, оформленное по тем же принципам
ясности/проверяемости, что и обычная спецификация требования.

**Пример использования.**
> Бизнес-требование: «Система должна анализировать документацию заказчика и
> извлекать требования». Операция 1.1 даёт спецификацию задачи (вход = PDF/DOCX ТЗ;
> выход = список кандидатов-требований с типом по Вигерсу). Операция 1.2 — черновик
> промпта. Операция 1.3 — прогон на корпусе тестовых ТЗ. Операция 1.5 — валидация
> по критериям качества и присвоение доверия E2 (`Candidate`).

**Критерии качества:** промпт соответствует бизнес-требованию; проходит
тестирование на тестовых данных; валидирован по критериям качества; вывод размечен
доверием E0–E4.

**Опора (trusted source).** LangChain (`ext-016`,
[реестр](../external-knowledge/external-sources-registry.md)) — паттерны
prompt-templates; инсайт о компонуемых паттернах

`ext-001`): промпт как простой компонуемый шаг, сложность — только под боль.

## ФТ-2. Эволюция операций в роадмапе Mango

Ключевой тезис: переход операции в AI-версию — **не «включение кнопки»**. Каждая
AI-ассистированная операция требует осознанного перехода: **обучение → настройка →
валидация → интеграция**. Классические операции при этом сохраняются.

### Карта четырёх фаз

| Фаза | Операции | AI-компоненты | Артефакты | Доверие/статус |
| --- | --- | --- | --- | --- |
| **Фаза 0 — текущая** | Классические (интервью, анализ документов, документирование) | Базовые промпты (без AI-assisted операций) | Стандартные (ТЗ по ГОСТ, User Story, Use Case) | — |
| **Фаза 1 — Trust & Evidence** | Классические + AI-assisted версии | Промпты с валидацией (E0–E4) | Стандартные + Prompt specification | E1–E2 `Candidate` |
| **Фаза 2 — AI-агенты** | Классические + AI-assisted + новые AI-операции | Полноценные подпроцессы (Prompt eng., RAG config, Agent orchestration) | Стандартные + AI-specific (Prompt spec, RAG config doc, Orchestration diagram) | E2–E3, часть `Applied` |
| **Фаза 3 — Мультиагентные системы** | Полная интеграция классических и AI-операций | Мультиагентные системы | Комплексные AI-решения | `Applied` |

> **Принцип эволюции, а не замены.** В каждой фазе классическая операция остаётся
> доступной. AI-версия — это **дополнительная** операция к классической (по образцу
> С3: alias-crosswalk без переименований в mango).

### Пример эволюции одной операции

| Аспект | Классическая | AI-версия (Фаза 1+) |
| --- | --- | --- |
| Название | «Анализ документов» / Document analysis | «AI-assisted document analysis» |
| Что делает | БА вручную читает ТЗ и извлекает требования | LLM извлекает кандидатов, БА валидирует |
| Артефакт | список требований | список требований + Prompt specification + метка доверия E0–E4 |
| Риск | трудозатраты, пропуски | галлюцинации → обязательна валидация (ОВ-4) |

Аналогично: «Проверка противоречий» → «AI-powered consistency checking» (опирается
на явные бизнес-правила С2 как формализуемую опору).

### План перехода для каждой операции (4 шага)

1. **Обучение / Training.** Обучение БА работе с AI-инструментами (что такое промпт,
   как читать доверие вывода, где AI ошибается).
2. **Настройка / Configuration.** Настройка AI-инструментов под операцию (промпт,
   RAG, при необходимости — оркестрация).
3. **Валидация / Validation.** Валидация AI-выводов: каждый вывод получает уровень
   доверия E0–E4; операция «Проверка AI-вывода» с критериями качества (ОВ-4).
4. **Интеграция / Integration.** Интеграция в роадмап: операция добавляется как
   alias к классической (С3), её артефакты — в реестр (С2-подход), уровень — по оси
   `requirement_level` (С1).

> **Когда начинать Фазу 2.** После завершения Фазы 1 (Trust & Evidence) и
> синхронизации С1/С2/С3 с `mango_ba_prompts`. Ориентир — сентябрь 2026 (ОВ-5).

## ФТ-3. Подготовка к синхронизации с `mango_ba_prompts` (С1, С2, С3)

Ниже — **заготовки**, пригодные для оформления issue во внешнем репозитории
`mango_ba_prompts`. Они **не** редактируют mango-ADR (внешний репозиторий), а
готовят материал. Приоритет — С1 и С2 (настоящие пробелы), С3 — лёгкий мост.

### С1 — Ось `requirement_level`

- **Описание.** Ортогональная ось-тег для семантической классификации уровня
  требования, поверх существующей классификации функциональности
  (`Domain→Capability→Feature→Atomic Function`), **не заменяя** её.
- **Значения:** `business` / `user` / `functional` / `non-functional`
  (Бизнес → Пользователь → Функция + НФТ).
- **Интеграция:** добавить в ADR-003 (онтология БА) как дополнительный тег-ось (по
  образцу того, как E0–E4 введены ортогонально Knowledge Status).
- **Пример:** «Система должна отвечать за 2 секунды» → `requirement_level:
  non-functional`. «Менеджер хочет видеть статус заявки» → `user`.
- **AI-ценность:** даёт опору для AI-трассировки вверх (функция → бизнес-цель).

### С2 — Тип «Бизнес-правило» + 5 категорий

- **Описание.** Новый тип-артефакт для явного представления бизнес-правил (внешних
  норм: политик, законов, формул), которым система обязана соответствовать.
- **Категории (5, по Вигерсу, Гл. 9):**

  | RU | EN | Пример |
  | --- | --- | --- |
  | Факты | Facts | «У заказа есть ровно один плательщик» |
  | Ограничения | Constraints | «Возврат возможен в течение 14 дней» |
  | Активаторы операций | Operation activators | «При просрочке оплаты — заблокировать номер» |
  | Выводы | Inferences | «Если клиент VIP, то скидка 10%» |
  | Вычисления | Computations | «Итог = сумма позиций × (1 − скидка)» |

- **Интеграция:** добавить в реестр артефактов ADR-003 тип `business-rule` со связью
  «бизнес-правило → порождает → функцию/ограничение».
- **Пример:** «Если клиент VIP, то скидка 10%» → тип `Business Rule`, категория
  `Inference`.
- **AI-ценность:** бизнес-правила — **первый кандидат** на автоматическую проверку
  согласованности ИИ (AI-powered consistency checking, Фаза 1).

### С3 — Crosswalk Вигерс ↔ операции mango ↔ BCREQ

- **Описание.** Справочная таблица соответствия процессов (alias-crosswalk, **без**
  переименований в mango — НФТ совместимости).
- **Интеграция:** жить в RFC Хаба; опционально — примечание со ссылкой в таксономии
  операций mango (ADR-004).

| Процесс Вигерса (RU / EN) | Операции mango (ADR-004) | Подпроцесс BCREQ (ADR-009) |
| --- | --- | --- |
| Выявление / Elicitation | `ingestion`, `understanding` | П1 |
| Анализ / Analysis | `modeling`, `research`, `impact_analysis`, `reverse_requirements` | П2, П3 |
| Спецификация / Specification | `documentation`, `solution_design` | П4 |
| Проверка / Validation | `validation`, `quality` | П5 |
| Управление / Management | `governance`, `release_readiness`, `risk_analysis` | П6 |

> **Связь AI-подпроцессов с crosswalk.** Новые AI-подпроцессы (ФТ-1) подвешиваются к
> процессам Вигерса так же, как операции mango: инжиниринг промптов → Спецификация;
> настройка RAG → Анализ/Спецификация; оркестрация → (Архитектура) Спецификация;
> тестирование AI → Проверка. Это сохраняет единую трассировку.

## ФТ-4. Полноценные примеры и критерии качества (сводка)

Для каждого нового элемента документ даёт: **описание** (что/зачем), **пример**
(конкретный кейс — см. §ФТ-1), **связь с классикой** (Вигерс/BABOK), **критерии
качества** (как проверить). Сквозной пример, связывающий всё воедино:

> **Кейс: AI-извлечение требований из ТЗ заказчика Mango.**
> 1. **Инжиниринг промптов** создаёт и валидирует промпт извлечения (артефакты:
>    Prompt spec + executable prompt). Связь: Документирование требований (Гл. 10).
> 2. **Настройка RAG** даёт промпту доступ к `kb/product-matrix.md` (артефакт: RAG
>    config doc). Связь: Проектирование решений (Гл. 12).
> 3. **Проектирование оркестрации** связывает агентов `ingestion → understanding →
>    validation` (артефакт: orchestration diagram). Связь: Архитектура (Гл. 19).
> 4. **Тестирование AI** прогоняет всё на 30 эталонных ТЗ (артефакт: AI test suite).
>    Связь: Валидация (Гл. 17).
> 5. Результат размечается доверием **E2 `Candidate`**; уровень требований — по оси
>    **С1**; найденные правила — тип **С2**; операции маппятся по **С3**.

Этот кейс соответствует Фазе 1→2 роадмапа и не заменяет ни одну классическую
операцию — он добавляет AI-версии поверх них.

## Открытые вопросы

**ОВ-1. Нужно ли создавать отдельные стандарты для AI-подпроцессов?**
Нет. Достаточно описания в этом документе/RFC. Стандарты создаются при возникновении
конкретной боли (Anti-Inflation).

**ОВ-2. Как интегрировать AI-assisted операции в существующие стандарты mango?**
Через обновление ADR-004 (таксономия операций) — добавить AI-версии как
**дополнительные** операции к классическим (alias, без переименований, С3).

**ОВ-3. Нужно ли создавать Prompt specification для всех промптов в mango?**
Нет, только для **критичных** (продакшен) промптов. Для экспериментальных достаточно
исполнимого промпта.

**ОВ-4. Как валидировать AI-выводы в AI-assisted операциях?**
Через Trust & Evidence (E0–E4): каждый AI-вывод получает уровень доверия. Валидация —
это операция «Проверка AI-вывода» с числовыми критериями качества.

**ОВ-5. Когда начинать Фазу 2 (AI-агенты) в роадмапе Mango?**
После завершения Фазы 1 (Trust & Evidence) и синхронизации С1/С2/С3 с
`mango_ba_prompts`. Ориентировочно — сентябрь 2026.

**ОВ-6. Кто владелец синхронизации между Хабом и `mango_ba_prompts`?**
Репозитории независимы; нужен явный канал. Решение и инициирование issue во внешнем
репозитории — за Пользователем (наследует Open Decision RFC).
