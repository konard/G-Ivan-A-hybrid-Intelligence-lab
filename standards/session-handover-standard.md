# Session Handover Standard

Стандарт задаёт обязательную структуру для `AI_SESSION_HANDOVER_PROMPT.md` в
HTOM-командах и проектах, которые наследуют governance Хаба. Цель — чтобы новый
чат или агент не путал **онбординг с доступом к репозиторию** и **передачу
контекста между чатами**, не применял правила spoke к HTOM-команде и не начинал
работу без проверки шаблонов и предыдущего диалога.

Draft-статус означает, что стандарт предложен для согласования по issue #215.
До утверждения он применяется к шаблону `templates/htom/AI_SESSION_HANDOVER_PROMPT.md`
как проверяемый draft-контракт.

## Область применения

Стандарт применяется к файлам с ролью `AI_SESSION_HANDOVER_PROMPT.md`: это
копируемый prompt-артефакт для старта нового чата и запуска *Runtime-онбординга*
по [governance/agent-onboarding-protocol.md](../governance/agent-onboarding-protocol.md).

Стандарт не заменяет сам протокол онбординга. Он определяет, какие блоки должны
быть в prompt-артефакте, чтобы пользователь мог безопасно передать контекст в
чат с разным уровнем доступа к репозиторию.

## Разделение типов передачи контекста

| Тип | Когда использовать | Что происходит | Результат |
| --- | --- | --- | --- |
| **Runtime-онбординг** | У агента есть доступ к репозиторию или пользователь может дать ссылки/файлы. | Агент читает правила, issue, README и релевантные артефакты из источника контекста. | Агент делает Readback и ждёт апрува; файлы не меняются до разрешения. |
| **Передача контекста между чатами** | Новый чат не имеет памяти предыдущего диалога или прямого доступа к репозиторию. | Пользователь вставляет краткий контекст диалога, ссылки, решения и ограничения. Агент явно отмечает, чего не хватает. | Новый чат продолжает работу без выдумывания недоступного контекста. |

Ключевое правило: handover prompt должен поддерживать оба сценария. Если доступ
к репозиторию есть, prompt запускает Runtime-онбординг. Если доступа нет, prompt
требует от агента запросить недостающие файлы или краткий контекст, а не
достраивать их догадками.

## Обязательные разделы

Каждый `AI_SESSION_HANDOVER_PROMPT.md` должен содержать минимум восемь разделов
или явно эквивалентных блоков внутри EXECUTION prompt.

| Раздел | Обязательное содержание | Критерий качества |
| --- | --- | --- |
| **Контекст проекта** | Имя репозитория, ссылка на Хаб, способ определить тип проекта: `HTOM-команда` или `Spoke-репозиторий`. | Prompt не называет HTOM-команду spoke и не заставляет агента применять production-spoke правила без проверки типа. |
| **Ключевые решения** | Уже принятые решения, ограничения, источники истины и известные запреты. | Агент видит решения до генерации новых вариантов. |
| **Роль и иерархия** | Роль агента, роль пользователя, право Пользователя / reviewer на финальные решения. | Prompt не передаёт AI право принимать governance-/publication-решения за человека. |
| **Контекст чата диалога** | Поле или инструкция для краткого summary предыдущего чата: что обсуждали, что решили, что осталось открытым. | Агент явно учитывает предыдущий диалог или фиксирует, что контекст не передан. |
| **Канал взаимодействия с репо** | Как агент получает файлы: прямой доступ, ссылки, вставки пользователя, GitHub issue/PR. | Если канал недоступен, агент задаёт вопрос вместо догадки. |
| **Проверка шаблонов** | Инструкция проверить наличие и актуальность локальных шаблонов (`AI_GOVERNANCE.md`, `AI_QUICK_RULES.md`, `CONTRIBUTING.md`, issue template) и ссылок на Хаб. | Агент не предполагает, что шаблон существует или совпадает с Хабом. |
| **Формат постановки задач** | Где искать issue, DoD, ограничения, operating mode и acceptance criteria. | Агент не меняет формат задачи и не заполняет пустые поля выдуманными значениями. |
| **Протокол бесшовной передачи проекта** | Ссылка на 4 шага: governance → контекст → Readback → стоп до апрува. | Prompt завершает подготовку Readback и стоп-точкой до изменений. |

## Минимальная структура файла

Рекомендуемая структура `AI_SESSION_HANDOVER_PROMPT.md`:

1. Frontmatter (`status`, `version`, `updated`, `temperature`, `executable`).
2. Короткая директива: это артефакт для копирования, не справка для пересказа.
3. `## ▶️ EXECUTION` с готовым prompt-блоком.
4. Внутри prompt-блока — восемь обязательных разделов из таблицы выше.
5. `## ℹ️ EXPLANATION` с источником истины, связью с Хабом и правилами
   синхронизации.
6. `## См. также` со ссылками на локальные правила и Хаб.

## Критерии готовности

- Файл не использует устаревшую формулировку, из которой следует, что
  `mango_ba_prompts` или другая HTOM-команда является spoke-репозиторием.
- Файл явно требует определить тип проекта перед применением правил.
- Файл содержит блок `Контекст чата диалога`.
- Файл содержит блок `Проверка шаблонов`.
- Файл ссылается на `governance/agent-onboarding-protocol.md` как canonical
  протокол Runtime-онбординга.
- Файл ссылается на этот стандарт как на критерий структуры.
- Файл оставляет финальные решения за человеком и останавливается до апрува.

## Антипаттерны

- Называть любой проект вокруг Хаба `spoke`, не проверив его тип.
- Считать, что новый чат помнит предыдущий диалог.
- Давать prompt, который требует читать файлы, но не говорит, что делать при
  отсутствии доступа к ним.
- Дублировать весь Хаб в prompt вместо ссылок и минимального context summary.
- Создавать новый governance-файл только для одного локального замечания, если
  достаточно обновить стандарт, шаблон или проверку.
