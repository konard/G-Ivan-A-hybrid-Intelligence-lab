# Contract Documentation Standard

Версия: 0.1

Дата: 2026-06-06

Статус: **черновик стандарта (`draft`)**, предложен на утверждение фаундеру.
До утверждения — рекомендация, не обязательное правило (AI_GOVERNANCE, правило 4).

Стандарт фиксирует формат **прочих контрактов** — governance-документов, которые
закрепляют **обязательства** сторон (роли, правила, политики, профили), но не
являются ни описательным исследованием, ни исполнимой процедурой. Принцип:
**обязательства сверху, обоснование под катом**.

Обоснование, сравнение практик и источники:
[research/governance/contract-documentation-format-2026-06.md](../research/governance/contract-documentation-format-2026-06.md).

## Назначение

Стандарт описывает структуру контракта из пяти элементов и единый нормативный
словарь. Он применяется к governance-контрактам Хаба (например,
[AI_GOVERNANCE.md](../AI_GOVERNANCE.md), [governance/REPO_MODEL.md](../governance/REPO_MODEL.md),
[standards/TEAM_CONTRACT.md](TEAM_CONTRACT.md), [standards/ISSUE_WORKFLOW.md](ISSUE_WORKFLOW.md),
профили) и наследуется споками.

Нормативные термины — по [RFC 2119 / BCP 14](https://www.rfc-editor.org/info/bcp14).

## Когда применять

| Документ | Стандарт |
| --- | --- |
| Фиксирует обязательства/правила/политику (кто кого к чему обязывает) | ✅ этот стандарт |
| Должен **исполняться** агентом как процедура | [EXECUTABLE_CONTRACT_STANDARD.md](EXECUTABLE_CONTRACT_STANDARD.md) |
| Производит знание для решения | [RESEARCH_DOCUMENTATION_STANDARD.md](RESEARCH_DOCUMENTATION_STANDARD.md) |

## Нормативный словарь (обязателен)

Контракт **ДОЛЖЕН** выражать обязательность через ключевые слова RFC 2119/BCP 14 с
русскими эквивалентами:

| Русское | English (RFC 2119) | Значение |
| --- | --- | --- |
| **ДОЛЖНО / НЕ ДОЛЖНО** | MUST / MUST NOT | абсолютное требование/запрет |
| **СЛЕДУЕТ / НЕ СЛЕДУЕТ** | SHOULD / SHOULD NOT | рекомендация; отклонение допустимо при веской причине |
| **МОЖНО** | MAY | необязательная возможность |

Прочие, нечёткие формулировки обязательности («желательно», «по возможности»)
**НЕ СЛЕДУЕТ** использовать в нормативных пунктах.

## Обязательная структура

Контракт **ДОЛЖЕН** располагать содержание в следующем порядке:

1. **Стороны и область** — кто кого обязывает; scope (`repo-wide` / проект /
   роль); к чему контракт применяется. **ДОЛЖНО** идти первым.
2. **Обязательства** — пункты на нормативном языке (§выше), оформленные списком
   или таблицей. **ДОЛЖНЫ** идти сразу за областью.
3. **Definition of Done / критерии соответствия** — как проверить выполнение;
   **СЛЕДУЕТ** ссылаться на исполнимую проверку, если она есть (например,
   `tools/validate-repository-structure.sh`).
4. **Эскалация / исключения** — что делать при конфликте, override, кто решает.
5. **Обоснование и контекст** — *под катом*, ниже обязательств. Читатель-исполнитель
   **НЕ ДОЛЖЕН** быть обязан читать обоснование, чтобы соблюсти контракт.

Термины контракт **НЕ ДОЛЖЕН** переопределять — он **ДОЛЖЕН** ссылаться на
[standards/GLOSSARY.md](GLOSSARY.md) (single source of truth).

## Скелет документа

```markdown
---
# frontmatter Хаба: status/version/updated/ai-generated; executable: false; scope; related_*
---

# <Имя контракта>

## Стороны и область
<кто кого обязывает; scope; к чему применяется>

## Обязательства
1. <Сторона> ДОЛЖНА …
2. <Сторона> СЛЕДУЕТ …

## Definition of Done / проверка
<как убедиться, что соблюдено; ссылка на валидатор, если есть>

## Эскалация и исключения
<конфликты, override, кто принимает решение>

## Обоснование
<контекст, альтернативы, последствия — под катом>
```

## Краевые случаи и границы

- **Микро-контракт.** Разделы «Стороны/Область» и «Обязательства» обязательны
  всегда; «Обоснование» **МОЖНО** опустить, если решение очевидно.
- **Контракт + исполнение.** Если документ ещё и *запускается* агентом,
  применяется [EXECUTABLE_CONTRACT_STANDARD.md](EXECUTABLE_CONTRACT_STANDARD.md)
  (команда сверху); контракт-формат — про *обязательства*, не про *запуск*.
- **Совместимость с frontmatter.** Стандарт описывает *тело* контракта и не
  отменяет обязательный frontmatter Хаба.

## Миграция

Существующие контракты **НЕ ДОЛЖНЫ** переписываться массово (Anti-Inflation).
Документ приводится к формату при следующем существенном изменении его
содержания.

## Проверка соответствия

- Стороны и область заявлены в первом разделе? → иначе добавить.
- Обязательства выражены через ДОЛЖНО/СЛЕДУЕТ/МОЖНО, без нечётких формулировок? →
  иначе нормализовать.
- Есть критерий проверки (DoD)? → иначе добавить.
- Обоснование вынесено *ниже* обязательств? → иначе переставить.
- Термины не переопределяются, а берутся из `GLOSSARY.md`? → иначе заменить
  ссылкой.
