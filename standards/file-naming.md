# Правило именования файлов

Стандарт фиксирует единое правило именования файлов для новых артефактов
репозитория. Цель — сохранить GitHub-конвенцию для корневых файлов и сделать
вложенные структуры читаемыми, удобными для терминала и устойчивыми к росту.

## Правило именования файлов

| Расположение | Стиль | Пример | Почему |
| --- | --- | --- | --- |
| **Корень репозитория** (`/`) | `UPPERCASE_WITH_HYPHENS.md` | `README.md`, `CONCEPT.md`, `CHANGELOG.md` | Конвенция GitHub, файлы видны сразу |
| **Вложенные каталоги** (`standards/`, `research/`, `projects/`, `kb/` и др.) | `lowercase-with-hyphens.md` | `classification-glossary.md`, `rag-mapping-roadmap.md` | Читаемость, сортировка, удобно в терминале, масштабируемость |

## Правила именования файлов в standards/ и governance/

- Все файлы в `standards/` и `governance/` должны быть в `kebab-case`: строчные
  буквы, цифры и дефисы.
- Исключения: только системные файлы `README.md`, `LICENSE`, `CHANGELOG.md`.
- `CAPS_LOCK` запрещён для документов внутри `standards/` и `governance/`.

### Примеры для standards/ и governance/

- Правильно: `research-profile.md`
- Правильно: `agent-onboarding-protocol.md`
- Правильно: `executable-contract-standard.md`
- Неправильно: `RESEARCH_DOCUMENTATION_STANDARD.md`
- Неправильно: `AGENT_ONBOARDING.md`
- Неправильно: `Research_Documentation_Standard.md`

### Исключения

- Файлы с версиями: `migration-audit-2026-05.md` (дата в имени)
- Файлы с префиксами: `README-old.md` (суффикс `-old` для унаследованных файлов)

### Проверка

- Новый файл не соответствует правилу? Отклонить в ревью.
- Существующий файл нарушает правило? Переименовать при следующем
  редактировании, не блокируя текущую работу.

## Правило суффиксов уровня детализации (`-Index` / `-Summary` / `-Full`)

Когда у одного концепта существуют файлы **разных уровней детализации**
(рамка `Index → Summary → Full` из
[governance/rfc/documentation-architecture-balance.md](../governance/rfc/documentation-architecture-balance.md)),
используется **базовое имя + суффикс уровня**. Суффикс делает уровень детализации
видимым прямо в имени файла, чтобы навигация и lazy loading были предсказуемыми:
агент открывает `-Index` для отбора, `-Summary` для контекста задачи и `-Full`
только когда срез выше не дал ответа.

| Суффикс | Уровень | Назначение | Пример |
| --- | --- | --- | --- |
| `-Index` | минимум | Точка входа и навигация: таблица/список ссылок без деталей. | `docs/ecosystem-map-Index.md` |
| `-Summary` | краткий | Сжатый контекст для задачи: цель, фокус, границы. | `AI_PROJECT_CONTEXT-Summary.md` |
| `-Full` | полный | Расширенный/полный документ со всеми деталями и rationale. | `solution-concept-Full.md` |

**Правила.**

- Суффикс пишется через дефис в `PascalCase`: ровно `-Index`, `-Summary`,
  `-Full`. Это сознательное исключение из общего lowercase-правила для вложенных
  файлов — суффикс маркирует уровень и должен читаться с одного взгляда.
- Базовое имя файла подчиняется обычному правилу расположения: корневые файлы —
  `UPPERCASE_WITH_HYPHENS` (например, `AI_PROJECT_CONTEXT-Summary.md`), вложенные
  — `lowercase-with-hyphens` (например, `docs/ecosystem-map-Index.md`).
- Суффикс добавляется **только при наличии нескольких уровней** одного концепта
  (Anti-Inflation): одиночный документ не получает суффикс «на вырост».
- Один концепт не обязан иметь все три уровня. Допустимы пары `Index + Summary`
  или `Summary + Full`; набор уровней определяется реальной потребностью.
- Повышение/понижение уровня (а значит, и переименование) — **решение человека**
  (см. Decision Scope в RFC выше), а не автоматическое действие агента.

## Правило для AI-ассистентов

В режиме `structured` строго следуй правилу: корневые файлы остаются в
`UPPERCASE_WITH_HYPHENS.md`, вложенные файлы создаются как
`lowercase-with-hyphens.md`. Не добавляй исключения без явного указания в issue,
PR comment или другом human-authored governance документе.
