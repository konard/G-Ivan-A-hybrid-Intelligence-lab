## Context

Репозиторий `mango_ba_prompts` совмещает две несовместимые роли. Первая —
операционка: боевые промпты с приватными данными Mango, журналы прогонов и
golden-sets на реальных кейсах. Вторая — методология: переиспользуемые паттерны,
стандарты и публичные примеры.

Совмещение создаёт три проблемы, каждая из которых блокирует одну из ролей.
Приватные данные не могут быть опубликованы без риска утечки, поэтому репозиторий
вынужденно остаётся закрытым. Из-за смешанного контента закрытый репозиторий не
может стать полноценной публичной витриной методологии. Разные темпы изменений
требуют разных процессов ревью: методология выигрывает от внешнего ревью, а
операционка требует внутреннего.

Фаундер принял решение 2026-07-10: разделить проект на два независимых
репозитория с разными жизненными циклами и аудиториями. Этот ADR не переоткрывает
решение и не принимает новых архитектурных решений — он переводит уже
утверждённый план в нормативную форму, необходимую для планирования миграции
(B-080) и создания репозиториев (B-081, B-082). Создание физических
репозиториев, физическая миграция файлов и настройка синхронизации остаются вне
scope этого ADR.

## Decision

Разделить `mango_ba_prompts` на два независимых репозитория: публичный
`ai-ba-playbooks` и приватный `mango-ba-prompt-library`. Решение фиксирует четыре
нормы.

### 1. Публичный репозиторий `ai-ba-playbooks`

Имя `ai-ba-playbooks` универсально и **не содержит упоминания бренда «Mango»**:
репозиторий является витриной методологии, а не продуктом одной компании. Запрет
на бренд относится к имени репозитория и к его публичному содержимому.

Репозиторий следует архетипу B (Prompt & Pattern Library) по
[ADR-001](2026-06-adr-001-ecosystem-infrastructure-methodology.md). Каталоги:

| Каталог | Назначение |
| --- | --- |
| `prompt-library/` | Специфичный продуктовый каталог промптов; на старте — `telecom/`. |
| `patterns/` | Переиспользуемые паттерны. |
| `standards/` | Публичные стандарты библиотеки. |
| `examples/` | Только обезличенные примеры. |
| `docs/` | Методологическая документация, RFC/ADR библиотеки. |
| `templates/` | Копируемые поверхности. |
| `app/` | Опционально: презентационный слой. |

Каталог `education/` в публичном репозитории **категорически отсутствует**:
образовательные активы живут в `education/` Хаба как активы архетипа A по
[ADR-007](2026-07-adr-007-hub-root-structure.md).

### 2. Приватный репозиторий `mango-ba-prompt-library`

Имя `mango-ba-prompt-library` сохраняет бренд: репозиторий закрыт и является
операционкой Mango. Каталоги:

| Каталог | Назначение |
| --- | --- |
| `prompts/` | Боевые промпты. |
| `kb/processes/` | Операционные знания и process prompts. |
| `runs/` | Записи прогонов. |
| `evals/` | Golden-sets и оценки на реальных кейсах. |
| `internal-rfc/` | Внутренние предложения. |
| `internal-docs/` | Внутренняя документация. |

Репозиторий работает **без GitHub Actions**: проверки выполняются только
локальными валидаторами. Приватные данные не покидают периметр CI-раннеров.

### 3. Синхронизация

Синхронизация **строго односторонняя: приватный → публичный**. Обратное
направление запрещено: публичный репозиторий не является источником операционных
изменений, и любой обратный поток вернул бы смешение ролей, ради устранения
которого выполняется разделение.

На старте синхронизация — **ручной отбор фаундером**: человек решает, какой
артефакт обезличен и готов к публикации. Автоматизация (Smart Sync с
фильтрацией) вводится **только под доказанную операционную боль**, а не
превентивно. Триггером служат данные эксплуатации, а не календарный срок.

### 4. Связь с Хабом

Публичный репозиторий `ai-ba-playbooks` **импортирует методологию из Хаба**
`hybrid-Intelligence-lab` и не переопределяет её. Зрелые практики, выросшие в
публичном репозитории, могут вернуться в Хаб **через RFC** — то есть через явный
human decision gate по
[ADR-002](2026-06-adr-002-artifact-document-methodology.md), а не прямым
копированием.

Этот ADR не меняет стандарты Хаба: ADR-001, ADR-002 и ADR-007 остаются
неизменными и применяются к конкретному случаю Mango.

## Decision Drivers

- Приватные данные и публичная витрина имеют несовместимые требования к
  видимости; разделение по репозиториям — единственная граница, которую нельзя
  нарушить по невнимательности.
- Разные аудитории требуют разных процессов ревью: внешнее для методологии,
  внутреннее для операционки.
- Универсальное имя без бренда даёт публичному репозиторию карьерную и
  репутационную ценность вне контекста одного работодателя.
- Односторонняя синхронизация делает направление потока данных проверяемым:
  утечка возможна только через один явный шлюз с человеком в петле.
- Отсутствие `education/` в публичном репо предотвращает конкуренцию с
  `education/` Хаба и раздвоение источника истины.

## Alternatives Considered

Решение принято фаундером 2026-07-10 в issues #411/#413; этот ADR фиксирует
выбор, а не проводит новый анализ вариантов. Решающая развилка — оставить один
смешанный репозиторий или разделить на два — закрыта в пользу разделения.
Альтернатива «один репозиторий с приватными ветками или submodules» отклонена
фаундером: она сохраняет риск утечки при ошибке конфигурации и не даёт разным
жизненным циклам разойтись. Альтернатива «автоматическая синхронизация с
самого старта» отложена: автоматизация фильтрации приватных данных дороже и
рискованнее ручного отбора, пока объём публикаций мал.

## Consequences

Положительные последствия:

- чистота данных: приватные кейсы, прогоны и golden-sets физически не попадают в
  публичный периметр;
- публичное портфолио: `ai-ba-playbooks` становится витриной методологии с
  карьерной ценностью, не привязанной к бренду;
- ускорение внутренней итерации: операционка меняется без внешнего ревью и без
  оглядки на публичную аудиторию;
- проверяемая граница: направление синхронизации однозначно, а Хаб остаётся
  единственным источником методологии.

Компромиссы и риски:

- ручной отбор промптов на старте требует времени фаундера и масштабируется
  плохо; при росте объёма это станет операционной болью и триггером для Smart
  Sync;
- поддержка двух репозиториев удваивает стоимость конфигурации, валидаторов и
  навигации;
- дрейф: публичная методология может разойтись с приватной практикой, если
  возврат зрелых практик через RFC не выполняется регулярно;
- отсутствие GitHub Actions в приватном репо переносит ответственность за
  проверки на дисциплину исполнения локальных валидаторов.

Downstream-работа не дублируется здесь: план миграции остаётся B-080, создание
репозиториев — B-081 и B-082, физическая миграция — B-083, настройка и
тестирование синхронизации — B-084 в [backlog](../../pr-ops/backlog.md).

## Compliance and Validation

Этот ADR соответствует
[ADR Structure Standard](../../standards/adr-structure-standard.md): использует
необходимый frontmatter и обязательные секции ADR, а детали плана миграции
делегирует B-080. Репозиторные проверки для этого draft:

- `bash tools/test-frontmatter-validator.sh`;
- `bash tools/test-smart-sync.sh`;
- `./tools/validate-file-naming.sh`;
- `./tools/validate-frontmatter.sh .`;
- `./tools/validate-repository-structure.sh`;
- `python3 tools/generate-manifest.py --check`.

Соблюдение решения проверяется человеком на downstream-задачах: B-081 проверяет
отсутствие `education/` и бренда «Mango» в публичном репозитории, B-082 —
отсутствие GitHub Actions в приватном, B-084 — одностороннее направление
синхронизации. Автоматизированная проверка периметра приватных данных не
вводится этим ADR.

## Lifecycle

ADR находится в статусе `proposed`: решение фаундера зафиксировано 2026-07-10, а
merge PR является явным acceptance gate по ADR Structure Standard. После принятия
B-079 переходит в `review`, а B-080 получает основание составить план миграции.

Пересмотр требует нового RFC/ADR или явного superseding decision. Ожидаемые
триггеры пересмотра: ручной отбор становится доказанной операционной болью (тогда
вводится Smart Sync), либо появляется потребность в третьем репозитории или в
обратном потоке синхронизации. Замещение требует backlink на новый decision
record; календарный срок сам по себе не является trigger.
