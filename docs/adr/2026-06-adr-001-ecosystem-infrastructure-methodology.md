# ADR-001: Методология инфраструктуры проектов экосистемы

## Контекст

Issue #276 переводит результаты двух входящих исследований в решение:

- [research/hub/2026-06-23-repository-structure-concept.md](../../research/hub/2026-06-23-repository-structure-concept.md)
  - концепция структуры каталогов для архетипов, включая до-исследование
  Соглашения "Архитектура инфраструктуры проектов экосистемы";
- [research/hub/2026-06-25-artifact-inventory-and-classification.md](../../research/hub/2026-06-25-artifact-inventory-and-classification.md)
  - инвентаризация артефактов и правила маршрутизации документов.

В issue также зафиксировано утвержденное Соглашение со структурой универсального
ядра, архетипами A-D, уровнями исполнимости IL-0..IL-3 и правилами размещения.
Этот ADR фиксирует решение. Он не выполняет физическую миграцию текущего дерева
Хаба: переименование каталогов и перенос существующих файлов остаются отдельными
reviewable PR.

## Решение

Принять методологию инфраструктуры проектов экосистемы как сочетание:

1. универсального базового ядра для любого репозитория;
2. архетипических расширений A-D;
3. правила "базовое / рекомендуемое / специфичное";
4. размещения по корню наименьшего контекста исполнения;
5. явного разделения операционно применимых знаний, описательной документации,
   решений и результатов выполнения.

Термин "боевые знания" заменяется на технически точный термин
**операционно применимые знания**: это знания, которые прямо используются
агентом, процессом, валидатором или человеком в выполнении задачи. Они могут
быть машинно-валидируемыми, AI-исполнимыми или краткими reference-артефактами,
но не являются накопительной библиотекой общего чтения.

## Универсальное ядро

Целевая структура нового репозитория экосистемы:

```text
{repo}/
├── ai-rules/              # правила AI-агента, плоско; подкаталоги через 2FA-обоснование
├── ai-governance/         # политики, compliance, риски AI
├── pr-ops/                # управление задачами, PR и review, плоско; подкаталоги через 2FA
├── standards/             # стандарты, плоско; подкаталоги через 2FA
├── docs/
│   ├── analysis/
│   ├── reports/
│   ├── adr/
│   ├── rfc/
│   └── practice/
├── kb/                    # операционно применимые знания
│   ├── taxonomy/
│   ├── roles/
│   ├── rules/
│   ├── processes/{name}/
│   ├── experiments/
│   └── patterns/
├── runs/                  # результаты выполнения и воспроизводимые run-records
├── templates/
│   ├── pr-ops/
│   ├── ai-ops/
│   └── kb/
├── tools/                 # поддерживающие утилиты с подкаталогами по необходимости
├── app/                   # приложение, если репозиторий сам содержит runtime
├── GOVERNANCE.md          # якорь org-governance
├── README.md
├── CHANGELOG.md
├── CONTRIBUTING.md
├── LICENSE
├── .gitignore
└── .gitattributes
```

Правило для плоских каталогов: подкаталог разрешен только при двухфакторном
обосновании - есть операционная боль и есть повторяемое использование, которое
невозможно выразить плоским именованием без потери ясности.

## Архетипы

| Архетип | Базовые расширения | Рекомендуемые расширения | Профильные расширения |
| --- | --- | --- | --- |
| A. Governance & Knowledge Hub | `research/`, `practices/` | `projects/`, `knowledge/` | каталоги для доменных research-направлений |
| B. Prompt & Pattern Library | `prompts/` | `patterns/`, `knowledge/` | `kb/processes/*` для процессных промптов |
| C. Product Spoke / Application Runtime | `src/`, `tests/` | `apps/`, `packages/`, `libs/`, `infra/` | `db/`, `config/`, `website/` или `apps/web` |
| D. Education / Learning Package | `examples/`, `exercises/` | `solutions/`, `assets/` | course/runtime demo folders |

Web Portal и Library/SDK не создают отдельные архетипы. Они являются профилями
архетипа C: одно веб-приложение может использовать `website/`, несколько
приложений используют `apps/*`; библиотека/SDK использует `packages/` или
`libs/` без `apps/`.

## Правила размещения

| Ситуация | Решение |
| --- | --- |
| Нужно создать инфраструктурный каталог. | Сначала определить архетип и уровень: базовый, рекомендуемый или специфичный. |
| Документ исполняется в конкретном процессе. | Размещать в корне наименьшего контекста исполнения, например `kb/processes/{name}/`. |
| Документ объясняет решение или вариант. | Размещать в `docs/analysis/`, `docs/rfc/` или `docs/adr/` по lifecycle. |
| Норма применяется повторно. | Размещать в `standards/`, если это формат/критерий, или `ai-rules/`, если это правило поведения AI-агента. |
| Файл фиксирует результат выполнения. | Размещать в `runs/`, не смешивать с `kb/`. |
| Нужен код проверки или обслуживания. | Размещать в `tools/`, не в `scripts/`, если проект не имеет уже принятой обратной конвенции. |

Хронологические research и analysis файлы используют `YYYY-MM-DD-name.md`.
ADR в `docs/adr/` используют формат Хаба `YYYY-MM-adr-NNN-name.md`.

## Переходный режим для текущего Хаба

Текущий репозиторий уже содержит действующие исторические пути:
`governance/`, `AI_GOVERNANCE.md`, `practices/`, `standards/`, `templates/htom/`
и `governance/rfc/`. Они остаются активными до отдельного решения о физической
миграции.

Для новых решений:

- ADR создаются в `docs/adr/`;
- существующие RFC Хаба остаются в `governance/rfc/` до миграционного RFC/ADR;
- новая структура не применяется ретроспективно к каждому существующему файлу;
- любое переименование корневых каталогов требует отдельного PR с картой
  перехода, обновлением ссылок и валидаторов.

## Последствия

Положительные:

- появляется единый целевой шаблон инфраструктуры для будущих репозиториев;
- архетипы расширяют ядро без копирования лишних пустых каталогов;
- граница `kb/` становится технической: операционно применимые знания, а не
  расплывчатая "knowledge base";
- `runs/` отделены от знаний и не становятся базой данных внутри `kb/`;
- дальнейшие миграции можно выполнять малыми reviewable PR.

Компромиссы:

- текущий Хаб на время сохраняет старые пути, поэтому карта "current -> target"
  должна быть явно указана в миграционных задачах;
- `governance/rfc/` и `docs/rfc/` будут сосуществовать до отдельного решения;
- AI-агенты обязаны читать этот ADR как решение о целевой методологии, а не как
  команду немедленно перестраивать дерево.

## Статус входящих исследований

Исследование структуры каталогов переведено в `status: reviewed`, потому что
его выводы рассмотрены и зафиксированы этим ADR. Само исследование остается
источником rationale и вариантов, а не canonical-нормой.
