# Стандарт оформления продуктового видения и концепции

Версия: 1

Дата среза: 25 мая 2026 г.

Связанная задача: [issue #11](https://github.com/G-Ivan-A/hybrid-Intelligence-lab/issues/11)

## Задача

Выбрать стандарт, по которому целесообразно вести продуктовое видение и
концепцию репозитория в GitHub, а также готовить материал для стейкхолдеров и
научного сообщества.

## Сравнение подходов

| Подход | Для чего хорош | Ограничения | Применимость к репозиторию |
| --- | --- | --- | --- |
| PRD | Подробно описывает продукт, требования, scope, метрики и ограничения. | Часто слишком implementation-heavy для knowledge hub и научной аудитории. | Использовать для конкретных проектов в `/projects`, но не как главный стандарт концепции репозитория. |
| Product Vision Board | Хорошо объясняет аудиторию, потребности, ценность, продукт и бизнес-цели. | Недостаточно фиксирует решения, варианты и rationale. | Подходит как основа `PRODUCT_VISION.md` и stakeholder-презентации. |
| Lean Canvas | Быстро показывает проблему, решение, каналы, аудиторию, economics. | Сильнее заточен под startup/business model, чем под research/education ecosystem. | Можно использовать как дополнительный one-pager, но не как основной стандарт. |
| Opportunity Solution Tree | Хорош для discovery и связи outcomes с решениями. | Не описывает governance, структуру репозитория и публикационные правила. | Полезен для отдельных проектных инициатив, но не для базовой концепции. |
| ADR | Отлично фиксирует контекст, варианты, решение и последствия. | Плохо работает как презентация видения для широкой аудитории. | Использовать для структурных решений и стандартов, но не как единственный формат. |
| RFC-lite | Удобен для обсуждения изменений: проблема, варианты, proposal, impact, open questions. | Требует дисциплины и владельца, иначе превращается в длинный черновик. | Подходит для концептуальных документов в `docs/concept`. |
| IMRAD / научная статья | Подходит научному сообществу: introduction, method, results, discussion. | Слишком формален для ежедневного ведения репозитория и PO-видения. | Использовать для публикационных research-артефактов, а не для базового governance. |

## Решение

Для этого репозитория выбран комбинированный стандарт:

1. **Product Vision Board** - для короткого публичного видения в
   [PRODUCT_VISION.md](../../PRODUCT_VISION.md).
2. **RFC-lite + ADR rationale** - для концептуальных решений в
   `docs/concept/`, где нужно сравнивать варианты и фиксировать последствия.
3. **IMRAD-like структура** - для исследований, которые готовятся к научной
   или международной публикации.

Такой стандарт лучше соответствует смешанной природе репозитория:
стейкхолдерам нужен понятный vision, команде нужен traceable rationale,
научному сообществу нужны метод, источники и воспроизводимость.

## Канонический формат `PRODUCT_VISION.md`

| Раздел | Назначение |
| --- | --- |
| Назначение | Что это за репозиторий и какую проблему он решает. |
| Целевые аудитории | Для кого создается hub и какую ценность получает каждая аудитория. |
| Принципы продукта | Небольшой набор решений, которые ограничивают будущие изменения. |
| Границы | Что входит и что явно не входит в репозиторий. |
| Целевая архитектура | Почему выбран central knowledge hub + spoke-проекты. |
| Ближайшие результаты | Что должно появиться на текущем и следующем этапах. |
| Связь с внешними репозиториями | Как hub соотносится с `open-ai.ru` и будущими проектами. |

## Канонический формат документов `docs/concept`

| Раздел | Назначение |
| --- | --- |
| Цель | Какой вопрос решает документ. |
| Контекст | Что известно из issue, предыдущих проектов и ограничений. |
| Сравнение вариантов | Таблица вариантов с плюсами, минусами и решением. |
| Выбранное решение | Описание решения, применимого сейчас. |
| Последствия | Что становится проще, какие риски остаются. |
| Следующие шаги | Какие шаблоны, проверки или follow-up нужны. |

## Формат для презентации стейкхолдерам

Презентация должна строиться из тех же разделов, чтобы не появлялась отдельная
несинхронизированная версия:

| Слайд | Источник в репозитории | Содержание |
| --- | --- | --- |
| 1. Название и цель | `PRODUCT_VISION.md` | Research and Edu AI Lab как knowledge hub. |
| 2. Зачем нужен hub | `PRODUCT_VISION.md` | Проблема смешения исследований, обучения, проектов и AI-governance. |
| 3. Аудитории | `PRODUCT_VISION.md` | Founder & PO, Команда C, исследователи, партнеры, AI-агенты. |
| 4. Сравнение структур | `docs/concept/repository-structure.md` | Почему выбран Hub-and-Spoke. |
| 5. Целевая структура | `README.md`, `docs/concept/repository-structure.md` | Каталоги и их назначение. |
| 6. Правила hybrid teams | `AI_GOVERNANCE.md` | Роли, режимы, Definition of Done. |
| 7. Языковая политика | `docs/concept/repository-structure.md` | ru/en публикации и отказ от асимметричного `ru_res`. |
| 8. Roadmap | `PRODUCT_VISION.md` | Ближайшие результаты и следующие шаги. |

## Формат для научного сообщества

Для исследований и фреймворков, которые выходят за пределы внутреннего
использования, поверх репозиторного артефакта готовится публикационная версия:

| Раздел | Что фиксировать |
| --- | --- |
| Introduction | Проблема, контекст, исследовательский вопрос. |
| Method | Источники, критерии отбора, эксперименты, ограничения. |
| Results | Таблицы, наблюдения, классификации, сравнение вариантов. |
| Discussion | Интерпретация, риски, применимость, связь с практикой hybrid teams. |
| Conclusion | Выводы, рекомендации, дальнейшие исследования. |
| Reproducibility | Ссылки на `experiments/`, данные, команды запуска, версии. |

## Правило синхронизации

`PRODUCT_VISION.md` остается коротким входным документом. Если меняется
структура, governance или стандарт оформления, нужно обновить ближайший
документ в `docs/concept/` или `docs/governance/` и добавить ссылку из README,
если путь важен для новых участников.
