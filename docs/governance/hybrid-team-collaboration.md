# Правила взаимодействия гибридных команд

Версия: 1

Дата среза: 25 мая 2026 г.

## Цель

Определить практические правила работы Founder & PO, Команды C, human
reviewers и AI-агентов при развитии репозитория как open knowledge hub.

## Ролевая модель

| Роль | Права | Обязанности | Типовые артефакты |
| --- | --- | --- | --- |
| Founder & PO | Утверждает видение, приоритеты, границы публикации, ключевые решения. | Формулирует цели, дает контекст, принимает спорные решения. | Vision, issue, комментарии, acceptance criteria. |
| Команда C | Предлагает структуру, помогает сопровождать материалы, готовит review. | Проверяет полноту, согласованность, читаемость и применимость. | Таблицы сравнения, ревью, предложения по структуре. |
| Human reviewer | Может запросить изменения перед merge. | Проверяет источники, риски, соответствие аудитории и governance. | PR review, inline comments, approval/request changes. |
| AI agent | Может создавать ветку, правки, тесты, черновики, PR update в пределах задачи. | Читает issue, не выходит за scope, объясняет решения, запускает проверки. | Markdown artifacts, scripts, PR body, summaries. |
| External contributor | Может предлагать улучшения через issue/PR. | Следует структуре, языковой политике и правилам безопасной публикации. | Research note, course module, framework proposal. |

## Поток работы

| Шаг | Ответственный | Результат |
| --- | --- | --- |
| 1. Постановка задачи | Founder & PO или уполномоченный участник | Issue с целью, контекстом, ограничениями и ожидаемым языком результата. |
| 2. Первичное исследование | AI agent / Команда C | Прочитаны issue, комментарии, связанные файлы, похожие PR. |
| 3. Сравнение вариантов | AI agent / Команда C | Таблица вариантов, recommendation, open questions. |
| 4. Создание артефакта | AI agent / contributor | Документ или структура в правильном каталоге. |
| 5. Проверка | AI agent / contributor | Локальная проверка, воспроизводимые команды, self-review diff. |
| 6. Human review | Founder & PO / reviewer | Решение: approve, request changes, clarify. |
| 7. Публикация | Maintainer | Merge PR, обновление ссылок, follow-up issue при необходимости. |

## Правила постановки задач для AI

Хорошая задача для AI-агента должна содержать:

1. Цель и контекст.
2. Тип результата: framework, education, research, project, standard или governance.
3. Язык результата: русский, английский или bilingual ru/en.
4. Ограничения: что нельзя менять, какие источники обязательны.
5. Критерии приемки: какие файлы или проверки должны появиться.
6. Требование к сравнительным таблицам, если есть выбор структуры, стандарта или подхода.

## Правила подготовки артефактов

| Тип | Обязательные элементы |
| --- | --- |
| Framework | Проблема, существующие подходы, gap, предлагаемая методология, ограничения, пример применения. |
| Education | Целевая аудитория, цели обучения, структура модулей, материалы, сценарий преподавателя, результаты. |
| Research | Вопрос, метод, источники, анализ, вывод, ограничения, воспроизводимость. |
| Project | Цель, роли, контекст, границы, knowledge base, промпты или процессы, связи с research/frameworks. |
| Governance | Роли, правила, workflow, критерии готовности, когда эскалировать. |

## Review checklist

| Вопрос | Да/Нет |
| --- | --- |
| Артефакт лежит в правильном каталоге? |  |
| Есть понятная аудитория и назначение? |  |
| Варианты сравнивались в таблице, если был выбор? |  |
| Выводы отделены от допущений? |  |
| Есть ссылки на связанные issue, PR, исследования или эксперименты? |  |
| Навигация обновлена? |  |
| Чувствительные данные отсутствуют или санитизированы? |  |
| Проверки запущены и указаны в PR? |  |

## Правила эскалации

Вопрос к Founder & PO или reviewer обязателен, если:

- нужно выбрать направление, влияющее на весь репозиторий;
- есть конфликт между open education и коммерческим проектом;
- нужно решить, публиковать ли материал во внешнюю аудиторию;
- исследование требует источников, которых нет в открытом доступе;
- AI-агент не может подтвердить выводы методически.
