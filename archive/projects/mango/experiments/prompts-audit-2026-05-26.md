# Аудит экспериментов и выделение готовых промптов

Версия: 0.1

Дата: 2026-05-26

## Назначение

Этот аудит отделяет исследовательские отчеты в `projects/mango/experiments/`
от готовых промптов, которые пользователь может скопировать в ChatGPT, Claude
или другой внешний ИИ-чат.

Критерий готовности промпта: он сам объясняет роль, порядок диалога, правила
качества, ожидаемый формат результата и явно просит пользователя прислать вход.

## `tz-stats-prototype-2026-05.md`

### Что работает

- Есть понятная задача: сопоставить ТЗ с классами Mango и обновить накопленную
  статистику.
- Зафиксирован полезный контракт таблицы и JSON-блока.
- Есть правило `current-iteration-delta = 1` на класс в одном ТЗ.
- Отдельно описаны `mapping-status`, `confidence` и логирование проблем.

### Что не работает как готовый промпт

- Файл смешивает исследование, формат хранения истории, тесты, результаты и
  предложения по улучшению.
- Пользователь не может быстро понять, что именно нужно скопировать в чат.
- Часть инструкций завязана на внутренние каталоги эксперимента, а не на живой
  диалог с внешним ИИ.

### Ядро промпта

1. Роль: Mango TZ Stats Analyst.
2. Входы: `current_tz_id`, текст ТЗ, предыдущий отчет, классификатор.
3. Правила: не выдумывать классы, считать один раз на класс в ТЗ, логировать
   отсутствующие классы.
4. Выход: Markdown-таблица и JSON с одинаковыми значениями.

### Упрощения для `_simple`

- Вместо ссылки на `classification.md` промпт сначала просит пользователя
  прислать рабочий список классов.
- Если списка нет, ИИ предлагает черновую классификацию и маркирует ее как
  неподтвержденную.
- Product и commercial элементы разделяются простым правилом: функция продукта
  отдельно, договорные условия отдельно.

### Критичные ссылки для `_exp`

- `research/mango/classification.md` - активный каталог Product Layer и
  Commercial Layer.
- `research/mango/classification-tz.md` - вспомогательный корпус паттернов ТЗ,
  не источник финальных `class-code`.

## `user-story_gen-from-raw-request_2026-05-26.md`

### Что работает

- Сначала выполняется детекция входа: роль, ценность, полнота, тип требования,
  сленг и неизвестные термины.
- Неполный запрос не превращается в финальную User Story без предупреждений.
- YAML-мета и логи помогают проверять `confidence` и `mapping_status`.
- Тесты A-D показывают работу с полным, неполным и разговорным входом.

### Что не работает как готовый промпт

- Файл является отчетом о прогоне, а не текстом для копирования.
- Прототип требует много служебных секций, которые замедляют обычный чат.
- RAG-навигация полезна только при наличии доступа к выдержкам из репозитория.

### Ядро промпта

1. Принять сырой запрос.
2. Проверить полноту роли, цели и ценности.
3. Нормализовать спорные термины.
4. Сгенерировать User Story и Acceptance Criteria только при достаточных
   данных.
5. Вернуть вопросы вместо домыслов.

### Упрощения для `_simple`

- Убрать обязательный `input_hash`, стабильность секций и полный логовый
  контракт.
- Оставить практичный формат: статус, детекция, User Story, Acceptance
  Criteria, вопросы и мета.
- Заменить RAG-mapping на эвристику: роль, возможность, ценность, границы,
  спорные термины.

### Критичные ссылки для `_exp`

- `research/mango/classification.md` - mapping `Domain -> Capability -> Feature
  -> Atomic Function`.
- `projects/mango/standards/classification-glossary.md` - определения уровней
  и базовая терминология Mango.
- `projects/mango/kb/glossary.md` - целевой словарь проекта, когда он появится.

## `usecase_gen-stepwise-alignment_2026-05-26.md`

### Что работает

- Use Case не генерируется сразу: сначала проверяется готовность, затем
  согласуются акторы и компоненты.
- Есть явная остановка после каждого шага.
- Разделены основной поток, альтернативы, исключения и постусловия.
- Логи согласования показывают `Proposed -> User Response -> Final`.

### Что не работает как готовый промпт

- Исходный файл фиксирует batch-прогон и эмулированные ответы, поэтому он
  выглядит как отчет, а не как команда для внешнего ИИ.
- Список service aliases полезен для эксперимента, но в copy-paste промпте
  должен быть короче и зависеть от подтверждения пользователя.
- `_exp` может ссылаться на `classification.md`, а `_simple` должен работать
  без знания внутренних capability id.

### Ядро промпта

1. Проверить готовность требования и User Story.
2. Согласовать акторов и остановиться.
3. Согласовать компоненты и остановиться.
4. Сгенерировать Use Case только после подтверждений.
5. Помечать неподтвержденные ветки как `Assumed`.

### Упрощения для `_simple`

- Использовать понятные роли и компоненты без Mango capability id.
- Оставить строгое правило остановки после акторов и компонентов.
- Формат результата оставить полным, но без YAML и репозиторных ссылок.

### Критичные ссылки для `_exp`

- `research/mango/classification.md` - source для capability mapping.
- `projects/mango/kb/glossary.md` - source для подтвержденных алиасов, когда
  файл доступен.
- `projects/mango/standards/classification-glossary.md` - fallback glossary.
