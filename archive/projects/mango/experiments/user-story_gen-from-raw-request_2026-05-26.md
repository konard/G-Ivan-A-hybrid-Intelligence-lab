# Эксперимент: генерация User Story из сырого запроса

Версия: 0.1

Дата: 2026-05-26

## Назначение

Эксперимент проверяет, может ли системный промпт Mango принимать сырой запрос
без заданной структуры, сначала детектировать свойства входа, затем
нормализовать термины, сопоставить запрос с Mango taxonomy и вернуть
контролируемый Markdown + YAML результат с логами в том же файле.

Результат не заменяет атомарный классификатор и не утверждает финальные правила
Mango. Это draft-прототип для проверки формы вывода, устойчивости секций и
качества вопросов при неполном входе.

## Метод

| Шаг | Действие | Контроль |
| --- | --- | --- |
| `0` | Детекция входа: роль, ценность, полнота, ФТ/НФТ, словарные совпадения. | Таблица `## 🔍 Детекция входа`. |
| `1` | Нормализация терминов к Mango glossary и `classification.md v3.0`. | Логи `normalized-term`, `unknown-term`, `needs-review`. |
| `2` | Маппинг на `Domain -> Capability -> Feature -> Atomic Function`. | YAML-мета с `confidence` и `mapping_status`. |
| `3` | Генерация User Story только из подтвержденных данных. | Story + acceptance criteria или вопросы вместо домысливания. |
| `4` | Запись логов в том же Markdown-файле. | Раздел `## 📋 Логи выполнения` с блоками ✅/⚠️/❌/❓. |

RAG-навигатор использует `projects/mango/kb/glossary.md` как целевой источник
терминов после его появления. До этого ближайший активный источник -
`projects/mango/standards/classification-glossary.md`; продуктовая иерархия
берется из `research/mango/classification.md v3.0`.

## 🚀 Предложения по улучшению

### Улучшения структуры промпта

1. Разделить генерацию на два режима: `story-ready` для входов с
   `completeness >= medium` и `clarification-only` для `completeness = low`.
   Это снижает риск черновых User Story без достаточного context.
2. Добавить `structure_signature` и `yaml_keys` в машинный блок каждого
   результата. Тогда Output Stability можно проверять автоматически без
   сравнения всего текста.
3. Добавить явное поле `assumptions`. Если промпт вынужден принять допущение,
   оно должно быть отдельным списком, а не скрываться в тексте story.

### Улучшения формата лога

1. Хранить каждый лог как запись с полями `severity`, `code`, `message`,
   `evidence`, `action_required`, сохраняя человекочитаемые блоки ✅/⚠️/❌/❓
   как представление поверх этих записей.
2. Добавить `blocking: true|false` к ошибкам. Например, `missing-role` блокирует
   финальную story, а `missing-timeout` может оставить draft с вопросом.
3. Добавить `source_artifact` для каждого mapping: glossary, classification,
   manual inference или user input. Это упростит review спорных решений.
